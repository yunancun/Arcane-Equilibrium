============================================================
三、项目最终定义（截至当前不应再遗忘）
============================================================

本项目不是普通自动下单 bot，也不是单纯量化脚本集合。

真正要做的是一个：
“以 OpenClaw 为中枢、Bybit 为主交易所、Binance 为辅助交易所的 AI Agent 交易系统。”

长期目标不是“尽快实盘”，而是：

1. 风险可控
2. 可审计
3. 可学习
4. 可逐步放权
5. 在成本、延迟、权限治理、系统稳定性都受控的前提下，追求长期 net PnL 最大化

绝对不能再丢的根原则：

A. 不只看 gross PnL，要看 net PnL
- AI API 成本
- 手续费
- 电费
- 设备折旧
- 网络与基础设施成本
- 未来维护成本
都必须纳入经营口径。

B. AI 输出不能当成即时命令
- AI 有调用延迟；
- 市场高度时效敏感；
- AI 更适合输出 decision lease（决策租约）或治理信息，而不是立刻买卖指令；
- 本地层必须继续做 freshness / timing / risk / permission / legality / execution gate。

C. 权限不能按时间自动升级
- 必须按表现赢得；
- 必须局部放权，不可全局放权；
- 真正高水平的 Agent 不是更莽，而是在优势区间更大胆、在非优势区间继续保守。

D. 最终形态不是机械 bot，而是长期进化型交易个体
它要会：
- 观察
- 判断
- 审计
- 学习
- 自我观测
- 产出经营报告
- 在严格边界下逐步争取更高权限

============================================================

从已上传资料综合判断，项目思路至少经历了以下几次真正的架构升级：

第一次升级：从“自动交易 bot”升级为“AI Agent 交易系统”
- 不再只是下单；
- 强调风险、审计、学习、分层放权；
- 明确 OpenClaw 是总编排中枢，Bybit 为主交易所，Binance 为辅助。

第二次升级：把 net PnL、decision lease、performance-based permission 变成核心原则
- 不再只看 gross PnL；
- AI 不能直接成为即时命令源；
- 权限不能按时间自动提升，而要按可量化表现赢得。

第三次升级：Revision 2 正式重排章节树
- 把原先的 D21 / D22 / D23 等理解，重排为 D/E/F/G/H/I/J/K/L/M/N；
- 正式承认 J / K 可以提前形成结构骨架，但后续主线仍必须回到 H / I。

第四次升级：H0 被正式从 H 前拆出
- 项目不再理解成“直接 H -> I”；
- 而是：
  H0 本地确定性判断内核
  -> H AI 治理
  -> I Decision Lease
  -> J Transition Engine
  -> K Paper / Demo Gate

第五次升级：3 月 22 日夜间把“合法 no-call / 合法 idle account”正式提升为语义层问题
- 这不是再加功能；
- 而是把“系统如何正确接受静默态、不调 AI 的合法终态”提升为主链语义要求；
- 也就是说，当前最新问题已不再是“代码没写”，而是“语义审计规则还没接受正确终态”。

6.7 2026-03-20：项目进入 H0 / H / I 深化阶段
------------------------------------------------------------

3 月 20 日是项目架构理解再次升级的关键节点。

最重要的方向变化有三件：

第一，H0 被正式提出
- H0 = Local Deterministic Judgment Core（本地确定性判断内核）
- 作用不是替代 AI，而是在本地先完成：
  - 交易资格判定
  - 风险包络判定
  - 市场摩擦判定
  - 系统摩擦判定
  - AI 升级前置筛选

第二，H 与 I 被重新理解为“大脑外壳”
- H（AI Governance）负责：
  - 该不该想
  - 值不值得问
  - 问谁最划算
  - 预算、路由、延迟、反滥用
- I（Decision Lease / Execution Authorization Control Plane）负责：
  - 谁能把看法变成权力
  - 如何把 AI 输出封装成可审计、可衰减、可撤销、可 supersede 的控制平面

第三，本地执行内核的重要性被正式放大
- 真正决定未来能否盈利的，很可能不是“AI 是否更聪明”，而是“本地确定性判断是否足够快、足够硬、足够不犯低级错误”。

七、关键设计结论：后续新对话必须继承什么
============================================================

------------------------------------------------------------
7.1 H0：Local Deterministic Judgment Core
------------------------------------------------------------

H0 的正式职责不是“直接预测买卖”，而是先在本地完成应该由本地完成的判断。

它应优先回答：
1. 现在是否具备交易资格？
2. 如果不具备，阻塞原因是什么？
3. 如果具备，属于哪种级别的资格？
4. 是否值得升级到 AI 治理层？
5. 扣除各种现实摩擦后，是否仍存在净可交易边际？

H0 的关键抓手包括：
- public microstructure
- market friction
- risk envelope
- trade eligibility
- H0 final audit / handoff

------------------------------------------------------------
7.2 H：AI Governance
------------------------------------------------------------

H 不是“让 AI 直接决定交易”，而是让 AI 成为受治理的认知资源。

H 解决的问题是：
- 该不该想
- 值不值得问
- 问谁最划算
- 调用多少预算
- 在何种情况下必须禁止调用

H 的关键定位：
- 只读
- 不放权
- 可审计
- 预算受控
- 可被本地层前置筛掉

------------------------------------------------------------
7.3 I：Decision Lease / Execution Authorization Control Plane
------------------------------------------------------------

I 不是“给 AI 开执行权限”，而是把 AI 输出变成一个：
可审计、可衰减、可撤销、可 supersede、可重放的控制平面。

关键理念：
1. lease 不是执行许可本体
2. lease 必须可审计、可撤销、可过期、可重放
3. 影子控制平面阶段必须维持：
   - execution_authority = not_granted
   - decision_lease_emitted = false
   - live_operator_ack_enabled = false
   - runtime_still_protected = true

------------------------------------------------------------
7.4 本地执行内核的重要性
------------------------------------------------------------

3 月 20 日的讨论反复强调：
未来真正决定系统能否盈利的，可能不是“AI 说得多聪明”，而是：
- 本地时效
- 本地合法性检查
- 本地风险包络
- 本地市场摩擦与系统摩擦
- 本地对 AI 升级价值的前置筛选

因此本地执行内核不是附属件，而是未来自动交易 Agent 的反射神经和最终落地系统。

------------------------------------------------------------
7.5 最新新增的两条语义原则
------------------------------------------------------------

这是 3 月 22 日夜间新锁死的规则，优先级极高：

语义 A：合法 idle account
如果同时满足：
- system_mode == read_only
- execution_state == disabled
- position_count == 0
- order_count == 0
- execution_count == 0

那么以下现象是 expected idle account，不应再进 blocker/failure：
- no_open_positions
- no recent order history
- no recent execution history
- private ws 只有 control message、没有 business topic event（只要 auth/subscribe 正常）

它们最多只能作为 info / idle_flags。

语义 B：合法 no-AI-call
如果同时满足：
- route_plan == route_skip
- selected_ai_tier == none
- should_call_ai == false

那么这代表：
“这一轮本来就不该调用 AI”，属于合法 no-call 终态。

不能再因为：
- invocation_attempted = false
- provider_response_present = false
- parsed_json_present = false
就把整条 H1-H5 链判失败。

十一、期间讨论了哪些关键问题、做了哪些重要决定

------------------------------------------------------------
11.1 API 成本与收益如何均衡
------------------------------------------------------------

3 月 20 日高强度讨论沉淀出的关键结论是：
- 不是每次都问 AI；
- AI 是昂贵稀缺的高阶认知资本；
- 每次 AI 调用都应被视为一笔必须回本的投资。

由此形成：
Expected Net Query Value
= 预期决策增益 - API 成本 - 延迟惩罚 - 错误风险惩罚 - 机会成本

------------------------------------------------------------
11.2 AI 与本地判断如何分层
------------------------------------------------------------

最终结论不是“让 AI 参与更多”，而是：
- Local Reflex Lane：毫秒级本地反射，不问 AI
- Local Tactical Lane：本地评分 / 过滤 / 风控 / 场景识别
- AI Deliberation Lane：秒级高阶抽象判断，仅在值得时调用

------------------------------------------------------------
11.3 Decision Lease 为什么重要
------------------------------------------------------------

AI 不能输出“立刻买卖”的即时命令。
更合理的形态应是：
- AI 输出带适用条件、时效、风险边界、失效条件的决策租约
- 本地层在最新事实下再复核是否允许真正消费

------------------------------------------------------------
11.4 权限如何治理
------------------------------------------------------------

最终否决了“按时间自动升级”的粗糙方案。
改为：
- 权限必须按表现赢得
- 只能在已验证有效的子场景中局部放权
- 真正高级的是“在优势区间更大胆，在非优势区间更保守”

------------------------------------------------------------
11.5 本地执行内核为什么必须做深
------------------------------------------------------------

最终沉淀出的方向是：
- 真正决定未来是否能把 AI 判断转化为真实净收益的，可能不是 AI 本身，而是本地执行内核的时效、合法性、边际判断和风险复核能力。

------------------------------------------------------------
11.6 route 与 active model 的关系
------------------------------------------------------------

这次返修中明确暴露出：
- route selector 决定 lane
- request envelope 真正读取的是 active provider/model env
- route 选择 与 active model 绑定不是完全同一层逻辑

------------------------------------------------------------
11.7 最新新增的 no-call / idle 语义
------------------------------------------------------------

这是目前最新的重要决定：
- 合法 no-call 不能被当成失败
- 合法 idle account 不能被当成异常
- 不允许为了“刷绿”而伪造一次 AI 调用
