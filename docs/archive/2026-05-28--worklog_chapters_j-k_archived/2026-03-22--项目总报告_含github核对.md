## 2. 项目最新正式定义

OpenClaw / Bybit 项目不是“让 AI 直接自动下单”的 bot，而是一个**受治理、可审计、可逐步放权**的交易 Agent 系统。

它的目标是：
- 先用本地确定性判断完成该本地就能完成的判断。
- 只在值得思考、值得花钱、值得承担延迟时，才进入 AI 治理层。
- AI 的输出只能先成为受治理的 observation，而不能直接变成执行许可。
- 任何未来执行权，都必须通过 lease / authority / approval 等控制平面逐步放开。
- 最终追求的是长期 **net PnL**，而不是表面上的 gross PnL。

一句话概括：
> H0 负责“本地先判断是否值得思考”，H 负责“AI 如何被治理地思考”，I 负责“谁有资格把思考结果变成权力”。

## 6. 后续 workflow / 维护准则（这部分是新加的重点）

### 6.1 GitHub / 仓库口径
以后凡是讨论“仓库里到底有没有这个脚本、正式文件名是什么、哪些内容已经提交到 main”，都要优先以 GitHub 仓库为准；日志/报告负责补充设计与历史，不能替代 repo 真相。

### 6.2 脚本定位顺序
先看 README → docs/SCRIPT_INDEX.md → 章节归位/蓝图文档 → 目标脚本头部 MODULE_NOTE → 对应 latest 输出 → final audit / handoff。

### 6.3 Shell 执行准则
关键脚本优先走 `run_with_trading_env.sh`，不要裸跑 python3；每次只给 1-2 步 shell 命令；不要把终端输出再粘回 shell；here-doc / `cat <<'PY'` 必须确认结束标记完整。

### 6.4 注释与文档准则
所有新核心脚本一开始就写 MODULE_NOTE；注明 role / purpose / upstream / output / notes；写完立即 py_compile；必要时同步补入 `docs/SCRIPT_INDEX.md`。

### 6.5 latest/dated 准则
所有核心产物优先保持 latest + dated 双写；字段、状态枚举、版本号必须稳定；任何字段名/版本号变更都要同步下游 checker、summary、audit、handoff。

### 6.6 builder / canonical 准则
仓库里若同时存在 builder / final / contract / runner / canonical 多套脚本，后续新对话不能随手混跑；必须先确认当前 canonical mainline。

### 6.7 安全边界准则
当前长期硬边界：system_mode=read_only、execution_state=disabled、execution_authority=not_granted、decision_lease_emitted=false、live_operator_ack_enabled=false。

### 6.8 truth 优先准则
“代码存在”不等于“truth-green 已闭环”；新的 closure/handoff 若与夜间 truth check 冲突，必须以 truth check 优先。

### 6.9 合法 no-call 准则
当 policy 已判定 `should_call_ai=false`、tier=none、route_skip 时，这可以是合法终态，不能再机械要求必须出现真实 provider response。

## 12. 期间最重要的坑与修复

### 12.1 路径 / 环境类
- secret 路径曾漏掉 `/home/ncyu`。
- 裸跑 `python3` 导致没进正确 venv/env 链。
- active env 与 route 默认 env 不是一回事；H1-E 实际读的是 `BYBIT_AI_ACTIVE_MODEL`，不是 `BYBIT_AI_ACTIVE_MODEL_NAME`。

### 12.2 Shell / 操作类
- `cat <<'PY'` / heredoc 经常因为结束标记没收好，看起来像终端卡死。
- 多次把终端输出重新粘回 shell，导致 JSON 字段被当命令执行。
- 长粘贴场景下出现截断、提示符混入、脚本写坏。

### 12.3 代码 / 编译类
- 批量注入头注释时曾造成 SyntaxError，后已修复。
- patch helper 曾因全角中文标点导致拼装错误。
- builder 与 canonical 脚本混用，导致 response/contract 一度短暂脏状态。
- refresh 脚本曾混用 H0 的旧文件名与新文件名，造成“脚本存在但跑错对象”的真实 bug。
