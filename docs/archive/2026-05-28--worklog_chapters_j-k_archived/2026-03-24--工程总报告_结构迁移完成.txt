七、今天新的 workflow（必须整理进去，后续照此合作）
------------------------------------------------------------
这是本文件对下个对话最重要的附加价值之一。

A. 以后默认采用“底部固定回贴块” workflow
------------------------------------------------------------
用户今天明确提出两个痛点：
1. shell 输出太长，滚动寻找需要复制的内容很麻烦；
2. 助手有时把要复制的段落分散在 1/3/5/7/9 多个位置，切窗口粘贴很不舒服。

因此今天定下的新规则应是：

规则 1：每轮 shell 命令结束后，真正需要用户回贴的内容，必须固定放在最下面。

规则 2：最下面统一使用：
==================== REPLY BLOCK START ====================
...
==================== REPLY BLOCK END ======================

规则 3：如果能只回贴 1 个 block，就不要拆成很多零散段。

规则 4：block 里面只放：
- report dir
- 最少的 rc / state / summary
- 必要的 compact preview
- DONE 标记
不要把整段长日志塞进 REPLY BLOCK。

规则 5：shell 输出设计时应做到：
- 前面可以执行很多检查；
- 但最后统一汇总成 1 个短 block 给用户复制；
- 用户只需要滚到底部复制一次，不需要来回翻很多屏。

B. shell 命令给法：短、小、分轮次
------------------------------------------------------------
后续新对话仍要继承：
- 一次只给 1 小组命令；
- 最多 1-2 个逻辑动作；
- 中英双语注释；
- 明确写清：
  - 只检查 / 只读
  - 会修改 / 会写文件
  - 会 commit / 会 push

不要再一次给很长很大的 shell 包，避免：
- paste 失败
- 终端卡顿
- 用户误操作
- 上下文被巨量输出占满

C. 已 push 的代码优先直接看 GitHub
------------------------------------------------------------
现在应默认：
- 已 push 文件：助手先自己看 GitHub / 文件内容；
- 本地-only 内容（runtime/latest/json/log/env/config）：再让用户用 shell 读给助手；
- 不要再默认让用户手贴整段代码正文。

D. shell 回贴应尽量压缩为“汇总块”，不是全量输出
------------------------------------------------------------
推荐格式：
- report dir
- 关键计数
- 关键状态
- top 10/20 结果
- DONE

不推荐：
- 把所有 compile/grep/head/full log 全贴回对话。

E. 节奏规则：稳定一轮再 commit / push
------------------------------------------------------------
后续新对话应继续：
1. 小范围修理
2. 小范围验证
