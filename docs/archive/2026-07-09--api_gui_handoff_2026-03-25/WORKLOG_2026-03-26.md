## 3. 今日已完成工作

### 3.1 后端新增端点（main_legacy.py +418 行）

| 端点 | 方法 | 功能 |
|---|---|---|
| `/api/v1/control/product-family/{family}/config` | POST | 修改指定产品族的 enabled_switch / visibility_switch / mode_switch / action_permissions |
| `/api/v1/system/business/summary` | GET | 完整经营摘要：每日 PnL + 最近费用/事件/PnL 历史 + 按类别成本分解 |
| `/api/v1/input/pnl-entry` | POST | 录入 PnL 条目：realized_pnl 累加、unrealized_pnl 快照覆盖 |

#### 3.1.1 产品族配置写接口

- **6 个产品族全部支持**：spot / margin / perp_linear / perp_inverse / options / other_derivatives_reserved
- **安全限制**：`mode_switch` 只允许 `disabled` / `observe_only` / `shadow_only`；尝试设置 `live_ready` 等会被拒绝并记录在 `rejected_fields` 中
- **动作权限控制**：可单独设置每个产品族的 `new_order` / `cancel` / `amend` / `reduce_only` / `increase_position` / `close_position` 开关
- **审计**：每次变更写入 audit_context，产品族自身记录 `last_change_ts_ms` / `last_change_by`
- **幂等**：复用现有 idempotency_key 机制

#### 3.1.2 经营摘要与 PnL 录入

- **PnL 语义**：`realized_pnl` 是增量累加；`unrealized_pnl` 是快照覆盖（取最新值，不累加）
- **net_operating_pnl = gross_pnl - total_cost**，其中 `gross_pnl = realized_pnl + unrealized_pnl`
- **Business Summary** 返回：daily 指标 + 最近 20 条费用/事件/PnL 历史 + 按 category 的成本分解 + 各类条目总计数
- **兼容旧状态文件**：`pnl_entries` 列表不存在时自动创建，不要求旧 JSON 有此字段

#### 3.1.3 新增 Pydantic 模型

- `ProductFamilyConfigData`：产品族配置写操作结果
- `PnLEntryData`：PnL 条目录入结果
- `BusinessSummaryData`：经营与收益完整汇总

### 3.2 GUI 全面升级（app.js 完全重写 → 830+ 行，styles.css +82 行）

#### 3.2.1 产品族配置设置台

- 每个产品族一个独立卡片：enabled / visible 勾选框 + mode 下拉选择（disabled / observe_only / shadow_only）
- "应用配置"按钮调用 `POST /control/product-family/{family}/config`
- 可展开"动作权限"面板，逐项勾选各动作的 configured switch，点"应用权限变更"提交
- 实时显示 capability_state / execution_authority_state 徽章

#### 3.2.2 收益与成本录入面板

- **费用录入表单**：金额 + 类别（manual / ai_api / exchange_fee / slippage / infra）+ 备注 → POST /input/cost
- **PnL 录入表单**：类型 + 已实现盈亏增量 + 未实现盈亏快照 + 标的 → POST /input/pnl-entry
- **历史条目展示**：最近费用记录列表 + 最近 PnL 记录列表（时间/类型/金额/备注）
- **成本分解面板**：按 category 汇总的成本分类展示
- **条目总计数**：费用/PnL/事件三类各自的总条目数

#### 3.2.3 系统设置台

- **风险策略**：default_guarded / manual_blocked 下拉选择 + 当前生效状态显示
- **Demo Ack 开关**：是否要求 demo 操作确认
- **学习审批开关**：实验是否需要人工审批
- 所有设置变更通过 `POST /input/config-change` 提交

#### 3.2.4 二次确认保护

- 新增 `pf-config` 和 `settings-change` 两类关键动作的确认弹窗元数据
- 所有写操作（产品族配置、权限变更、费用录入、PnL 录入、设置变更）都经过确认流程

#### 3.2.5 数据加载升级

- `loadDashboard()` 现在并发加载 6 个端点（新增 `/system/business/summary`）
- 经营面板直接使用 summary 端点数据，不再依赖 overview 中的简化版
