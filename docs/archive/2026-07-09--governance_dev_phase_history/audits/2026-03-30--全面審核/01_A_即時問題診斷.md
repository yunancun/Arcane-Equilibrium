## 根因分析

### 🔴 P0-A1：MA_Cross 信號永遠被 regime 過濾器阻擋

**影響：** 7 個 MA_Crossover 策略（PRLUSDT/BIRBUSDT/BSBUSDT/ONTUSDT/BTRUSDT/4USDT/ARIAUSDT）全部失效

**觸發路徑：**

```
SignalEngine → MACrossoverRule.evaluate() → Signal(metadata=None, ...)
                                                        ↓
Orchestrator._on_signal():
    # MA_Cross 的 metadata 是 {} (空字典)
    if hasattr(signal, "metadata") and signal.metadata and ...:
    #                                  ^^^^^^^^^^^^^^^^^
    #                                  {} 是 falsy！→ 不執行
    # _regime 永遠不被注入到 MA_Cross 信號的 metadata 中
                                                        ↓
MACrossoverStrategy.on_signal():
    signal_regime = signal.metadata.get("_regime", "unknown")
    # → "unknown"（因為 _regime 從未被注入）
    if signal_regime in ("ranging", "squeeze", "unknown"):
        return  # ← 永遠在這裡 return，永遠不生成 intent
```

**根本原因：**
1. `MACrossoverRule.evaluate()` 創建 `Signal` 時不傳 `metadata` 參數
2. `Signal.__init__` 中 `self.metadata = metadata or {}` → 空字典
3. Orchestrator 的注入條件 `if signal.metadata` → `{}` 為 **falsy**，注入被跳過
4. Session 7 修改後，`"unknown"` 加入過濾列表，使此 bug 成為致命問題

**時間線（重要）：**
- Session 7（2026-03-28）：將 `"unknown"` regime 加入禁止列表
- **效果：從那時起，MA_Crossover 策略永遠無法生成 intent**
- 舊 session 684 fills 是在 Session 7 之前的不同 server process 產生的
- 現 server 啟動於 2026-03-29 14:31 → 迄今 **0 fills**（11h 舊 session + 10.6h 新 session）

**程式碼位置：**
- `local_model_tools/signal_generator.py:369–376`（Signal 創建無 metadata）
- `app/pipeline_bridge.py:259`（orchestrator._on_signal，falsy 檢查）
- `local_model_tools/strategies/ma_crossover.py:110–116`（regime 過濾）

**修復方向：**
```python
# 選項 A（建議）：orchestrator 注入時不依賴 metadata truthiness
if hasattr(signal, "metadata") and self._current_regime not in ("unknown",):
    if signal.metadata is None:
        signal.metadata = {}
    signal.metadata["_regime"] = self._current_regime

# 選項 B：MACrossoverRule 在創建 Signal 時傳入 regime 佔位符
return Signal(..., metadata={"_regime": "unknown"})
# 然後 orchestrator 的 truthiness 檢查改成 `is not None`
```

### 🔴 P0-A2：FundingRate_Arb 策略只接收 BTC/ETH funding rate

**影響：** 3 個 FundingRate_Arb 策略（CFGUSDT 107.9bps / ENJUSDT 166.7bps / SENTUSDT 94.0bps）**從未正確評估**

**觸發路徑：**

```python
# pipeline_bridge.py:1432 _check_funding_rates()
for symbol in ["BTCUSDT", "ETHUSDT"]:   # ← 只有 BTC 和 ETH！
    # 獲取該 symbol 的 funding rate
    # 然後：
    for strategy in self._orch._strategies.values():
        if hasattr(strategy, "evaluate_funding_opportunity"):
            strategy.evaluate_funding_opportunity(
                funding_rate=btcusdt_or_ethusdt_rate,  # ← 錯誤！CFGUSDT 策略收到的是 BTC 的 rate
```

**問題：**
- 部署的 FundingRate_Arb 策略（CFGUSDT/ENJUSDT/SENTUSDT）收到 BTCUSDT/ETHUSDT 的 funding rate
- BTCUSDT 的 funding rate 通常接近 0（中性市場），不觸發策略閾值
- 即便觸發，也是用錯誤品種的 rate 做決策，語義上錯誤

**程式碼位置：**
- `app/pipeline_bridge.py:1436`（只循環 BTC/ETH）

**修復方向：**
```python
# 應為每個 FundingRate_Arb 策略獲取其自己 symbol 的 rate
for strategy in self._orch._strategies.values():
    if hasattr(strategy, "evaluate_funding_opportunity"):
        symbol = getattr(strategy, '_symbol', None)
        if symbol:
            # 為該 symbol 獲取 funding rate
            rate = self._fetch_funding_rate(symbol)
            strategy.evaluate_funding_opportunity(rate, ...)
```
