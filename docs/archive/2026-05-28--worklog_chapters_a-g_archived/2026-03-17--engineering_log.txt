============================================================
一、项目总目的（这部分非常重要，后续不可遗忘）
============================================================

这个项目的目标，不是做一个普通的自动交易脚本，也不是只做一个会下单的 bot。

真正要做的是一个：
“以 OpenClaw 为中枢、Bybit 为主交易所、Binance 为辅助交易所的 AI Agent 交易系统。”
它的长期目标是：
- 风险可控；
- 可审计；
- 可学习；
- 可逐步放权；
- 在成本、延迟、风控、权限治理都受控的前提下，追求长期净收益最大化。

这里有几个不能忘记的根原则：

1. 不是只看 gross PnL，而是必须看 net PnL
- 交易收益不是唯一目标；
- 必须同时考虑：
  - AI API 成本；
  - 电费；
  - 设备折旧；
  - 手续费；
  - 基础设施长期成本；
  - 未来可能的人力维护成本。
- 所以本项目本质上不是单纯“交易系统”，而是“交易系统 + 经营系统”。

2. AI 输出不能当成即时命令
- 用户明确指出：AI 调用和分析有延迟；
- 市场很多变化是瞬时的；
- 因此 AI 输出不能直接视为“立即执行命令”；
- AI 更应输出 decision lease（决策租约）；
- 本地执行层必须再做：
  - 时效校验；
  - 风控校验；
  - 最新市场状态校验；
  - 权限与环境校验；
  - 最终允许或拒绝执行。

3. 权限不能按时间自动升级，必须按表现赢得
- 不能因为系统运行久了就自动放权；
- 权限要基于可量化表现逐步提升；
- 不是全局放权，而是：
  - 在已经验证有效的子场景中局部放权；
  - 在未验证区间继续保持保守；
- 真正高级的 Agent 不是更莽，而是更会在优势区间使用更高权限。

4. Agent 的长期形态，不是机械 bot，而是长期进化型交易个体
- 不只是执行；
- 还必须能够：
  - 自主分析；
  - 总结经验；
  - 学习与迭代；
  - 感知自身系统状态；
  - 产出经营级报告；
  - 在未来根据结果修正自己的行为。

============================================================
二、Agent 核心需求（这部分也非常重要）
============================================================

Agent 的核心需求不是单一策略，而是一整套系统能力。至少包括：

A. 安全与治理
- 默认只读；
- 默认不允许 live execution；
- 默认不允许 AI 直接驱动真实执行；
- 所有升级必须遵循分阶段推进；
- 所有关键状态必须可审计、可解释、可追溯。

B. 数据感知能力
- 能读真实交易所公共与私有只读数据；
- 能建立统一 snapshot；
- 能接入 WS；
- 能区分控制面消息与真实 business-topic 事件；
- 能维护 freshness / health / state。

C. 决策输入能力
- 把账户、持仓、订单、成交、WS 状态、business event 状态统一加工成标准决策输入；
- 决策输入必须结构化；
- 不是“为了问 AI 而问 AI”，而是先做好本地输入治理。

D. AI 调用治理能力
- 只有在值得思考时才调用 AI；
- 本地能决定的，先本地决定；
- AI 调用必须被预算控制；
- 后续必须有 abuse prevention / compute governor / model router。

E. 学习能力
- 未来必须记录：
  - 过去做过什么判断；
  - 哪些判断是对的；
  - 哪些判断是错的；
  - 哪些环境下什么策略有效；
  - 哪些 AI 调用纯属浪费。
- 要形成“事实 -> 决策 -> 结果 -> 归因 -> 更新”的闭环。

F. 自我观测能力
- 不只是看市场，也要看自己；
- 包括未来对：
  - CPU / RAM；
  - 网络延迟；
  - REST 延迟；
  - WS 稳定性；
  - DB 写入耗时；
  - 模块阻塞点；
  - 整体效率瓶颈
进行持续观测。

G. 经营报告能力
- 未来要能生成 daily / weekly / monthly 级别的：
  - 收益；
  - 成本；
  - 净收益；
  - 问题归因；
  - 优化建议；
- 最终让系统不仅“能跑”，还“会交经营报告”。

八、D22 完成情况：business event state 集成
============================================================

D22 的核心，不是让系统“已经看到了真实业务事件”，而是让系统能够正确地区分三件事：
1. WS 健康但没有真实业务事件；
2. WS 看到了真实 business-topic 事件；
3. business-event 侧失真、过期、或不可信。

这个阶段最关键的概念是：
“healthy_no_business_events_yet”
也就是：
“健康空态”。

这个状态绝对不能被误判为故障，也不能被误判为已经具备交易事件流。

十一、今天及最近几天遇到的主要问题与修复
============================================================

1. heredoc / cat <<'PY' 终端看似“卡住”
- 常见原因是 shell 仍在等待 heredoc 结束标记，或脚本本身仍在运行；
- 这是近期多次出现的操作层问题。

2. 多次把终端输出又粘回 shell
- 导致 JSON 字段和终端输出被误当命令执行；
- 这是粘贴误操作，不是程序逻辑错误。

3. 刷新顺序错误导致 freshness stale
- 改完 runtime / summary / audit 后直接跑集成层，会因为上游 latest 没刷新而出现 stale；
- 正确经验：
  改完核心脚本后先刷新 readonly chain，再跑集成层。

4. 版本漂移问题
- handoff 升级到 v3 后，audit 还按 v2 校验；
- 已修复；
- 以后版本号一旦改动，要同步所有下游 checker / audit / summary / handoff / policy。

5. 字段名漂移问题
- snapshot 文件实际使用 ts_ms；
- 某些检查脚本曾按 snapshot_ts_ms 取值；
- 导致 null；
- 已修复。

6. handoff contract checker 文件漏建
- D23.8 首次运行时发现 bybit_event_driven_handoff_contract_check.py 不存在；
- 后续补建并通过；
- 说明新阶段结束前要确保“主文件 + contract checker + 如有需要则 final audit”成套存在。
