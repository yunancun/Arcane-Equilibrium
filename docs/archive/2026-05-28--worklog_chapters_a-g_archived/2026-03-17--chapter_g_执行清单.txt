==================================================
三、P0：G 章执行清单（下一分支起点）
==================================================

--------------------------------------------------
G1. replay harness / fixtures
--------------------------------------------------
目标：
- 建立可重复使用的真实/半真实 business event 输入夹具
- 能重放 wallet / order / execution / position 事件样本
- 为后续非空业务事件验证打底

完成标准：
- 能从样本文件或构造 fixture 重放事件
- 输出 latest / dated 结果
- 有 contract check
- 有基础说明注释

验收重点：
- 事件输入格式稳定
- 能用于重复测试
- 不会污染现有 readonly observer 主链

--------------------------------------------------
G2. 非空真实业务事件验证
--------------------------------------------------
目标：
- 验证出现真实 wallet/order/execution/position 事件时，
  系统是否能从 empty 正确切换到 non-empty 语义

完成标准：
- 至少验证 wallet / order / execution / position 四类中的主要事件流
- business_event_state 能从 healthy_no_business_events_yet 正确过渡
- event-driven state / phase / input / decision / outcome 的语义同步变化

验收重点：
- topic_counts 不再为空
- normalized_count > 0
- has_business_events = true
- 相关 readiness / phase / decision 不再停留 empty 分支

--------------------------------------------------
G3. 幂等 / 去重 / 指纹 / 乱序 / replay 验证
--------------------------------------------------
目标：
- 确保相同事件不会被重复计入
- 确保乱序、重放、重复到达时语义仍可控

完成标准：
- 有 event fingerprint 相关验证
- 对重复事件、乱序事件、重复回放有测试输出
- contract / consistency 层能反映异常或正确处理结果

验收重点：
- normalized_count 不应因重复事件失真
- 同一事件不可重复推进状态
- replay 行为要可解释、可审计

--------------------------------------------------
G4. 非空事件 acceptance / regression 套件
--------------------------------------------------
目标：
- 建立一组专门针对“真实 business-topic 非空流”的回归检查

完成标准：
- 新 acceptance 或 regression 检查存在
- 能区分 empty healthy 与 non-empty healthy
- 有 latest 审计文件
- 有失败检查输出结构

验收重点：
- 后续改脚本时，不会把 non-empty 路径搞坏而无人发现

四、P1：H 章执行清单（AI 治理）
==================================================

--------------------------------------------------
H1. thought_gate
--------------------------------------------------
目标：
- 明确什么情况下允许进入 AI 思考/调用阶段
- 默认不问 AI，满足条件才问

完成标准：
- 有 gating 输出文件
- 有 allowed / blocked 语义
- 有原因字段

--------------------------------------------------
H2. query_budget
--------------------------------------------------
目标：
- 让 AI 调用受预算控制，而不是无限使用

完成标准：
- 有 budget state
- 有 remaining / blocked / exhausted 等字段
- 有 contract check

--------------------------------------------------
H3. model_router v2
--------------------------------------------------
目标：
- 建立不同任务去不同模型的路由逻辑
- 明确本地模型与云模型职责

完成标准：
- 有 router 输出
- 有 route reason
- 可解释为什么走某模型

--------------------------------------------------
H4. compute governor / anti-abuse
--------------------------------------------------
目标：
- 限制过度推理、重复提问、无意义计算

完成标准：
- 有 governor 判断输出
- 有 block / allow / throttle 语义

--------------------------------------------------
H5. AI cost logging / governance audit
--------------------------------------------------
目标：
- 所有 AI 调用都能记账、审计、回看

完成标准：
- 至少有成本记录 latest
- 有治理审计输出
- 能看见为什么调用、调用了什么层级

五、P1：I 章执行清单（Decision Lease）
==================================================

--------------------------------------------------
I1. decision lease schema
--------------------------------------------------
目标：
- 把未来 AI 结果变成受控的 decision lease，而不是直接命令

--------------------------------------------------
I2. lease freshness / validity
--------------------------------------------------
目标：
- 决策必须有有效期、过期判断

--------------------------------------------------
I3. revoke / supersede / expiry
--------------------------------------------------
目标：
- 新租约替代旧租约
- 旧租约可撤销、可过期

--------------------------------------------------
I4. lease 与 risk gate 集成
--------------------------------------------------
目标：
- 即使 lease 存在，也必须过风险门

--------------------------------------------------
I5. lease audit / replay
--------------------------------------------------
目标：
- 租约全链路可审计、可回放

这一章统一验收口径：
- lease 不能直接等于“执行许可”
- lease 只是受治理的决策载体

==================================================
十一、每次修改后的建议检查顺序
==================================================

若改的是 readonly observer 主链相关：
1. python3 -m py_compile
2. bybit_failure_policy_builder.py
3. bybit_full_readonly_observer_cycle.py
4. bybit_observer_acceptance_check.py
5. bybit_runtime_state_resolver.py
6. bybit_readonly_final_summary.py
7. bybit_next_phase_handoff.py
8. bybit_readonly_audit.py
9. bybit_latest_consistency_check.py

若改的是 business-event 或 event-driven 子链：
1. 相关 builder
2. 对应 contract check
3. chain consistency（若该层已有）
4. readiness / handoff / final audit（若该层已有）
