OpenClaw / Bybit 项目

一、报告目的
本报告用于在后续新对话、换分支、或阶段性复盘时，快速、准确地恢复当前项目在 H 章节（AI 治理链）与 I 章节（decision lease / approval / authority shadow control plane）上的真实完工状态，避免出现以下常见混乱：
1) 把 shadow/control-plane 完工误判为 live execution ready；
2) 把已经完成的 H/I 子章节重复重做；
3) 忽略 hard boundary，误把只读治理链推进到执行链；
4) 后续接手时忘记 H 与 I 的先后依赖关系和验收结果。

本报告严格基于本轮终端实测输出整理，不对未验证内容做乐观补充。

五、H 与 I 的依赖关系和现在的正确理解
======================================================================

必须牢记：
1) H 先于 I 完成，且 I 依赖 H 的治理结果；
2) H 的 governed observation 是 I 的上游输入之一；
3) I 的所有结构都建立在“read_only / no authority / no emit”的严格前提之上；
4) H/I 同时完成，并不等于 J/K 就自动完成，更不等于系统能跳到 live。

一句话概括：
H 解决“AI 是否受治理”；
I 解决“未来如果要把 AI 结果变成可执行决策控制平面，结构与审计链如何搭好”；
但目前仍没有解决“live execution activation”。

七、当前 warning_flags 的正确解释
======================================================================

当前 warning_flags 很长，但大部分属于“软警告 / 结构提醒”，不是章节失败。

它们大致可以分为五类：

A. 市场数据 / 新鲜度类
- recent_trade_last_price_missing
- recent_trade_last_ts_missing
- runtime_state_reference_old
- freshness_soft_warning_present
- last_trade_fields_missing
- volatility_band_not_preferred

含义：
当前市场事实仍不够强，AI 结果更适合作为 observation-only，不适合升级执行。

B. Shadow-only 结构类
- decision_lease_shadow_only_mode
- decision_lease_replay_shadow_only_mode
- decision_lease_replay_guard_shadow_only_mode
- decision_lease_adaptive_ttl_shadow_only_mode
- decision_lease_approval_bridge_shadow_only_mode
- execution_authority_aggregator_shadow_only_mode
- manual_approval_packet_review_only_mode
- operator_ack_shadow_only_mode

含义：
这些是在提醒“该章已完成，但完成的是 shadow 版本”。

C. 时序 / consume 现实摩擦类
- consume_now_would_be_expired
- consume_now_outside_recommended_window
- consume_now_exceeds_freshness_grace
- consume_now_path_would_not_pass
- lease_now_path_negative_headroom
- lease_friction_latency_elevated
- adaptive_ttl_recommendation_differs_from_current
- last_call_latency_exceeds_deadline_hint

含义：
这些是 future live design 的重要依据，说明真实时延与 TTL/窗口之间的摩擦不能被忽略。

D. 授权 / 审批尚未打开类
- decision_lease_operator_review_pending
- decision_lease_execution_authority_not_granted
- operator_review_still_pending
- execution_authority_not_granted
- live_authority_grant_not_active
- manual_approval_packet_live_submit_inactive
- operator_review_pending
- live_operator_ack_not_active

含义：
这类警告恰恰是在保护系统，告诉我们：别把结构闭环误当作授权闭环。

E. 成本 / pricing 绑定类
- provider_pricing_table_not_bound_in_mainline

含义：
当前成本治理主要基于受治理上限与 usage trace，并没有把真实 pricing table 绑定进主链。这不是造假，但代表后续若进入更高阶段，需要更正式的价格口径绑定。

总结：
这些 warning_flags 目前更多是“未来 live design 的问题清单”，不是“当前 H/I 闭环失败的证据”。

十、后续主线建议
======================================================================

本报告不建议下一步马上继续堆更多脚本。
更合理的主线顺序是：

第一步：
先把 H/I 正式完工状态固化为接手基线（本报告即承担该作用）。

第二步：
后续若继续推进，应明确分成两条线：
A) 结构/审计/测试层继续硬化；
B) future live design 单独开章，不得偷渡到现有主 runtime。

第三步：
在任何 future live design 开始前，必须先重新确认：
- 谁来授予 execution authority；
- 什么条件下 lease 才允许 emit；
- 谁审批；
- 哪个 freshness threshold 才是 live 可接受值；
- 真实 pricing / latency / market freshness 是否足以支持 live path；
- paper/demo execution gate 是否要先独立构建。

十二、最终结论
======================================================================

截至 2026-03-22 本轮终端实测，H 与 I 两章已经完成正式收口。

这代表：
- AI 治理层已闭环；
- decision lease shadow control plane 已闭环；
- 系统保护边界仍完好无损；
- 未发生“未授权推进到 live execution”的错误；
- 后续已具备进入“future live design”讨论的前提。

但这绝不代表：
- 可以实盘自动下单；
- 可以发 live decision lease；
- 可以把 manual approval / operator ack 当作 live 授权；
- 可以跳过新的 live execution 设计阶段。

一句话收尾：
H/I 已正式完工，且完工质量是“结构闭环 + 边界未破 + shadow 完整”，这正是当前阶段最正确的结果。
