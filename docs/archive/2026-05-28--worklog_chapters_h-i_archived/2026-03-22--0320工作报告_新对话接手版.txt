============================================================
一、项目真正目标（新对话最先要记住）
============================================================

这个项目不是普通自动下单 bot，而是一个：

“以 OpenClaw 为中枢、Bybit 为主交易所、Binance 为辅助交易所的 AI Agent 交易系统。”

长期目标不是“尽快实盘”，而是：

1. 风险可控
2. 可审计
3. 可学习
4. 可逐步放权
5. 在成本、延迟、权限治理、系统稳定性都受控的前提下，追求长期 net PnL 最大化

绝对不能忘的根原则：

A. 不只看 gross PnL，要看 net PnL
- AI API 成本
- 手续费
- 电费 / 设备折旧
- 网络 / 基础设施成本
- 后续维护成本
都要纳入经营口径。

B. AI 输出不能当成即时命令
- AI 有延迟；
- 市场高度时效敏感；
- AI 更适合输出 decision lease（决策租约），而不是“现在立刻买/卖”的直接命令；
- 本地层必须继续做 freshness / timing / risk / permission / execution gate。

C. 权限不能按时间自动升级
- 必须按表现赢得；
- 必须局部放权，不可全局放权；
- 真正高级的 Agent，不是更莽，而是在优势区间更大胆，在非优势区间更保守。

D. 最终形态不是机械 bot，而是长期进化的交易个体
- 会观察
- 会判断
- 会审计
- 会学习
- 会自我观测
- 会产出经营报告
- 会在严格边界下逐步争取更高权限

4.3 中前段：围绕 H / I 进行了高强度设计讨论（“天才般想法”要求）
------------------------------------------------------------

你在这条对话里明确要求过：

1. 如何最佳均衡 API 成本与交易 Agent 收益；
2. 如何以最低花费最大程度榨取 AI 价值；
3. 哪些基础且安全的交易判断应本地完成；
4. 本地判断与 AI 判断如何关联；
5. 如何处理 AI 回复延迟；
6. 如何把设备、网络、本地计算、DB、WS 等系统卡顿正式纳入框架；
7. 希望 H 与 I 拿出“天才般”的思路与框架，并执行到极致。

这一轮讨论沉淀出的正式结论，必须被新对话继续继承：

【结论 1：H 与 I 不能孤立理解，要作为同一个“大脑外壳”设计】
- H（AI Governance）负责：
  - 该不该想
  - 值不值得问
  - 问谁最划算
  - 预算、路由、滥用防护、延迟治理
- I（Decision Lease）负责：
  - 把 AI 的慢速认知结果，封装成可验证、可衰减、可撤销、可复核的行动许可外壳
- 它们是统一脑壳的两面，不该割裂。

【结论 2：AI 不是默认参与者，而是稀缺昂贵的“高阶认知资本”】
系统应分三条速度层：

- Local Reflex Lane（毫秒级）
  只做本地、确定性、必须快的判断，绝不问 AI。
- Local Tactical Lane（几十到几百毫秒）
  做本地评分、过滤、场景识别、风控复核，尽量不问 AI。
- AI Deliberation Lane（秒级）
  只处理慢变量、高不确定、跨上下文、需要抽象推理的问题。

【结论 3：每次 AI 调用都应被视作一笔必须回本的投资】
要建立：
Expected Net Query Value
= 预期决策增益 - API成本 - 延迟惩罚 - 错误风险惩罚 - 机会成本

只有明显为正时，AI 才允许参与。

【结论 4：预算不只是按“次数”，而要按“价值层级”控】
建议五层预算：
- 全局日预算
- 会话预算
- 符号预算
- 场景预算
- 失败预算

【结论 5：本地能做的，必须先本地做】
绝对本地层包括：
- freshness
- runtime health
- WS / REST / DB 状态
- 持仓/订单/成交事实判断
- spread / slippage / fee / min edge
- kill switch / circuit breaker
- permission / system_mode / execution legality

AI 只应处理：
- regime 判断
- 多因素冲突解释
- 场景重构
- 风险边界重估
- 结构变化解释
- 策略切换建议
而不应处理高频重复的小判断。

【结论 6：AI 输出必须变成 decision lease，而不是即时命令】
AI 不输出：
- “现在立刻买”

AI 应输出：
- 适用条件集
- 时间窗口
- 风险边界
- 场景前提
- 失效条件
- 必须再次通过的本地检查
- 更快本地事件对它的 supersede / revoke 机制

【结论 7：Decision Lease 至少要分三层】
- Intent Lease：方向/场景层
- Action Envelope Lease：交易框架层
- Execution Window Lease：极短时执行许可层（应更多由本地层派生）

【结论 8：必须引入 System Friction Score（系统摩擦分）】
这个概念非常重要，是本对话里讨论出来的关键“天才级抓手”之一。

它综合反映：
- CPU 压力
- RAM 压力
- DB 延迟
- REST 往返延迟
- WS lag
- 本地队列积压
- 本地计算延迟
- 网络 jitter / reconnect / 丢包

它不能只是监控指标，而必须直接影响：
- Thought Gate
- Query Budget
- Lease Validity

【结论 9：真正高效的方向不是“让 AI 更频繁说话”，而是让 AI 只处理结构变化】
AI 更像：
- regime architect
- exception interpreter
- capital aggressiveness advisor
- post-trade reviewer

而本地层负责：
- 速度
- 纪律
- 风控
- 时效
- 最终合法执行

4.4 中段：H0 本地确定性判断内核（Local Deterministic Judgment Core）被正式提出并落地
------------------------------------------------------------

这是 3 月 20 日对话里非常重要的一次方向升级。

在讨论 H / I 时，结论进一步变成：

真正强的 H / I，不能建立在“本地层空心化”的前提上。
因此正式主线在 H1 前，被进一步外扩成：

H0：Local Deterministic Judgment Core（本地确定性判断内核）

它的目标不是预测方向，而是先回答：
“当前这单/这个时刻，是否具备交易资格，是否仍有净优势，是否本地层已经具备足够的合法性、时效性和风险可控性。”

你当时要求“非常精巧准确且接近完美的思路与方案”，最终形成的 H0 关键抓手包括：

1. local_market_friction
- 不是看方向；
- 而是标准化市场摩擦、最小边际优势要求、滑点/成本/覆盖度；
- 为 risk_envelope 和 eligibility 提供上游基础。

2. local_risk_envelope
- 看当前仓位/挂单/执行数/kill switch/cooldown 等；
- 不是给策略，而是给“此刻能不能继续进入资格判断”。

3. local_trade_eligibility
- 把 market friction + risk envelope 合并成是否可进入 H1 的本地门禁。

4. H0 final audit
- 明确 H0 是否已 structurally valid and ready for H1。

这意味着项目主线在 3 月 20 日之后，不再只是：
H1 -> H2 -> H3 -> I

而是：

H0 本地确定性判断内核
-> H1 thought_gate
-> H2 query_budget
-> H3 model_router / compute governance
-> I decision lease

6.6 问题：H1-E provider_target / model_name 为空，导致 H1-F blocked_before_invocation
------------------------------------------------------------

  should_call_ai = True

  route_ready_light

  model_name = ""

- H1-G/H1-H/H1-I 随后全部不绿。

这是本对话后半段最关键的真实阻塞点之一。

根因有两层：

第一层：
- 你最初 export 的环境变量是：
  BYBIT_AI_ACTIVE_PROVIDER_TARGET
  BYBIT_AI_ACTIVE_MODEL_NAME
- 但 H1-E 实际读取的不是 MODEL_NAME，而是：
  BYBIT_AI_ACTIVE_MODEL

第二层：
- H1-R 的 route_a_light 虽然选中了 route A，
  但 H1-E 并不会自动把 route A 的 provider/model 绑定成请求 payload；
- H1-E 明确读取：
  BYBIT_AI_ACTIVE_PROVIDER_TARGET
  BYBIT_AI_ACTIVE_MODEL
- 也就是说，route decision 与 active provider/model env 是两层概念，不能混为一谈。

处理：
- 正确 export：
  export BYBIT_AI_ACTIVE_PROVIDER_TARGET=openai_native
  export BYBIT_AI_ACTIVE_MODEL=gpt-5-mini
- 然后通过 wrapper 重新运行：
  bybit_ai_request_envelope_builder.py
  bybit_ai_invocation_attempt_builder.py
  bybit_ai_response_check.py
  bybit_ai_governed_decision.py
  bybit_thought_gate_acceptance_suite.py
  bybit_thought_gate_regression_summary.py
  bybit_thought_gate_handoff.py
  bybit_thought_gate_final_audit.py

最终结果：
- H1-E request_state = ready_provider_native_ai_request
- H1-F invocation_state = invocation_success_json_ready
- H1-G response_state = response_json_contract_satisfied
- H1 final audit overall_ok = True

这个点必须被新对话牢牢记住：
【H1-E 使用的是 BYBIT_AI_ACTIVE_MODEL，不是 BYBIT_AI_ACTIVE_MODEL_NAME】
