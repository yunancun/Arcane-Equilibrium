二、项目总目标（再次定锚，防止新对话跑偏）
======================================================================

本项目不是普通自动下单 bot，也不是单纯的量化脚本集合。
真正目标始终是：

“构建一个以 OpenClaw 为中枢、Bybit 为主交易所、Binance 为辅助交易所的 AI Agent 交易系统，
长期追求在风险可控、权限可审计、成本可治理、行为可回放、边界可逐步放权的前提下，实现长期 net PnL 最大化。”

四条根原则仍然不变：

1. 看 net PnL，不看 gross PnL
必须计入 AI API 成本、手续费、基础设施、电费、维护成本等现实摩擦。

2. AI 输出不能直接当执行命令
AI 输出更适合理解为 governed observation 或 future decision lease，而不是“立刻下单”。

3. 权限不能因为时间自动升级
execution authority 必须靠验证、审计、表现逐步赢得，不允许偷渡放权。

4. 项目最终形态不是机械 bot，而是可长期进化、可学习、可自我观测、可被严格约束的交易个体。

5.4 当前大量红灯，本质是合法 no-call / 合法 idle account 被误判为失败
----------------------------------------------------------------------

这是今晚最重要的新思路，也是后续必须继承的设计结论。

当前链路里，至少以下部分存在大量“语义误报”：

- H1-G bybit_ai_response_check
- H1-H bybit_ai_governed_decision
- H1-I bybit_thought_gate_acceptance_suite
- H1 final audit
- H2 query_budget 部分脚本
- H4 compute_governor 部分脚本
- H5 ai_cost_governance 部分脚本

它们普遍沿用了旧假设：
“如果这轮没有真实 invocation / 没有 provider response / 没有 parsed json / 没有 token / 没有 latency，那就应该失败。”

但今晚已经确认：
这条假设不再成立。
因为当前最新链路里存在一种合法终态：

- route_plan = route_skip
- selected_ai_tier = none
- should_call_ai = false

这表示：
“本轮设计上本来就不应该调用 AI。”

因此：
- invocation_attempted = false
- provider_response_present = false
- parsed_json_present = false
- latency_ms = None
- input_tokens/output_tokens = None

在这种 no-call 终态下，不应该再被判失败。

同理，在当前阶段如果同时满足：
- read_only
- execution disabled
- 无持仓
- 无挂单
- 无执行历史

那么：
- 没有 recent execution
- 没有 order history
- private ws 只有 control message 而没有 business topic event

也应默认视为 expected idle account，而不是异常。

这就是今晚形成的核心语义升级：

1. 合法 no-AI-call 是有效 observation terminal state；
2. 合法 idle account 是 info/idle，不是 blocker/failure；
3. 不能为了“把红灯改绿”而伪造一笔 AI 调用或伪造一笔交易行为。

----------------------------------------------------------------------
7.1 合法 no-call 必须成为正式受支持的 observation terminal state
----------------------------------------------------------------------

这不是临时 workaround，而是设计升级。

过去系统隐含假设：
“如果 AI 没被调用，那就是失败。”

今天确认后，这条假设必须改为：
“如果 policy/router/selected tier 一致地得出 should_call_ai = false，
那么 no-call 就是合法终态，而不是失败态。”

这将直接影响：
- H1-G
- H1-H
- H1-I
- H1 final audit
- H2/H4/H5 的下游语义

这也是今天最关键的新思路之一。

----------------------------------------------------------------------
7.2 合法 idle account 必须降级为 info/idle，而不是 blocker
----------------------------------------------------------------------

在当前 read_only + disabled + 无持仓/无挂单/无执行 的真实阶段里，
“什么都没发生”经常就是正确状态。

所以后续要把以下内容从 blocker/warning 调整为 info/idle_flags（在满足静默态前提时）：
- no_open_positions
- no recent order history
- no recent execution history
- private ws control-only traffic

这条原则，将显著减少“系统没坏却一直报错”的语义噪音。

----------------------------------------------------------------------
7.3 未来 live 设计前，先把“语义误报”清掉，再看真实 blocker
----------------------------------------------------------------------

今天之后，后续主线不应再是：
“看到 H2/H4/H5 很红 -> 直接大修很多模块”。

而应是：
第一步，先清合法 no-call / legal idle 的误报；
第二步，再看 H1-B 的 budget_latency_cap_unacceptable 是否成为唯一或主要真实 blocker；
第三步，再决定是调整 budget/policy/latency 设计，还是继续下游章节。

这比以前的大范围猜测方式更科学，也更符合“先验证、后开发”的总原则。
