======================================================================
一、第一阶段总目标（这三天到底在做什么）
======================================================================

这三天工作的总体目标，不是让 Agent 直接开始交易，而是先把最底层、最关键、最容易埋雷的只读观察器基础设施打牢。

这三天围绕的是同一个核心逻辑：

1. 让 Agent 能可靠读取 Bybit 账户真实状态。
2. 把账户、持仓、订单、成交等私有数据标准化落地。
3. 建立持续、可验证的观察器链路，而不是一次性脚本。
4. 为以后进入 AI 判断、paper trading、live trading 做前置防呆、可追踪、可审计的基础设施。

换句话说，这三天不是在“做一个会交易的 Agent”，而是在“做一个不会乱交易、不会因为小细节就误判、而且每一步都有证据链的交易系统底座”。

六、第一阶段中途遇到过的主要问题，以及如何解决
======================================================================

1. Bybit 私有 REST 出现 retCode=10010（Unmatched IP）
原因：
- 公网出口 IP 与 Bybit API 白名单不匹配。
处理：
- 排查 trade-core 网络链路与公网出口。
- 修复 5GbE/VLAN 网络状态。
- 恢复正确公网出口后，私有 REST 全部恢复 retCode=0。

2. persistent WS 只有控制面消息，没有业务 topic 消息
现状：
- auth / subscribe success 持续发生
- 但 topic_message_count 始终为空
处理：
- 不做错误乐观假设
- 明确定义为 control_only / none_seen
- 在 packet / verdict / runtime / handoff 中全部显式表达

3. packet / verdict 时间语义不够严谨
现象：
- 不同来源文件时间不一致时，freshness 容易表达含糊
处理：
- 加入 source_refs
- 加入 packet_generated_ts_ms / freshness_reference_ts_ms / verdict_generated_ts_ms
- 强制 observer pipeline 先刷新 ws facts

4. snapshot payload 读取层级 bug
现象：
- 一度导致 account_summary 变成 null
处理：
- 检查 snapshot 顶层结构
- 修正读取到 payload.account 等正确层级

5. snapshot 文件时间与 payload 实际时间混淆
现象：
- 容易把旧 payload 当成新数据
处理：
- 新增 payload_time_summary
- runtime state 增加 payload_freshness

6. full cycle 输出不够干净
处理：
- 清理 wrapper stderr/marker
- 让 cycle 结果更适合审计与后续自动化使用

- 没有 persistent business-event ingestion
- 没有 event-driven state transitions

所以这三天的成果是：
- 第一阶段底座已经做得很像样。
- 但它是“稳固的起跑线”，不是“终点线”。
