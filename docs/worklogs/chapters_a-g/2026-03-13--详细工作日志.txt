四、网络故障插曲：5GbE Thunderbolt 网卡与 VLAN / DHCP 问题排查
——————————————————————————————————————

当天中途还处理了一个直接影响 Bybit 私有 API 可用性的底层网络问题。

问题现象：
- trade-core 接上 Ubiquiti 5GbE thunderbolt 网卡后，接口拿到了 192.168.30.30/24。
- default route 指向 192.168.30.1。
- 但无法 ping 通网关，也无法访问 1.1.1.1。
- ARP 显示 192.168.30.1 INCOMPLETE。
- 这说明不是驱动没识别，而是二层/三层链路没有真正打通。

排查过程：
1. 查看 ip -br addr / ip route / nmcli device status。
2. 确认接口 enx00242715937f 已经 UP，且 DHCP 领到地址。
3. 结合 Ubiquiti 截图确认：
   - trade-core 端口在 VLAN 30 上。
   - Netgear 设备已被彻底移除。
   - 但端口与配置切换后，NetworkManager 侧仍可能残留旧状态。
4. 执行：
   - nmcli device disconnect enx00242715937f
   - nmcli device connect enx00242715937f
5. 重连后再次测试：
   - ping 192.168.30.1 成功
   - ping 1.1.1.1 成功

结论：
- 不是驱动问题。
- 更像是 VLAN / 交换机端口切换后，本机网络状态没有完全刷新，重新 disconnect/connect 后恢复正常。
- 这一步很关键，因为此前 Bybit retCode=10010（Unmatched IP）问题与公网出口 IP 密切相关。

后续验证：
- 查询当前公网 egress IP 为 79.112.40.190。
- 再次测试 Bybit 私有 REST：
  - account / positions / order history / execution history 全部 retCode=0。
- 说明新出口 IP 与 Bybit API key 白名单匹配，私有只读访问恢复正常。

九、snapshot 时间语义修正
——————————————————————————————————————

1. 发现问题
- snapshot 文件的 ts_ms 是 snapshot 写入时间。
- 但 payload.account.check_time_utc / payload.account.response.time 等，可能来自更早的真实源数据时间。
- 如果只看 snapshot 文件时间，容易高估底层 payload 的新鲜度。

2. 解决方案
- 在 bybit_snapshot_to_postgres.py / snapshot 构建逻辑中，加入 payload_time_summary：
  - account_payload_ts_ms
  - positions_payload_ts_ms
  - order_history_payload_ts_ms
  - execution_history_payload_ts_ms
- 同时在 sources.* 下加入 payload_ts_ms。

3. acceptance v4 / runtime state v4 中使用 payload 时间
- acceptance v4 增加：
  - snapshot_version_v2
  - snapshot_payload_times_present
- runtime state v4 增加 payload_freshness：
  - account_payload_age_ms / state
  - positions_payload_age_ms / state
  - order_history_payload_age_ms / state
  - execution_history_payload_age_ms / state

意义：
- 这一步把“文件新”与“数据源新”区分开了。
- 对交易 Agent 尤其重要，因为后续一旦进入更高频策略，错把旧 payload 当新数据，风险很大。
