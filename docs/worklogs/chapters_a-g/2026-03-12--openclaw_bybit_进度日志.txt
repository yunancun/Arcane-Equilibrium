四、今日新确认/重申的总体设计目标
============================================================

今日再次明确并重申以下长期目标：

1. Agent 不应只是“会调接口”，而应具备：
- Learning Engine（越跑越聪明）
- Self-Observability Engine（感知自身瓶颈）
- Performance & PnL Reporting Engine（日/周/月经营报告）

2. Agent 未来应具备学习能力：
- 记住过去的判断
- 记住哪些判断事后是对/错
- 记住哪些市场环境适合某类判断
- 记住哪些 AI 调用浪费 token 且无收益
- 记住哪些模块常出错
- 形成：
  事实 -> 决策 -> 结果 -> 归因 -> 更新

3. Agent 未来应具备自我感知能力：
- CPU / RAM / 磁盘 I/O
- 网络延迟
- REST 请求耗时
- WS 稳定性
- DB 写入耗时
- 本地推理耗时
- 模块阻塞点
- 效率拖累来源
- 并能输出针对性建议

4. Agent 未来应具备经营报告能力：
- Daily / Weekly / Monthly summary
- 总收益 / 总亏损 / 净收益
- 手续费 / AI 成本 / 基础设施成本
- 问题归因
- 优化建议
- 且输出要一目了然

5. 这些能力不是边角增强，而应是后续主线的一部分。

六、今日出现的主要问题与修复
============================================================

1. secrets 路径误写
- execution history 脚本最初写成 /srv/settings/...
- 实际为 /home/ncyu/srv/settings/...
- 修正后恢复正常

2. 多次 cat <<'PY' / heredoc 输入时终端看似“卡住”
- 本质是 shell 仍在等待结束标记，或脚本本身仍在运行
- 后续需继续注意：
  - heredoc 结束标记必须单独占一行
  - 某些脚本本身有 20~40 秒的等待时间（尤其 WS smoke）

3. Ubuntu 24.04 下 pip 安装限制（PEP 668）
- 系统 Python 不能直接用 pip 安装 websocket-client
- 最终使用 venv 解决：
  /home/ncyu/srv/venvs/trading_ws

4. 标准化表重复插入问题
- 原因：同一 snapshot 重跑会重复 insert
- 通过去重、唯一索引、ON CONFLICT DO NOTHING 修复

5. 多次把终端输出又粘回 shell
- 导致 shell 将 JSON 字段、表格内容误当命令执行
- 属于终端操作层误操作，不是系统逻辑错误
- 后续继续注意只粘“命令块”，不要把输出再贴回 shell
