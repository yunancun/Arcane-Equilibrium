============================================================
一、项目概述
============================================================

1. 项目目标
- 构建一个以 OpenClaw 为中枢、以 Bybit 为主交易所、Binance 为辅助交易所的 AI Agent 交易系统。
- 系统目标不是简单自动下单，而是：
  1) 风险可控；
  2) 可审计；
  3) 可学习；
  4) 可逐步放权；
  5) 在成本、延迟、风控、权限治理前提下追求长期净收益最大化。
- AI 模型分工已定：
  - OpenAI：复杂决策、结构化输出、策略判断；
  - Claude：复杂长文本分析、研究、审阅；
  - 后续原则：不是“每次都问 AI”，而是“只在值得思考时调用 AI”。

2. 硬件与基础环境
- 主机：Ubuntu（trade-core）
- CPU/硬件方向：AMD AI MAX 395 + 128G 内存
- 还有 Synology NAS 可作为后续存储/归档支持
- Docker / Docker Compose 已部署
- OpenClaw 采用：主机安装 + Docker 沙箱思路

3. 当前总方向
- 交易所优先级：Bybit 主力，Binance 辅助
- 当前系统阶段：Bybit 只读（public + private readonly）已打通到余额/持仓/订单历史层面；尚未进入写权限与真实执行阶段
- 当前 Agent 权限：Level 0 / observer
- 当前执行状态：autonomous execution = false

============================================================
二、Brainstorm / 设计阶段演进过程
============================================================

1. 初始构想
- 用户希望构建一个更像“人”的 AI Agent，不只是机械执行，而是能够：
  - 自主分析；
  - 总结经验；
  - 学习与进化；
  - 在市场中长期优化自身行为。

2. 模型与平台架构的明确化
- 确定 OpenClaw 作为总编排层。
- 确定 OpenAI 用于复杂决策、结构化输出、策略判断。
- 确定 Claude 用于长文本分析、研究、审阅。
- 考虑过 ChatGPT / Kimi / DeepSeek / Claude 的组合，但实际落地目前以 OpenAI + Claude 为核心。

3. 成本概念的引入（关键转折点）
- 用户明确提出：Agent 不能只盯交易盈亏，必须考虑系统经营成本。
- 已纳入的成本概念包括：
  - 初始交易资金：500 欧元（或等值虚拟币）
  - 已投入基础设施：Ubiquiti 设备约 700 欧元
  - OpenAI API：30 欧元
  - Claude API：20 欧元
  - 运行电费：约 0.5 欧元 / 小时
  - 设备折旧：按电脑 3000 欧元、5 年折旧折算日成本
- 关键结论：
  - Agent 不能只优化 gross PnL；
  - 必须优化 net PnL（交易收益 - 模型成本 - 电费 - 折旧 - 手续费等）。

4. 延迟问题被提升为核心架构问题（关键转折点）
- 用户明确指出：AI 分析与给出结论存在延迟，而市场波动很多时候是瞬时的。
- 由此引出一个关键架构结论：
  - 不应把 AI 输出当作“即时命令”；
  - 应把 AI 输出变成“决策租约（decision lease）”；
  - 本地执行层根据最新市场状态、时效校验、风控校验做最后裁决。

5. 权限治理体系的重大升级（关键转折点）
- 原本曾讨论简单的阶段式放权，但后续升级为“按表现赢得权限”的框架。
- 最终关键结论：
  - 权限不按时间自动升级；
  - 权限应按可量化表现赢得；
  - 权限不应全局提升，而应在已证明有优势的子场景中局部提升；
  - 真正高水平的系统不是“更大胆”，而是“在优势区间更大胆，在非优势区间继续保守”。
- 最终形成 6 级权限体系：
  - Level 0：observer
  - Level 1：simulated_executor
  - Level 2：micro_live_probe
  - Level 3：constrained_profit_mode
  - Level 4：conditional_expansion
  - Level 5：controlled_evolution

6. API 调用治理的重大升级（关键转折点）
- 经过深入思考，形成了“不是每次都问 AI”的核心架构结论：
  - 先由本地层判断“是否值得调用 AI”；
  - 只有在值得的时候才调用大模型；
  - 调用本身也要计入成本收益分析。
- 由此形成新的架构链：
  1) 市场快照；
  2) thought_gate（是否值得调用 AI）；
  3) model_router（选便宜模型 / OpenAI / Claude / 双模型）；
  4) AI 输出 decision lease；
  5) pretrade_risk_gate；
  6) freshness / validity check；
  7) execution_gate；
  8) audit_logger；
  9) review_and_learning。
