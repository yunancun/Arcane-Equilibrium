==============================
0. 这份 handoff 的用途与可信度说明
==============================

这份文件的目标：
1) 先把“本对话里已经完成了什么、顺序是什么、返修了什么”尽量完整固定下来，防止后续继续遗忘。
2) 作为新聊天接手时的主线说明书，让后续继续工作的版本能迅速对齐当前真实状态。
3) 把本对话里已经确认过的路径、env、secret、执行方式、关键坑点，集中记录下来。

重要说明：
- 本文件不参考更早上传给模型的旧总结文件；它只基于本对话、当前还可见的终端结果、以及用户在本对话里对整体时间线的明确纠正来重建。
- 因为本对话里后半段可见信息主要集中在“H 章节返修”，所以对更早那段“I 章节设计与完成”的具体逐脚本细节，当前只能按“本对话已被用户明确确认的事实”记录，不能伪造未重新看到的具体代码文件名。
- 也就是说：
  A. “H 先完成 -> 讨论 I -> I 也做完 -> 准备去 J 前发现问题 -> 回头修 H” 这条总线，是用户在本对话中明确纠正并确认过的，应视为当前分支真实主线。
  B. I 章节的精确脚本清单 / 文件名 / 每一步终端输出，当前仍需要后续专门在代码树和运行产物里反查定位。


==============================
1. 本对话的真实时间线（非常重要）
==============================

必须先纠正一件事：
这条对话并不是“今晚从 H 做到 I”。
真实顺序应当是：

1) 今天前半段，先把 H 章节完成。
2) 随后，围绕 I 章节应该怎么写，进行过专门讨论。
3) 用户当时特别提醒：关于 I 章节的讨论内容，要认真记录，之后会来要。
4) 在那之后，I 章节也已经完成。
5) 然后在准备继续推进、也就是准备从 I 往后走、接近 J 章节之前，发现 H 章节底层仍有问题。
6) 因此，本对话后半段的主要工作，变成了“回头返修 H”，把 H 从底层重新修绿、重新闭环、重新验证。

所以：
- 本对话后半段看到的大量终端输出，大多不是“第一次做 H”，而是“返修 H”。
- I 章节不是未来要做，而是本对话更早阶段就已经做完。

这条总线必须牢记，否则后续会把项目位置判断错。


==============================
2. 本对话里一共完成了哪些大块工作
==============================

按主线可分为四大块：

--------------------------------
2.1 Bybit 接入 / 权限边界 / 只读策略的前置规划
--------------------------------

本对话较早阶段已经明确过 Bybit API 权限策略，核心原则是“最小权限、先只读、后执行、职责分离”。

当时形成的方向包括：
- 现在优先创建生产只读主 key，用于真实私有读取与核验。
- 真正执行用的 key 不在当前阶段立刻启用，待更后面受控执行阶段再考虑。
- demo / testnet key 可作为额外演练，但不是当前主链重点。
- 权限上偏向：Read-Only、Unified Trading、Assets、Trade 历史 / Orders / Positions / Convert 等与当前阶段相关的读取能力。
- 不建议乱开与本项目当前阶段无关权限（Fiat、P2P、Ads、Broker only 等）。
- 网络安全上，如果后续具备固定公网 IP，应优先切到白名单；无固定 IP 时，当前只能临时放宽但不应视作长期最佳方案。

这一段为整个系统定下了安全总原则：
“当前主线是只读观察系统，而非执行系统。”

--------------------------------
2.2 H 章节最初版本已完成
--------------------------------

用户已明确纠正：今天前半段不是在修 H，而是先把 H 做完。

H 章节的结构已经明确固定为：
- H1 thought gate
- H2 query budget
- H3 model router
- H4 compute governor
- H5 AI cost governance

H 章节的总目标：
- 把 AI 放入一个受治理、只读、预算受控、可审计的观察链。
- 它可以接收本地事实并返回 compact JSON 观察结果。
- 但它绝不能被误认为“已授予执行权”。

在 H 章中反复被固定下来的硬边界：
- system_mode = read_only
- execution_state = disabled
- execution_authority = not_granted

也就是说，H 的角色是：
“AI 观察判断层”，不是“执行授权层”。

--------------------------------
2.4 在进入更后章节前发现问题，因此回头返修 H
--------------------------------

本对话后半段的绝大多数终端工作，实际都属于这一步。

逻辑上不是：
“因为 H 没做，所以才做 H”。

- H 做完了。
- I 也做完了。
- 准备继续往后推进时，发现 H 的底层并没有真正稳固，或者存在表面通过但内核有问题的点。
- 所以决定回头返修 H，把底座修干净，再继续往后。

这一决策本身很重要，说明项目推进原则是：
“宁可回头修基础，不带着脏底座继续叠层。”

==============================
4. 本对话里反复固定下来的关键概念
==============================

--------------------------------
4.1 H 不是执行系统，而是受治理的 AI 只读观察链
--------------------------------

H 可以：
- 接受本地 deterministic facts
- 经过 thought gate / budget / router / governor
- 调用 compact JSON AI observation
- 对 response 做 contract check
- 形成 governed observation

H 不可以：
- 授予 live execution authority
- 把 AI 返回当作执行许可
- 跳过 operator review
- 发出 decision lease（那是更后面的事）

--------------------------------
4.2 H 的硬安全边界必须始终保持
--------------------------------

贯穿本对话反复确认的边界：
- system_mode = read_only
- execution_state = disabled
- execution_authority = not_granted
- live_execution_allowed = false
- decision_lease_emitted = false

这几条是 H 的地基。不能在后续工作中被误删或误解。

--------------------------------
4.3 I 的角色不是“让 H 更聪明”，而是“把 H 的观察结果转成受治理的下一层权限对象”
--------------------------------

虽然当前还没把 I 的具体代码树反查出来，但从本对话总线与 H->I 接口语义来看，I 的角色应当是：
- 承接 H 的 governed observation
- 管理 execution authority 的授予与不授予
- 引入 decision lease / operator review / expiry / revoke 等机制
- 把“观察结果”与“执行权”彻底分层

--------------------------------
4.4 route 与 active model 当前是两层绑定
--------------------------------

当前实现下：
- route selector 选的是 lane（route_a / route_b / route_c / skip）
- request envelope 绑定的是 active provider/model

所以不能简单以为：
“route_a_light 被选中 -> 就一定自动使用 route A model 变量”。

当前真正影响 H1-E 的变量还是：
- BYBIT_AI_ACTIVE_PROVIDER_TARGET
- BYBIT_AI_ACTIVE_MODEL

--------------------------------
4.5 系统已经验证过 read-only AI 调用链能真实跑通
--------------------------------

最终成功打通的 active binding 是：
- provider_target = openai_native
- model_name = gpt-5-mini

并且：
- invocation_success_json_ready
- response_json_contract_satisfied
- H1 final audit green

这说明“真实只读 AI 观察链”不是纸面设计，而是已实际跑通过。
