OpenClaw / Bybit 项目接手摘要（A-I 闭环后）
生成日期：2026-03-22
用途：供新对话快速接手，避免把 shadow-only 状态误判为 live-ready

==================================================
一、项目当前总定位
==================================================

当前系统不是 live trading infrastructure，也不是 autonomous live trading ready。
当前应准确表述为：

production-grade readonly observer
+ governed AI observation chain
+ shadow decision control plane

这意味着：
1. 主系统已具备较高完整度的只读观测、审计、治理、影子控制平面能力。
2. 但并未授予执行权限。
3. 也没有发出可被真实消费的 execution lease。
4. 任何后续推进都必须先验证、后开发，不能把 shadow 逻辑误当成 live path。

==================================================
二、当前关键安全状态
==================================================

当前关键 runtime / authority 状态：

- system_mode = read_only
- execution_state = disabled
- execution_authority = not_granted
- decision_lease_emitted = false
- live_execution_allowed = false

这四条是整个项目当前最重要的硬边界：
- 不得越过 read_only
- 不得隐式启用 execution
- 不得隐式提升 execution_authority
- 不得把未发放的 decision lease 说成可 live consume

==================================================
八、当前项目准确结论
==================================================

可以说：
- A-I 在 read_only / shadow / governance 范围内已经测试闭环完成
- 当前系统可以作为 production-grade readonly observer + governed AI + shadow decision control plane

不可以说：
- 已经 live-ready
- 已经可以自动实盘下单
- 已经可以 autonomous trading
- 已具备 live execution authority
- 已具备可消费的真实 execution lease

更准确的表述应为：
“当前系统已经完成 A-I 阶段的只读观测、治理 AI、影子控制平面闭环，为 future live design 做好了结构准备，但仍未进入 live execution 实施阶段。”

==================================================
九、后续主线不应做什么
==================================================

新对话中，以下动作不应直接进行：

1. 不要直接写 live order submitter
2. 不要直接接入真实下单 adapter
3. 不要跳过 authority grant 设计
4. 不要把 approval packet 当成 live approval
5. 不要把 operator ack shadow 当成 live operator approval
6. 不要把 shadow candidate 当成可执行 lease
7. 不要在 freshness / pricing / timing / accounting 未核验前推进 live
8. 不要因为结构都“像已经完整了”就默认能上线

==================================================
十、后续主线应该做什么
==================================================

下一阶段正确主线应该是：

第一优先级：live design 前置条件核验
- 核验 freshness 真正闭合方式
- 核验 recent trade 字段补全策略
- 核验 runtime reference old 的来源与解决路径
- 核验 provider pricing table 如何正式绑定主线
- 核验 latency / ttl / consume timing 是否达到真实 live 需要

第二优先级：先设计 contract，再写实现
应先设计：
- authority grant contract
- manual approval contract
- live approval transition contract
- paper execution gate
- simulator execution gate
- pre-trade risk gate
- post-trade reconciliation
- accounting / audit trail
- kill switch / revoke path
- execution adapter contract
- failure recovery contract

第三优先级：路线顺序
推荐：
1. fresh/live design prerequisites audit
2. paper execution design
3. paper execution adapter + reconciliation
4. supervised live design
5. live authority grant model
6. live execution adapter
7. constrained live pilot

==================================================
十一、给新对话中的模型的工作方式要求
==================================================

新对话必须遵守：

1. 先验证，后开发
2. 所有结论优先基于 latest artifacts / latest json / final audit
3. 不得把 shadow-only 误说成 live-ready
4. 不得隐式提升 execution authority
5. 先 contract，后 implementation
6. shell 命令必须分小步给，一次 1-2 步，等用户反馈后再继续
7. 先做核验清单与设计清单，再动 live 相关代码

==================================================
十二、如果新对话第一轮要做什么
==================================================

建议新对话第一轮只做四件事：

A. 复述当前准确状态
B. 列出 live design 前必须先核验的前置条件
C. 给出新的章节树 / build order
D. 只给 1-2 步 shell 检查命令，不要一次给很多命令

==================================================
十三、最关键的一句话总结
==================================================

当前 OpenClaw / Bybit 项目，A-I 已完成只读观测、治理 AI、影子控制平面的正式闭环；
它已经为 future live design 做好结构准备，但仍明确处于 read_only / disabled / not_granted / no-live-submit 状态，
任何后续推进都必须围绕“live design 前置条件核验与契约设计”展开，而不是直接进入实盘执行开发。
