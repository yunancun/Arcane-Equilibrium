三、项目总目的（绝对不能再遗忘）
============================================================

本项目不是一个普通自动交易脚本，也不是一个单纯会下单的 bot。

本项目真正要做的是一个：
“以 OpenClaw 为中枢、Bybit 为主交易所、Binance 为辅助交易所的 AI Agent 交易系统。”

长期目标：
1. 风险可控；
2. 可审计；
3. 可学习；
4. 可逐步放权；
5. 在成本、延迟、风控、权限治理都受控的前提下，追求长期净收益最大化。

几个绝对不能忘的根原则：

A. 不是只看 gross PnL，而必须看 net PnL
- 交易收益不是唯一目标；
- 必须同时考虑 AI API 成本、电费、设备折旧、手续费、基础设施成本、未来维护成本；
- 所以本项目本质上不是单纯“交易系统”，而是“交易系统 + 经营系统”。

B. AI 输出不能当成即时命令
- AI 有延迟；
- 市场会变化；
- 因此 AI 更适合输出 decision lease（决策租约），而不是立即执行命令；
- 本地层必须继续做 freshness / timing / risk / permission / execution gate。

C. 权限不能按时间自动升级，必须按表现赢得
- 不能因为系统运行久了就自动放权；
- 不能全局放权；
- 必须在已验证有效的子场景中局部放权；
- 真正高级的 Agent 不是更莽，而是在优势区间更大胆、在非优势区间更保守。

D. 项目最终形态不是机械 bot，而是长期进化型交易个体
- 会观察；
- 会判断；
- 会审计；
- 会学习；
- 会自我观测；
- 会产出经营报告；
- 会在严格边界下逐步赢得更高权限。

四、Agent 核心需求（正式口径）
============================================================

A. 安全与治理
- 默认 read_only；
- 默认 execution disabled；
- 默认 AI 不可直接跨过本地 gate 驱动真实执行；
- 任何阶段都必须可审计、可解释、可回放。

B. 数据感知能力
- 读取真实交易所 public / private readonly 数据；
- 建立统一 snapshot；
- 接入 WS；
- 区分控制面消息与真实 business-topic 事件；
- 维护 freshness / health / runtime state。

C. 决策输入能力
- 把账户、持仓、订单、成交、WS 状态、business event 状态加工成结构化输入；
- 先做好本地输入治理，再谈 AI。

D. AI 调用治理能力
- 不是每次都问 AI；
- 只有值得时才问；
- 必须有 thought gate / query budget / model router / compute governor；
- AI 成本必须纳入 net PnL。

E. 学习能力
- 未来必须形成：事实 -> 决策 -> 结果 -> 归因 -> 更新；
- 要能记住哪些判断有效、哪些 AI 调用浪费、哪些环境适合哪些策略。

F. 自我观测能力
- 未来要观测 CPU / RAM / 网络 / REST / WS / DB / 模块阻塞点 / 本地推理延迟等。

G. 经营报告能力
- 未来要能输出 daily / weekly / monthly 级别的收益、成本、净收益、问题归因、优化建议。

十四、概念词典（新聊天最容易接不住的概念）
============================================================

- readonly observer：当前主系统的真实角色：只读观察，不允许执行。
- healthy_no_business_events_yet：业务事件侧健康，但当前还没有看到真实 wallet/order/execution/position 业务事件。这是健康空态，不是故障。
- healthy_but_empty：event-driven scaffold 当前高层语义：结构健康，但为空，不能推进到交易态。
- observe_only_retained：D23/F 章收口结论：系统合理地继续保持 observe-only，而不是因为出错。
- decision lease：未来 AI 输出的受治理决策租约，不是即时执行命令。
- thought_gate：本地层先判断“是否值得问 AI”的门。
- query_budget：控制 AI 调用预算，防止无限询问。
- model_router v2：根据任务价值、复杂度、成本去路由模型。
- structural hardening：不是正式业务完工，而是先把结构层、测试层、审计层补强到可维护状态。
- chapter consistency：章节级一致性检查，用于确认某章所有 latest/contract/summary/handoff/final audit 语义一致。
- master regression：跨章节总回归，统一检查 D/G/J/K 是否同时成立。
