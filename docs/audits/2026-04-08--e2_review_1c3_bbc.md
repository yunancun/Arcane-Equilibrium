## Blocking / Must-Fix summary (top)

- **M-1 (1C-3-B-2, test gap):** The IPC-layer guards in `handlers.rs::ForceGovernorLooser` (reason_code whitelist, CB/MR lockout, single-step enforcement, 24h cooldown) are **not exercised by any unit test**. The four `ipc_server` "dispatch" tests use `setup_governor_override_channel` — a fake event-consumer that bypasses the guard code entirely and returns canned Ok/Err strings. The "cooldown rejection" test is a fake, not a real cooldown check. This means a refactor could silently remove every guard and the test suite would still go green. Before any live consideration this must be fixed with real `TickPipeline`-driven tests that invoke `handle_paper_command` directly. **Not blocking for 1C-3-D**, blocking for live.

- **M-2 (1C-3-B-2, audit hole):** `spawn_governor_audit_row` is only invoked on the `Ok(Ok(Ok(json_str)))` success branch in both `handle_force_governor_tighter` and `handle_force_governor_looser`. **Rejected attempts leave no audit trail.** For principle #8 "交易可解釋 — every risk-touching change must be explainable and auditable", a denied operator attempt at de-escalation (e.g. cooldown reject, bad reason_code, CB lockout) is exactly the kind of event that needs a V014 row — otherwise an operator can probe the guards invisibly. Fix: audit both branches, add a `result: "rejected"` / `error` field to the payload. **Not blocking for 1C-3-D**, blocking for live.

Everything else below is a nit.

## Commit 3 — `9f46b06` 1C-3-B-2

### Threat-model walkthrough of the guards (what I was asked to verify)

**Source of truth for the guards:** `rust/openclaw_engine/src/event_consumer/handlers.rs` `PaperSessionCommand::ForceGovernorLooser` arm (diff lines 207–282 of commit).

Guards claimed by commit message:
1. reason_code whitelist
2. 24h cooldown
3. single-step only
4. CB/ManualReview cannot be unlocked via IPC

#### Guard 1 — reason_code whitelist
- `handlers.rs`: `if !TickPipeline::VALID_DE_ESCALATION_REASONS.contains(&reason_code.as_str()) { return Err(...) }`.
- Whitelist constant in `tick_pipeline.rs`: `["false_positive", "root_cause_fixed", "accept_risk"]`.
- Case-sensitive (unlike `parse_risk_level` which is case-insensitive). `"False_Positive"` would be rejected. Inconsistent but on the safe side — rejects more, not fewer. **Nit N-8**: align to lowercase normalize, or document the asymmetry.
- Checked BEFORE `parse_risk_level` → fail-fast. ✓
- Python side `RiskViewClient.force_governor_tier_looser` does NOT pre-validate — forwards whatever string came in. Correct: server-side is the authority. ✓
- **Enforcement status: REAL, but NO real test** (see M-1).

#### Guard 2 — 24h cooldown
- Constant: `GOVERNOR_DE_ESCALATION_COOLDOWN_MS = 24 * 60 * 60 * 1000 = 86_400_000`. ✓
- Check in `handlers.rs`:
  ```
  if let Some(last) = pipeline.last_governor_de_escalation_ms() {
      let elapsed = now_ms.saturating_sub(last);
      if elapsed < TickPipeline::GOVERNOR_DE_ESCALATION_COOLDOWN_MS { return Err(...) }
  }
  ```
- **TOCTOU analysis:** check-then-set happens inside `handle_paper_command`, which is the single consumer of the `paper_cmd_rx` mpsc channel in the event_consumer task. Commands are processed **strictly sequentially** — `while let Some(cmd) = rx.recv().await`. Two concurrent `force_governor_tier_looser` IPC calls get serialized at the channel receiver. **No TOCTOU.** ✓
- **Demo/live caveat documented by author:** in-memory only; resets on engine restart. On every boot, operator gets a free de-escalation. Documented in-commit and in tick_pipeline field docstring. Demo-acceptable. **Must be PG-persisted before live** (already in 1C-4 backlog per commit message). ✓
- **Enforcement status: REAL, but tested only by `setup_governor_override_channel` fake** (see M-1).

#### Guard 3 — single-step only
- `handlers.rs`: `if (current.value() as i32) - (target.value() as i32) != 1 { return Err(...) }`.
- Requires exact delta of +1 in current-minus-target direction. Rejects jumps (e.g. `Defensive → Normal` = delta 3) AND same-level (delta 0) AND reverse (negative delta).
- Tighter path uses `target - current != 1` symmetrically.
- `RiskLevel.value()` ordering verified in `risk_gov.rs` — `Normal=0, Cautious=1, Reduced=2, Defensive=3, CircuitBreaker=4, ManualReview=5`. Delta arithmetic holds. ✓
- **Enforcement status: REAL, not directly unit-tested** (see M-1).

#### Guard 4 — CB / ManualReview lockout
- `handlers.rs`: `if current >= RiskLevel::CircuitBreaker { return Err(...) }`. Covers both CB and MR via PartialOrd. ✓
- **Defense in depth — SM layer:** `lookup_rule(CircuitBreaker, Defensive)` returns `allowed: OP_ONLY, requires_approval: true`. The SM call `de_escalate_to(level, "operator_ipc", ...)` passes `Some("operator_ipc")` as `approved_by`, so approval check passes — the lookup_rule path does NOT block CB unlock on its own. **The IPC-layer guard is the ONLY real lockout.** ✓ — acceptable since the commit message is honest about it ("cannot be unlocked from the GUI on purpose"), but means Nit M-1 (test gap) is more important: if the IPC `if current >= CircuitBreaker` line is ever removed, the SM will happily accept it.
- **Enforcement status: REAL, single-line, not unit-tested** (see M-1).
