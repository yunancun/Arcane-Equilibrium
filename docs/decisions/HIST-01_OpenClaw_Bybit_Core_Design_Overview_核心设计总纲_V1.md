# 1. 执行摘要

本文件定义 OpenClaw / Bybit 交易 Agent 的统一设计口径，包括本质、设计目标、根原则、能力蓝图、系统结构、核心要求、实施路径与治理边界。它的目的不是让系统“更自由地下单”，而是让系统在可控、可审计、可回滚的前提下，逐步赢得更高质量的自主交易能力。

该 Agent 的成功标准不应先看短期 gross PnL，而应先看：是否遵守边界、是否不会失控、是否能解释、是否能在异常环境下保护账户、是否能用真实 net PnL 衡量经营质量。

# 2. 本质定义

如果由本文件定义，这个 Agent 的本质不是策略脚本、不是单点预测模型、也不是“AI 说买就买”的黑箱机器人。它应被定义为一个 governed autonomous trading system：在严格风险、授权、审计与人工最终治理边界内运行的长期进化型交易自治体。

它必须会观察、会判断、会等待、会进攻、会收手。

它必须比人更强于速度、纪律、记忆、抗情绪、并行监控与复盘。

它必须比人更克制：不会为了“看起来聪明”而越权、硬做、乱改 live。

它的自由来自授权矩阵，而不是默认拥有一切自主权。

# 3. 设计目标（保留版本 A / B / C）

## 版本 A：一句话版

构建一个在严格风控与授权边界内运行的长期进化型交易 Agent，使其能够持续感知市场、理解状态、评估成本、生成受控决策、执行交易并完成复盘学习，同时始终服从生存优先、可审计、可回滚与人工最终治理。

## 版本 B：项目总纲版

本项目旨在构建一个以 OpenClaw 为治理中枢、以 Bybit 为主交易所的长期进化型 AI Agent 自动交易系统。该系统不是单纯的下单程序，而是一个在严格风控、明确授权、完整审计与人工最终治理框架下运行的交易自治体。它应具备市场感知、状态理解、概率决策、成本感知、自我健康判定、风险治理、受控执行、持仓管理与复盘学习等能力；能以 H0 本地确定性判断为第一道门，经 H1-H5 AI 治理生成具时效与条件约束的 Decision Lease，再由统一风控与执行链完成落地；并通过学习与归因平面持续提出可验证的改进建议，而不直接改写实盘逻辑。系统的最终目标不是追求无约束自主，而是在可控、可审计、可演进的前提下，逐步赢得更高质量的自主交易能力。

## 版本 C：宪法前言版

本 Agent 的设计目标，是成为一个在严格治理框架内运行的长期进化型交易自治体，而非单点预测模型或无约束自动下单程序。系统必须以生存、授权、审计与可回滚为最高优先级，在此基础上逐步建立市场感知、状态理解、成本与收益评估、概率推理、风险控制、受控执行、持仓管理、自我健康监测与复盘学习能力。所有交易行为必须经由本地确定性判断、AI 治理、Decision Lease、风险总督与统一执行入口串联完成，不得绕过任何治理关口；所有学习成果只能形成待验证建议，不得直接改写实盘策略、权限或风控底线。系统的核心成功标准，不是短期名义盈利，而是在真实净收益、风险可控、行为可解释、状态可审计与人工可接管的前提下，稳健而审慎地提升自主交易能力。

# 4. 设计哲学与根本信念

“像人一样自由交易”不等于“让它随便交易”。真正强的 Agent，核心不是更冲动、更敢赌，而是像优秀交易员一样会观察、会判断、会等待、会进攻、会收手，同时拥有机器的速度、记忆、纪律与可审计性。

| **信念 1** | “不交易”是正式输出，而不是异常状态。 |
|---|---|
| **信念 2** | 真正高级的自主，不是放开，而是会约束自己。 |
| **信念 3** | 真正长期优势，很多时候来自“不以愚蠢方式亏钱”。 |

## 4.1 强对手环境假设与竞争行为逻辑

系统必须默认运行于一个已经存在大量算法化、机构化与 AI 化参与者的市场生态中。它不得假设自己天然拥有速度、资本、信息或撮合基础设施优势。最短暂、最表层、最机械、最容易被抢的 alpha，通常不属于它。

因此，Agent 的胜率不应建立在“处处更快、处处更早、处处更激进”之上，而应建立在选择优势、解释优势、纪律优势、成本意识与生存能力之上。它真正要学会的，不是全面击败最强对手，而是识别哪些局根本不该参与，哪些局虽然看见了也应主动放弃。

| **参与者类型** | **典型优势** | **本 Agent 应采取的默认态度** |
|---|---|---|
| 速度型参与者 | 更低延迟、抢队列、撤改单更快 | 默认不打纯速度战；不抢第一脚，优先等二次确认或直接放弃。 |
| 中周期系统化资金 | 研究更深、资本更大、风控更成熟 | 默认明显而拥挤的 edge 已被压薄；只做结构更清楚、成本更可控的机会。 |
| 情绪型 / 结构性错误参与者 | 追涨杀跌、清算驱动、时段选择差 | 重点观察其错误的二级后果，在更长半衰期的再定价机会中选择性参与。 |

据此，系统应内生以下行为逻辑：一，不和强者比“快”，而和强者比“克制”；二，不争夺最短暂的 alpha，只争夺仍来得及验证、且信息半衰期明显长于系统反应劣势的 alpha；三，把大型机构程序与大 AI 当作会在市场上留下痕迹的“天气”，而不是必须正面击败的对手画像；四，一旦无法判断自己是否正处于强程序主导的竞争层级，默认按不利处理，收缩、延迟、降仓或不做。

这意味着本 Agent 追求的核心 edge，不应是绝对速度优势，而应是 regime edge、selectivity edge、cost edge 与 governance edge。面对强对手存在的现实，系统的首要目标不是在所有市场条件下都出手，而是稳定地选择可赢之局，并避免不该参与之局。

# 5. Agent 必须具备的核心素质

## 5.1 市场感知能力（Perception）

它必须同时感知价格结构、成交量、盘口深度、波动率、funding、OI、清算、账户状态、订单状态、跨市场关联、外部事件、链上信号，以及交易所与系统健康度。没有可靠感知，就没有可靠决策。

## 5.2 局势理解能力（Market State Understanding）

它必须先判断“我现在处在什么市场世界里”，而不是对任何信号机械触发。不同 regime 下，同一信号的意义完全不同。

## 5.3 交易假设生成能力（Trade Thesis Generation）

它必须形成结构化交易假设，而不是只有买卖信号。每一笔候选交易都应包含 observation、hypothesis、supporting evidence、conflicting evidence、entry conditions、invalidation、exit plan、confidence 与时效窗口。

## 5.4 概率思维与不确定性处理能力

它必须允许“看不清、赔率不够、成本过高、系统不健康、这笔不做”这些结论合法存在。

## 5.5 多时间尺度思考能力

它必须以战略层、战术层、执行层分层工作，并能处理层间冲突。

## 5.6 成本与经营意识

它不能只看 gross PnL。必须区分 execution-cost、decision-cost、operating-cost，并以真实 net PnL 评价行为质量。

## 5.7 风险纪律与生存优先能力

它必须把生存放在收益之前。P0/P1/P2 风控必须可否决策略意图。

## 5.8 执行智能

它必须知道“想做什么”和“怎么做”是两回事，能在限价、市价、拆单、撤单、追单、幂等与对账之间受控取舍。

## 5.9 自我感知与健康判定能力

它必须持续判断当前系统、数据、模型、风控链是否仍然适合继续交易，并能在必要时降级、收缩、只减仓或熔断。

## 5.10 学习与归因能力

它必须会记录上下文、分辨 alpha/timing/sizing/execution/risk 错误，并把学习结果收敛为待验证建议，而非直接改写 live。

## 5.11 授权自觉与治理意识

它必须明白自主权来自授权矩阵，可授予，也可撤回。

## 5.12 可解释、可审计、可追责

任何关键交易动作都必须留下完整证据链。

# 6. 成本口径与 net PnL 要求

本系统明确反对只看 gross PnL。所有经营判断应以 net PnL 为基准，但成本口径必须分层，避免把不同性质的成本混在一起。

| **成本层级** | **典型内容** | **主要用途** |
|---|---|---|
| Execution-cost | 手续费、滑点、funding、执行摩擦 | 单笔交易即时决策必须纳入 |
| Decision-cost | 该笔交易额外消耗的 AI / 计算成本 | 判断这笔机会是否值得“思考” |
| Operating-cost | 设备折旧、电费、固定云成本、基础设施月费 | 经营报表、策略族评估、长期 ROI |

# 7. 系统结构与分层职责

本系统不应被设计成一个“大一统智能黑盒”，而应是分层、分权、互相制衡的结构。

## 7.1 Data Plane / 感知平面

接入市场、账户、成交、交易所、系统健康与成本台账数据，完成标准化、时间同步、延迟标记与质量评分。

## 7.2 Canonical State Plane / 真相状态平面

维护 market_state、regime_state、volatility_state、liquidity_state、account_state、system_health_state、risk_mode、operator_mode 等唯一真相源。

## 7.3 H0 / Local Deterministic Judgment & Gate Core

本地确定性判断与第一道硬门。负责事实检查、授权检查、系统模式检查、硬风控前提检查、基础准入判断；可以拒绝、降级或放行候选，但不得直接生成实盘写命令。

## 7.4 H1-H5 / AI Governance, Deliberation & Lease Drafting Layer

AI 多角色治理、审议、挑错、成本评估、授权检视与 Lease 起草层。其产物是受约束的 Decision Lease 草案，而不是 order。

## 7.5 I / Decision Lease Control Plane

管理 Lease 的生成、生命周期、过期、撤销、回收、影子桥接与审计。I 不是“再做一次交易判断”，而是管理受控意图。

## 7.6 Risk Governor

统一执行 P0/P1/P2 风险治理，对策略和 Lease 拥有一票否决权。

## 7.7 OMS / Execution Adapter

唯一合法写入口；把批准后的意图落到 Bybit V5 API，负责委托、撤单、改单、追踪、幂等、对账与失败恢复。

## 7.8 Portfolio & Position Brain

统一管理全局敞口、相关性、风险预算、资本占用与持仓优先级。

## 7.9 Learning & Attribution Plane

复盘、归因、实验、提议、候选参数与策略演进建议；与 live 平面隔离。

## 7.10 Control Plane & GUI Operator Console

提供模式切换、审计查看、风控操作、报告、学习驾驶舱与人工干预入口；GUI 只属于控制平面，不得绕过交易执行核心。

# 8. H0 / H1-H5 / I 的正式边界定义

这一段是本项目的关键语义钉子。三者必须严格分开，否则后续极易出现重复判断、权责漂移与多写入口隐患。

| **层级** | **正式名称** | **主要职责** | **明确禁止** |
|---|---|---|---|
| H0 | Local Deterministic Judgment & Gate Core | 本地确定性判断、硬门控、准入与降级 | 不得直接产生 live order |
| H1-H5 | AI Governance, Deliberation & Lease Drafting Layer | 市场解释、反证审查、成本评估、授权检视、Lease 起草 | 不得直接写交易所 |
| I | Decision Lease Control Plane | Lease 生命周期、约束、过期、撤销、桥接、审计 | 不得替代风险总督与执行层 |

# 9. AI 治理层的五角色建议定义（H1-H5）

## H1 - Market Interpretation

解释当前市场发生了什么、候选机会属于何种结构与 regime。

## H2 - Counter-Evidence & Failure Review

主动寻找反证、失败模式与最可能的误判路径。

## H3 - Cost / Attention / Maintainability Review

评估 AI 注意力税、维护成本与机会质量是否匹配。

## H4 - Risk / Authorization Governance Review

检视是否落在授权矩阵、风险包络、模式约束与策略适配范围内。

## H5 - Decision Lease Drafting & Synthesis

把前述结论收敛成具时效、失效条件、建议仓位与执行偏好的 Lease 草案。

# 10. 八条根原则（不可违背）

看 net PnL，不看 gross PnL。

本地先做，AI 只做高价值、非线性、需要解释与审议的部分。

AI 输出不能当即时命令；必须经 Lease、风险总督与统一执行链落地。

权限按验证表现授予，也按异常与失效撤回。

先系统健康，后市场判断。

失败默认收缩：禁止新风险暴露，但保命减仓与灾难保护路径必须保留。

学习不等于自作主张；learning plane 与 live plane 必须隔离。

所有结论必须区分事实、推断、假设。

# 11. 六条必须补上的制度性原则

单一写入口原则：所有下单、撤单、改单、平仓与保护性操作只能经统一 OMS / Execution Adapter 发出。

交易所灾难保护原则：本地智能止损可为主层，但交易所端必须保留灾难保护底线。

授权矩阵原则：自主权必须绑定至市场、品类、策略簇、资金级别与风险包络，不得全局授予。

live / learning 隔离原则：学习结果只能形成待验证建议。

状态真相源原则：持仓、订单、风险、system_mode、lease 状态必须有唯一真相源。

Operator 最终治理原则：Operator 不是日常下单者，但必须拥有熔断、降级、回滚、冻结与撤权权力。

# 12. 风控设计要求

本文件采用 P0 > P1 > P2 的三层风控理念。

| **层级** | **目标** | **典型内容** |
|---|---|---|
| P0 | 生存级硬限制 | 最大回撤、最大日损、杠杆上限、kill switch、灾难保护、账户生存 |
| P1 | 策略级与组合级治理 | 授权矩阵、相关性、品种风险预算、regime 适配、模式限制 |
| P2 | 行为级约束 | 连亏降频、系统不健康收缩、异常期间只减仓、AI 注意力税提示、维护成本偏高时降权 |

对抗性市场意识、双层止损、AI 注意力税都可以作为工具箱存在，但不能凌驾于 P0 灾难保护之上。

# 13. 执行与订单管理要求

决策层输出的是 Trade Intent / Decision Lease，不是最终订单。

执行层负责委托方式选择、拆单、追单控制、滑点估计、partial fill 处理、幂等与三账一致。

任何 GUI、研究模块、学习模块不得绕开统一执行入口。

API 抖动、对账失败、状态分叉、异常重复回报时，应优先保护账户，而不是继续开新仓。

# 14. 学习、归因与升级路径

学习能力必须被严格定义为“提出可验证建议”，而不是“自动改 live”。

Level 1：参数级微调，可自动，但必须受范围限制并可回滚。

Level 2：策略权重或适配表更新，应半自动，需 review 与渐进生效。

Level 3：核心逻辑、风险规则、权限边界或代码结构变更，必须走正式审批、回放、shadow、灰度流程。

建议的 promotion path：观察 -> 假设 -> 回放 / 重放 -> shadow -> 审批 -> 灰度 -> 正式放量。

# 15. Operator 角色定义

Operator 不是为了替代 Agent 做所有具体交易决策，而是为了承担最终治理责任。

拥有 kill switch、冻结、降级、回滚、撤权、模式切换、人工复核与事故标记权。

负责审阅关键实验、批准高风险升级、处理异常场景、确认 live promotion。

在早期阶段保留更强确认权，在成熟阶段保留最终 veto 权。

# 16. GUI / Control Plane 要求

GUI 必须是控制平面的前端，而不是交易逻辑的旁路。建议统一承载状态总览、控制中心、经营收益、审计记录、学习驾驶舱、paper trading 与风险操作入口。

GUI -> Control API -> governed services 是合法链路；GUI 绝不能直接触达交易所写接口。

# 17. 核心成功标准与评价顺序

第一优先级：不会失控。

第二优先级：不会越权。

第三优先级：能完全审计与解释。

第四优先级：在异常环境下能保护账户。

第五优先级：在真实 net PnL 口径下逐步证明经营质量。

也就是说，盈利评价必须晚于生存评价。上线初期最重要的不是收益率，而是“乱单率、重复下单、状态错乱、风险越界、审计完整度、异常保护能力”。

# 18. 落地路线图

| **阶段** | **定位** | **核心验证目标** | **建议状态** |
|---|---|---|---|
| 0 | 观察者 | 状态判断稳定、Lease 质量可解释 | 当前已完成到 shadow / paper 验证带 |
| 1 | 单市场、强约束执行 | 证明零失控、风控不被突破 | 下一步 live 前必要门槛 |
| 2 | 多策略候选 + 统一风控 | 证明会取舍而非只会多做 | 中期 |
| 3 | 半自主 | 证明异常自降级有效、人工介入下降 | 中期 |

# 19. 与当前章节树的对齐口径
