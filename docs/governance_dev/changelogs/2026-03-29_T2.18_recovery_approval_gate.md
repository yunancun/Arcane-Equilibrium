## Overview / 概览

### English
Implemented formal Recovery Approval Gating system for the OpenClaw/Bybit trading system. This module enforces the governance principle that state machine recovery operations (de-escalation/relaxation) require explicit Operator approval, while escalation (moving to more conservative states) proceeds automatically without approval.

The system provides:
- 6 recovery types covering Authorization and Risk Governor state machines
- Formal request and approval dataclasses with full audit traceability
- Mandatory observation period support for certain recoveries
- Thread-safe gate engine with complete lifecycle management
- 100% test coverage (56 tests)

## Key Features / 关键特性

### 1. Formal Recovery Types / 正式恢复类型

```python
AUTH_UNFREEZE       # FROZEN → RESTRICTED/ACTIVE (授权解冻)
AUTH_RESTORE        # RESTRICTED → ACTIVE (授权恢复)
RISK_DEESCALATE     # Escalation ladder downgrade (风险降级)
RISK_UNFREEZE       # CIRCUIT_BREAKER/DEFENSIVE → less conservative (风险解冻)
INCIDENT_RESOLVE    # Close incident recovery path (事故解决)
TRADING_RESUME      # Resume trading after freeze (交易恢复)
```

### 2. Request Submission / 请求提交
- Anyone can submit recovery requests
- Requests start in PENDING state
- Include reason, observation period requirements, and supporting evidence
- All submissions audited

### 3. Operator Approval / 运营商批准
- Only Operator role can approve/reject
- Approval includes conditions and observation period configuration
- Observation periods enforce gradual recovery (from DOC-07 § 9.2)
- Optional observation period override at approval time

### 4. Observation Period Management / 观察期管理
- Some recoveries require mandatory observation periods
- Observation status: NOT_REQUIRED, PENDING, COMPLETED, FAILED
- Automatic tracking of observation window (start_ms, end_ms)
- Can mark observation as failed if violation occurs during window

### 5. Query Interface / 查询接口
- `get_pending_requests()` - All unapproved requests
- `get_approved_requests()` - All approved requests
- `get_requests_by_type()` - Filter by recovery type
- `get_pending_observations()` - Requests in observation period
- `check_observation_period()` - Check specific observation status
- `get_stats()` - Processing statistics

### 6. Thread Safety / 线程安全
- Full lock-based synchronization
- Safe concurrent submissions and approvals
- Safe concurrent queries

## Governance Alignment / 治理对齐

### From DOC-07 (Incident Policy) / 来自 DOC-07（事故政策）

**§9.1 Recovery Not Default Action**
- System does not auto-recover; explicit approval required
- All recovery conditions must be met before approval
- Observation periods prevent uncontrolled recovery

**§9.2 Recovery Gradation**
- Recoveries follow stepwise progression: CIRCUIT_BREAKER → DEFENSIVE → REDUCED → CAUTIOUS → NORMAL
- Gate enforces observation periods between steps
- No cross-level jumps without conditions

**§9.3 Recovery Approval Rights**
- Operator must approve recovery in these cases:
  - critical_incident recovery
  - True source integrity issues
  - Authorization reopening
  - Unified write entry anomalies
- Gate implements this by gating all approvals through Operator role

### From SM-01 (Authorization State Machine) / 来自 SM-01（授权状态机）

**§11 Authorization Recovery Principles**
- Recovery not default after freeze/restriction
- Must verify: frozen cause eliminated, risk mode permits, incident processed, audit complete
- Conservative recovery: FROZEN → RESTRICTED before FROZEN → ACTIVE
- Gate supports both AUTH_UNFREEZE and AUTH_RESTORE types with separate workflows

### From SM-04 (Risk Governor State Machine) / 来自 SM-04（风险管理状态机）

**§11 Observation Window Requirements**
- Any relaxation must consider observation window
- Within observation: no further relaxation beyond approved level, higher audit density
- Observation end: no repeated issues, root cause gone, chains stable, no Operator objection
- Gate implements observation tracking and completion checks

## Usage Example / 使用示例

### Basic Recovery Flow / 基础恢复流程

```python
from app.recovery_approval_gate import (
    RecoveryApprovalGate,
    RecoveryType,
)

# Initialize gate with audit callback
def audit_callback(event):
    print(f"Audit: {event['event_type']}")

gate = RecoveryApprovalGate(audit_callback=audit_callback)

# 1. System detects incident resolved, submits recovery request
request = gate.submit_recovery_request(
    recovery_type=RecoveryType.INCIDENT_RESOLVE,
    from_state="DEFENSIVE",
    to_state="REDUCED",
    requested_by="System:incident_policy",
    reason="Incident contained, entering observation",
    observation_period_hours=24,
    evidence={"incident_id": "inc:12345"},
)
print(f"Request submitted: {request.request_id}")

# 2. Operator reviews and approves with conditions
approval = gate.approve_recovery(
    request_id=request.request_id,
    approved_by="Operator:nancun",
    conditions=[
        "No new incidents during observation",
        "Health metrics stable",
    ],
    observation_period_hours=24,
    notes="Incident recovery approved, starting 24h observation",
)
print(f"Approval granted: {approval.approval_id}")

# 3. System monitors observation period
obs_status = gate.check_observation_period(request.request_id)
print(f"Observation status: {obs_status.value}")

# 4. If violation during observation, mark as failed
if violation_detected:
    gate.mark_observation_failed(
        request_id=request.request_id,
        reason="Near-miss detected during observation",
    )
```

### Complex Multi-Type Recovery / 复杂多类型恢复

```python
# Authorization recovery (separate from risk recovery)
auth_req = gate.submit_recovery_request(
    recovery_type=RecoveryType.AUTH_UNFREEZE,
    from_state="FROZEN",
    to_state="RESTRICTED",
    requested_by="System:incident_policy",
    reason="Authorization freeze lifted",
    observation_period_hours=24,
)

# Risk de-escalation (separate track)
risk_req = gate.submit_recovery_request(
    recovery_type=RecoveryType.RISK_DEESCALATE,
    from_state="DEFENSIVE",
    to_state="REDUCED",
    requested_by="System:incident_policy",
    reason="Risk conditions improving",
    observation_period_hours=12,
)

# Operator can approve both independently
gate.approve_recovery(request_id=auth_req.request_id, approved_by="Operator:nancun")
gate.approve_recovery(request_id=risk_req.request_id, approved_by="Operator:nancun")
```
