### Core Classes

#### 1. ProtectiveOrderType (Enum)
```python
HARD_STOP_LOSS      # Absolute defense; cannot be disabled (DOC-01 §5.9)
SOFT_STOP_LOSS      # Conditional; can be adjusted
TAKE_PROFIT         # Profit target; can be disabled
TRAILING_STOP       # Dynamic stop following price
POSITION_CLOSE      # Close entire position at trigger
EMERGENCY_CLOSE_ALL # Circuit breaker: close all positions
```

**Safety Invariant:** HARD_STOP_LOSS cannot be cancelled or disabled by any Agent. This is constitutional-level enforcement per DOC-01 §5.9.

#### 2. ProtectiveOrderStatus (Enum)
```python
CREATED        # Just created
ARMED          # Monitoring active
TRIGGERED      # Trigger condition met
EXECUTED       # Action completed on exchange
CANCELLED      # Cancelled before trigger
FAILED         # Execution failed
EXPIRED        # Expired due to time or manual override
```

#### 3. ProtectiveOrderConfig (Dataclass)
Configuration template for protective orders. Used to standardize parameters across positions and strategies.


#### 4. ProtectiveOrder (Dataclass)
Single instance of a protective order. Represents a specific protection on a specific position.

**Key Fields:**

- `atr_value`, `random_offset`: Anti-hunt parameters
- `trailing_high`, `trailing_distance`: Trailing stop tracking

#### 5. ProtectiveOrderManager (Engine)

**Core Methods:**

##### Creation & Tracking
```python
create_protective_order(
    symbol: str,
    side: ProtectiveOrderSide,
    order_type: ProtectiveOrderType,
    entry_price: float,
    trigger_price_pct: float,
    quantity: float,
    position_id: Optional[str] = None,
    strategy_id: Optional[str] = None,
    atr_value: Optional[float] = None,
    random_offset_pct: Optional[float] = None,
    tags: Optional[Dict] = None,
) -> ProtectiveOrder
```

Creates and registers a protective order. Automatically:
- Generates unique order_id with "pord_" prefix
- Sets timestamp in milliseconds
- Calculates trigger price based on side (long: below entry, short: above entry)
- Enforces `can_be_disabled=False` for HARD_STOP_LOSS
- Fires audit callback

##### Trigger Evaluation
```python
check_triggers(
    market_state: Dict[str, Dict[str, float]],
    skip_soft_stops: bool = False,
) -> ProtectiveOrderCheckResult
```

Evaluates all protective orders against current market prices. Call on every market tick or reasonable frequency.

**Logic:**
- Skips terminal-status orders (EXECUTED, CANCELLED, EXPIRED, FAILED)
- Optionally skips soft stops in low-confidence markets
- Checks trigger condition per order type:
  - HARD/SOFT_STOP_LOSS: price touches trigger level
  - TAKE_PROFIT: price reaches target
  - TRAILING_STOP: price falls below trailing high - distance
  - POSITION_CLOSE: always trigger
  - EMERGENCY_CLOSE_ALL: always trigger
- Fires audit callback on trigger
- Returns ProtectiveOrderCheckResult with:
  - List of triggered orders
  - List of unprotected positions
  - Coverage percentage

##### Execution
```python
execute_protective_action(
    order: ProtectiveOrder,
    market_state: Dict[str, Any],
) -> bool
```

Executes a protective action (callback when trigger fires).

**Process:**
1. Validate order is in TRIGGERED or ARMED status
2. Invoke `on_execute_callback` (delegated to execution layer)
3. Set order status to EXECUTED
4. Generate exchange_order_id
5. Fire audit callback
6. Return True on success, False on failure

##### Coverage Validation
```python
validate_coverage(

get_unprotected_positions(
```

Per DOC-01 §5.9: All positions must have at least a hard stop-loss.

Returns:
- `is_valid`: True if all positions protected
- `missing`: List of position IDs without hard stops

##### Cancellation (with Hard Stop Protection)
```python
cancel_order(

```

**Enforcement:** HARD_STOP_LOSS orders cannot be cancelled. Returns False and logs audit event if attempted.

## Safety Features

### 1. Hard Stop Non-Disablability (DOC-01 §5.9)
```python
if order.order_type == ProtectiveOrderType.HARD_STOP_LOSS:
    return False  # Cannot cancel
```

Hard stops are constitutional-level protection. No Agent can disable them.

### 2. Stop Concealment (EX-01 §4.2)
Orders are created in CREATED status and monitored locally. They are NOT placed on the exchange order book until triggered. This prevents market makers from hunting stops.

Implementation detail:
- `check_triggers()` monitors locally
- `execute_protective_action()` places on exchange only after trigger
- WebSocket disconnection > 30s triggers fallback: place emergency hard stop on exchange

### 3. ATR-Dynamic Distance + Random Offset (EX-01 §4.3)
```python
order.atr_value = atr  # Store ATR at creation
order.random_offset = random_offset_pct  # Add randomness
```

Makes stop levels unpredictable. Combined with regime-aware adjustments, prevents algorithmic stop-hunting.

### 4. Thread-Safety
```python
self._lock = threading.RLock()
with self._lock:
    # All operations protected
```

Concurrent order creation, trigger checks, and cancellations are thread-safe.

### 5. Complete Audit Trail
Every operation fires audit callback:
- `protective_order_created`
- `protective_order_triggered`
- `protective_action_executed`
- `protective_action_failed`
- `protective_order_cancelled`
- `hard_stop_cancel_rejected` (enforcement)
- `coverage_validation_failed`
- `emergency_close_all`
- `manager_state_restored`

### 6. Unprotected Position Detection
```python
unprotected = manager.get_unprotected_positions(open_positions)
if unprotected:
    # Auto-assign hard stops to all unprotected positions
```

Prevents drift: ensures no position ever exists without mandatory protection.

## Usage Example

```python
from app.protective_order_manager import (
    ProtectiveOrderManager,
    ProtectiveOrderType,
    ProtectiveOrderSide,
)

# Initialize with callbacks
audit_log = []
execution_queue = []

def audit_callback(event_type: str, details: dict):
    audit_log.append((event_type, details))

def exec_callback(order, market_state):
    execution_queue.append(order)

manager = ProtectiveOrderManager(
    audit_callback=audit_callback,
    on_execute_callback=exec_callback,
)

# Create protective orders
order = manager.create_protective_order(
    symbol="BTCUSDT",
    side=ProtectiveOrderSide.LONG_POSITION,
    order_type=ProtectiveOrderType.HARD_STOP_LOSS,
    entry_price=42000.0,
    trigger_price_pct=5.0,  # Stop 5% below entry
    quantity=1.0,
    position_id="pos_123",
)

# Validate coverage before trading
is_valid, missing = manager.validate_coverage([
    {"position_id": "pos_123", "symbol": "BTCUSDT"},
])

# Monitor on market tick
market_state = {"BTCUSDT": {"price": 39000.0}}
result = manager.check_triggers(market_state)

if result.triggered_orders:
    for order in result.triggered_orders:
        manager.execute_protective_action(order, market_state)

# Emergency: close all positions
manager.emergency_close_all({}, reason="CIRCUIT_BREAKER")

# Serialize state
snapshot = manager.to_dict()
manager.export_json()  # JSON backup
```

## Known Limitations

1. **Manual Regime Detection:** ATR adjustment not yet automated. Guardian agent will call `create_protective_order()` with regime-aware ATR values.

2. **Exchange Order Placement:** This implementation tracks protective orders but delegates actual exchange-side placement to the execution layer. The `on_execute_callback` receives the order; the caller is responsible for placing the exchange-side order.

3. **Realistic Stop Prices:** The trigger_price calculation is simplified. In production, account for:
   - Bid-ask spread
   - Slippage on market order
   - Exchange minimum order size
   - Maker/taker fee asymmetry

4. **Liquidation Handling:** Not yet integrated with margin monitoring. When margin gets critical, emergency_close_all() should be called automatically.

## Appendix: DOC-01 §5.9 (Full Text)

> **5.9 交易所端灾难保护原则 / Exchange-Side Disaster Protection Principle**
>
> 本地智能止损可作为主层保护，但交易所端必须始终保留灾难保护底线。当本地系统完全失效时，交易所端预挂的条件单是账户生存的最后一道防线。
>
> **English:** Local smart stop-loss serves as primary protection, but the exchange side must always maintain a disaster protection baseline. When the local system fails completely, pre-staged conditional orders on the exchange are the account's last survival line.

This implementation fully realizes this principle through:
1. Local monitoring (check_triggers() per tick)
2. Hard stop enforcement (cannot be disabled)
3. Fallback to exchange-side orders (on trigger)
4. Complete auditability (every decision logged)
5. Emergency fail-safe (emergency_close_all())
