## 3. AI 在交易管線中的角色 / AI's Role in Trading Pipeline

### 3.1 設計意圖（Design Intent）

```
市場數據 → H0 本地判斷（L0，必經）
         → H1 thought_gate（是否需要 AI？）
           → H2 budget（預算夠嗎？）
             → H3 model_router（選哪個模型？）
               → AI 推理（regime 識別、策略評估、風險評估）
                 → H4 governor（AI 結果可信嗎？）
                   → H5 cost_logging（記錄成本）
                     → I Decision Lease（帶時效的決策許可）
                       → 執行
```

### 3.2 當前實際狀態（Current Reality）

```
市場數據 → H0 本地判斷（✅ 運作中）
         → [H1-H5 全部跳過]
         → strategy_orchestrator（純本地策略 L0）
           → pipeline_bridge → paper_trading_engine
         → I Decision Lease = shadow-only（不發射）

AI 治理層存在但從未被調用。
交易決策 100% 基於本地確定性計算（MA 交叉、布林帶、RSI 等）。
```

### 3.3 GAP 分析：設計 vs 現實

| 能力 | 設計目標 | 當前實現 | Gap |
|------|----------|----------|-----|

## 5. AI 接入路線圖 / AI Integration Roadmap

### 5.1 當前決策（CLAUDE.md 記錄）

> **決策：win_rate > 20% 前不接入 AI 咨詢（C1/I1/A1），避免在隨機決策上疊加 AI 成本**

這是正確的決策。理由：
1. 當前勝率 0%（162 round trips, 0 wins — 截至 Session 8），本地策略本身需要先優化
2. 在壞策略上加 AI 只會增加成本而不改善結果
3. Session 12 F1/F2 修復（fill 碎片化 + 注意力稅最低 edge）可能顯著改善勝率，需要時間驗證

### 5.2 建議接入順序（Phase 1 後）

```
第一步：AI 輔助 Regime 識別（低風險、高價值）
  - 替換 signal_generator 中的本地 regime 規則
  - 使用 L1.5（Haiku）進行快速分類
  - 頻率：每 5 分鐘一次（隨掃描器），成本可控
  - 預算：~$5/天 上限

第二步：AI 輔助入場評估（中風險）
  - strategy_auto_deployer 部署前，AI 評估機會質量
  - 使用 L1.5→L2 降級策略
  - 頻率：每次新信號（~10-50/天），成本需控制
  - 預算：~$10/天 上限

第三步：AI 輔助持倉管理（中風險）
  - 持倉期間 AI 定期評估是否繼續持有
  - 整合注意力稅機制
  - 使用 L1.5（常規）+ L2（異常時升級）
  - 預算：按持倉數量動態調整

第四步：完整 H1-H5 治理鏈啟用（高複雜度）
  - 啟用 thought_gate → budget → model_router → governor → cost_logging
  - 需要 Decision Lease 從 shadow 轉為 active
  - 需要 Operator 明確授權
```
