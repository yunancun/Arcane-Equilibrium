## 0. 背景

DOC-08 定義了四層計算路徑（L0-L2），其中 L1 層（本地 Ollama）負責三大核心任務：Regime 分類、機會篩選、情緒打分。DOC-08 撰寫時假設本地 AI 能力有限，僅規劃了這三項任務。

然而，當前硬體環境已超出 DOC-08 的原始假設：

| 計算單元 | 設備 | 架構 | DOC-08 是否考慮 |
|----------|------|------|-----------------|
| Agent 1 | CPU (32核) | x86_64 | ✅ 已考慮 |
| Agent 2 | GPU Radeon 8060S | gfx1151 (RDNA 3.5) | ⚠️ 僅提及硬體，未啟用加速 |
| Agent 3 | NPU RyzenAI-npu5 | aie2p (XDNA 2) | ❌ 完全未考慮 |

ROCm 已安裝，三個計算單元全部識別成功。本文分析在 L1 三大核心任務之外，本地 AI 還能承擔哪些工作，以及對系統的實際價值。

## 1. 三大計算單元分工策略

```
┌──────────┬────────────────────┬────────────────┬──────────────────────┐
│  單元    │  擅長              │  資源成本      │  適合任務            │
├──────────┼────────────────────┼────────────────┼──────────────────────┤
│ NPU      │ 低功耗持續推理，   │ ≈0（獨立功耗   │ 常駐監控型、         │
│ XDNA 2   │ 不搶 CPU/GPU 資源  │  域）          │ 小模型分類           │
├──────────┼────────────────────┼────────────────┼──────────────────────┤
│ GPU      │ 批量並行矩陣運算   │ 與 CPU 共享    │ 7B-14B 主力模型推理  │
│ RDNA 3.5 │                    │ 統一記憶體     │                      │
├──────────┼────────────────────┼────────────────┼──────────────────────┤
│ CPU      │ 通用計算、         │ 與 GPU 共享    │ L0 確定性計算 +      │
│ 32 核    │ 低延遲單線程       │ 統一記憶體     │ 交易系統主進程       │
└──────────┴────────────────────┴────────────────┴──────────────────────┘
```

關鍵洞察：NPU 擁有獨立的功耗域和計算通道，跑推理不會搶 CPU/GPU 資源，也不受 DOC-08 §2.3 的 `MemoryMax=12G` / `CPUQuota=150%` 限制。這意味著可以在 NPU 上跑額外的常駐任務，完全不影響交易系統。

## 3. L1 之外的 7 個擴展場景

### 3.1 高價值 / 低風險（建議立即規劃）

#### ① 審計日誌異常偵測（NPU 常駐）

- **現狀：** `audit_log.py` 只做 append-only 寫入，沒有即時分析
- **方案：** 用 1-3B 小模型跑在 NPU 上，持續掃描審計日誌流，偵測異常模式（如：短時間內大量 RESTRICTED→ACTIVE 轉換、非預期的 authorization 變更）
- **治理價值：** DOC-08 §12 安全不變量要求「審計鏈不可被靜默繞過」——AI 監控是最佳防線
- **資源影響：** NPU 獨立運行，對交易系統零影響
- **計算單元：** NPU (XDNA 2)

#### ② L1.5 降級替代——Haiku 二次確認本地化

- **現狀：** DOC-08 §3 定義 L1.5 用 Haiku (~$0.001/次) 做「L1 結果二次確認」，每天 10 次
- **方案：** GPU 加速後 7B 模型完全能在 3 秒內完成二次確認，無需雲端調用
- **治理價值：** 完全消除 L1.5 Haiku 的雲端依賴。省錢是次要的，關鍵是離線穩健性——斷網時 L1→L1.5 降級鏈不會斷
- **資源影響：** 與 L1 共享 Ollama 實例，無需額外模型
- **計算單元：** GPU (RDNA 3.5)
- **成本節省：** ~$3/月

#### ③ Perception Data Plane 增強——Fact vs Inference 自動標記

- **現狀：** `perception_data_plane.py`（587 行）做 fact/inference 標記，目前是規則驅動
- **方案：** 用 7B 模型對邊界情況做語義判斷——「這個信號到底是事實觀察還是推斷？」
- **治理價值：** DOC-08 的治理精神是「未標記的推斷不應觸發交易」，AI 判斷比硬規則更準確
- **資源影響：** 搭車現有 L1 調用，每次額外 20-30 tokens
- **計算單元：** GPU (RDNA 3.5)

### 3.2 中等價值（Phase 2-3 引入）

#### ④ 交易歸因分析（Trade Attribution）

- **現狀：** `trade_attribution.py` 記錄 round-trip，只做記錄不做分析
- **方案：** 每筆交易完成後，7B 模型生成結構化歸因報告——「這筆盈利的主因是 RSI 信號還是 regime 判斷？」
- **治理價值：** 為 Phase 4 的 L2-L5 學習系統提供結構化輸入數據
- **資源影響：** 非即時，可在低負載時段（亞洲 00:00-08:00 UTC）批量處理
- **計算單元：** GPU (RDNA 3.5)

#### ⑤ 策略參數自然語言解釋

- **現狀：** 5 個策略（MA crossover、BB breakout 等）的參數調整沒有可解釋性
- **方案：** 生成如「當前 BB breakout 策略的 width=2.0 在 ranging regime 下偏寬，建議收窄到 1.5，原因是...」的解釋
- **治理價值：** 提升 Operator 決策效率，最終審批人需要理解「為什麼 AI 建議這樣調」
- **資源影響：** 按需觸發
- **計算單元：** GPU (RDNA 3.5)

#### ⑥ 新聞搜索結果本地預處理（替代部分 Perplexity）

- **現狀：** DOC-08 §6 搜索降級體系中，Tier 2 是「本地 Ollama + web-pilot」，Tier 3 是「本地 Ollama 獨立分析」
- **方案：** DuckDuckGo 抓取的原始新聞 → 本地 7B 做摘要+情緒打分 → 只有高置信度的才送雲端深度分析
- **治理價值：** 降低對外部 API 的依賴，提升離線能力
- **資源影響：** 每 30 分鐘一次
- **計算單元：** GPU (RDNA 3.5)
- **成本節省：** ~$3/月

### 3.3 探索性（Phase 4+ 考慮）

#### ⑦ GovernanceHub 決策解釋生成

- **現狀：** GovernanceHub（852 行）做跨 SM 級聯，但決策路徑不透明
- **方案：** 每次 governance 決策後，生成自然語言解釋鏈——「此訂單被拒絕，因為 SM-01 授權狀態為 RESTRICTED，觸發源是 SM-04 風險升級事件」
- **治理價值：** Phase 5 生產化時，審計和監管合規需要可解釋的決策鏈
- **資源影響：** 低頻觸發
- **計算單元：** GPU (RDNA 3.5)

## 4. 量化影響

### 4.1 雲端成本影響

| 編號 | 擴展任務 | 計算單元 | 月雲端節省 | 治理合規價值 |
|------|----------|----------|-----------|-------------|
| ① | 審計異常偵測 | NPU | $0 | 極高 |
| ② | Haiku 二次確認本地化 | GPU | ~$3 | 高 |
| ③ | Fact/Inference 標記 | GPU | $0 | 高 |
| ④ | 交易歸因分析 | GPU | $0 | 中 |
| ⑤ | 參數自然語言解釋 | GPU | $0 | 中 |
| ⑥ | 新聞本地預處理 | GPU | ~$3 | 中低 |
| ⑦ | 治理決策解釋生成 | GPU | $0 | 中 |

- 原 DOC-08 月成本估算：~$17/月
- 優化後月成本估算：~$11/月（節省 ~35%）

### 4.2 離線穩健性提升

擴展後，系統在完全斷網情況下仍能運行到 L1.5 水準（原設計只能降級到 L1）。這是 DOC-08 §12 安全不變量「交易系統永遠不等 AI，AI 是增強層不是依賴層」的最強保障。

## 5. 模型配置建議

| 部署位置 | 模型 | 用途 | 記憶體佔用 |
|----------|------|------|-----------|
| GPU (Ollama) | `qwen2.5:7b-instruct-q5_K_M` | L1 三大核心 + ②③④⑤⑥⑦ 全部 GPU 擴展任務 | ~5.5GB |
| NPU | 待調研 ONNX 小模型 (1-3B) | ① 審計異常偵測（常駐） | ~1-2GB |

GPU 上只需一個 7B 模型即可覆蓋所有場景——不同任務靠 prompt 路由區分，不需要多個模型。NPU 小模型需後續調研 AMD XDNA 2 的 ONNX Runtime 支援情況。

## 7. 風險與注意事項

1. **NPU 生態成熟度：** AMD XDNA 2 的 ONNX Runtime 支援仍在早期階段，需要實測驗證穩定性
2. **資源隔離仍需遵守：** 所有 GPU 擴展任務共享 DOC-08 §2.3 的 `MemoryMax=12G` 限制，不可超出
3. **不可改變 L0 確定性：** 擴展任務僅為增強層，DOC-08 §1 的降級規則不變——Ollama 崩潰即退化到 L0
4. **如與《項目憲法》衝突，以憲法為最高約束**
