**3 最關鍵 finding**：
1. **Bybit V5 doc 不顯式 permit PostOnly+reduceOnly 並行**：官方 create-order doc 只各別描述兩 flag，無 explicit「合法組合」明文。社區 SDK（tiagosiebler/bybit-api、ccxt）+ OpenClaw `CreateOrderRequest` 兩 flag 是 orthogonal struct field（order_manager.rs:140, 381），實證可並用，但 BB 立場必須要求字典手冊 §1.2 顯式記錄這個組合的合法性與 reject 行為。
2. **api-demo endpoint PostOnly + reduceOnly 行為 doc gap**：`https://bybit-exchange.github.io/docs/v5/demo` 明文「demo 不是完整功能複製」+「not every API is available」，但未顯式列 PostOnly close 行為。AMD §3 Phase 2a Demo 7d + Phase 2b LiveDemo 7d 都是 api-demo endpoint，這個 doc gap 是真實 PASS criteria 風險（若 demo 對 PostOnly + reduceOnly 有不為人知的 silent degradation → 7d 樣本不代表 Mainnet 行為）。
3. **AMD §5.4 "TooManyPending → 全域 5min pause" mitigation 在 Bybit rate-limit 模型下偏弱**：Bybit V5 Order group 20 req/s per UID（字典 §4.1 + WebFetch 確認），不是「每 5 min 解凍」式 sliding window；rate-limit 風暴觸發 retCode=10006 IpRateLimit + EC_ReachMaxPendingOrders 都是 sub-second 級復原，5min global pause 過於保守會 starve close path。建議 mitigation 改 dynamic backoff（rate utilization > 80% 才 throttle）。

## §1 PostOnly + reduceOnly 組合合法性

**Bybit 官方狀態**：WebFetch `create-order` doc + changelog last 6mo 均未顯式 permit 或 prohibit 此組合，亦未列任何 reject reason 串「EC_PostOnlyCannotBeReduceOnly」之類字串。社區實證（tiagosiebler/bybit-api master、ccxt master）兩個 flag 完全 orthogonal 並用無限制。

**OpenClaw 端實證**：`order_manager.rs:140` `reduce_only: Option<bool>`、`:381` 直接 `body["reduceOnly"] = ...` 寫入 request body；`CreateOrderRequest` struct（字典 §1.2 line 308-327）`time_in_force` 與 `reduce_only` 並列為 independent field。OpenClaw entry side maker 沒帶 reduceOnly（合理），close side 必須 `reduceOnly=true`（spec §4.2 寫死）。組合在 Rust 結構層 100% 支援。

**BB 結論**：技術可行，但**字典手冊 SSOT 缺顯式條目**——這是 BB ship-stop level 的 governance gap：未來 audit 若有人質疑「PostOnly+reduceOnly 是否合法」會找不到權威依據。

**必修 1 (HIGH)**：字典手冊 §1.2 `place_order` 段或 §4.3 已知陷阱新增條目：
> 「PostOnly + reduceOnly 並用合法：兩個 flag 在 Bybit V5 `POST /v5/order/create` request body orthogonal（無互斥）。若限價已過市仍 reject 為 `EC_PostOnlyWillTakeLiquidity`（與 entry 同），不是 reduceOnly 衍生 reject。」

## §2 Cancel rate limit 與風暴風險

**Bybit V5 Order group 20 req/s per UID**（字典 §4.1，2026-04-20 EDGE-P2-3 Phase 1B-1 確認；WebFetch 復驗一致）；該 group cover `/v5/order/create + /v5/order/cancel + /v5/order/amend + /v5/execution/*`，**create 和 cancel 共用 quota**。

**算術核驗 AMD 估算**：
- Spec §1.1 demo grid 7d 203 active closes ÷ 7d ÷ 24h ≈ **1.21 closes/h** 全 grid
- 25 symbols × ~5 策略 = ~125 (strategy, symbol) cell，但同 symbol 一般 ≤1 pending close
- 30% maker timeout fallback rate × 1 cancel + 1 market re-dispatch = 0.36 req/close 額外
- PostOnlyWillTakeLiquidity reject rate（entry side empirical baseline 4 天 178 case 後 ~0%，close side 未知但 spec 預估 < 5%）
- Worst-case 1h 並發：~2 close/h × 25 symbols × (1 create + 0.3 cancel + 0.3 market re-dispatch) ≈ **~80 req/h** 全帳戶，遠低於 20 req/s = 72,000 req/h cap

**Burst window 風險**：5s sweep cycle 內若 25 個 symbol 同時 timeout → 25 cancel + 25 market re-dispatch = **50 req / 5s = 10 req/s**，仍 50% 餘裕。**Rate limit 不是真實 ship-stop**。

**但 AMD §5.4 mitigation 設計過保守**：「TooManyPending → 全域 5min pause」會在單次 rate-limit-adjacent 觸發後 5 min 內 starve close path → close 全走 market → 失去整個 maker 優化的價值。Bybit rate-limit recovery 是 100ms 級（sliding 1s window），5 min 是 3000x overshoot。

**必修 2 (HIGH)**：AMD §5.4 + spec §5.4 Race D 改為 dynamic：
- TooManyPending 觸發 → 立即 market re-dispatch + **本 symbol** 60s close-maker pause（不 global）
- 連續 3 次 TooManyPending in 60s → 才升為 global 60s pause（不 5 min）
- 跑 `client.rate_limit_remaining()` (字典 §4.1) gate：utilization > 80% → throttle new close maker submit

**應修 1 (MEDIUM)**：加 `[64] close_maker_rate_limit_pause_total_duration_per_day` healthcheck，per env 7d 累計 pause time > 5 min/day → WARN，> 30 min/day → FAIL（防 maker path silent-dead）。

## §3 PostOnlyWillTakeLiquidity close-side 語義

**Bybit reject reason 字典**（字典 §4.2.1）：`EC_PostOnlyWillTakeLiquidity` 表示 PostOnly 限價單在訂單到達 matching engine 時已越過 BBO。**這個語義對 entry/close 是相同的 mechanical condition**，與訂單 side 無關。

**但策略語義不同**：
- **Entry side**：reject = 入場價偏離過遠 → 信號可能失效 → `compute_post_only_price` strict-skip + reject cooldown 1min（已驗）
- **Close side**：reject = 限價未追上市場（如止盈價已過市）→ **持倉方向已是錯的**（市場跑贏你的限價）→ AMD §5.3 / spec §5.3 Race C 「直接 market 不重 quote」**符合 Bybit best practice**

**閉環檢查**：Race C 「不進 cooldown」決策正確——entry side cooldown 是「防策略 spam 同 reject」，close side 是「reduceOnly 必執行」，cooldown 反而會 violate 原則 #5 生存 > 利潤。

**spec §6.1 reject_cooldown 拆分 entry/close** 設計正確且修現存 bug，但 **bug 嚴重度 BB 提升**：當前 entry reject 凍住 close path 是「entry side 觸 rate-limit-adjacent 條件後 close 永遠走 market」的 silent degradation——應寫入 PA verdict §1 verified bug 表，**這個 fix 是 prerequisite，不是 by-the-way scope**。

**必修 3 (HIGH)**：reject_cooldown 拆分必在 Phase 2a Demo enable 前 land。E1-D 的 ticket 升 P0 priority。

## §4 Fee tier / broker rebate 影響

**Bybit V5 USDT perpetual fee tier**（snapshot, verify via `/v5/account/fee-rate`）：
- Tier 0 / VIP 0：taker 5.5 bps / maker 2.0 bps（spec §1.2 引用 5.5 / 1 bps 是舊估值，BB 字典 §1.5 line 656 default `taker 0.055% maker 0.02%` = 5.5 / 2.0 bps 為準）
- VIP 1+：階梯 taker 4.5/3.5/3.0/2.5/2.0 bps；maker 1.5/1.0/0.5/0.0 bps
- Maker Maker rebate VIP：負費率 −0.5 bps 起

**Spec §1.2 estimate 修正**：
- 真實 saving per close = (taker 5.5 − maker 2.0) = **3.5 bps**（不是 4.5 bps）
- 跨 5 策略 + 25 symbols + 30d 估算需要拿真實 `account_manager::refresh_fee_rates` 後 `taker_fee()/maker_fee()` 算 — 不要用字典 default 估算 SoT

**Maker / taker volume ratio 影響 fee tier**：Bybit VIP tier 升級看 30d total volume + maker/taker ratio。當前 OpenClaw 月成交量太小（demo 7d 203 closes × ~$300 = $60k/week → $260k/30d）達不到 VIP 1 ($1M/30d)，本次 close-maker 提升 maker ratio 對 fee tier 升級貢獻 < 1% 邊際。**Broker rebate**: 需 30d cumulative ≥ $10M（字典 §4.2），完全不適用。

**BB 結論**：Fee tier 影響本次 refactor edge 約 $200-$500/year（spec §1.2 estimate 對齊），不能救 -110 USDT structural deficit（AMD §1 已承認，#13 cost-aware 合規）。

**應修 2 (MEDIUM)**：spec §1.2 把「4.5 bps」改成「3.5 bps」並引用真實 `account_manager.taker_fee(symbol) - account_manager.maker_fee(symbol)`，避免 fee estimate drift。

## §5 WS private order channel 行為

**Bybit V5 WS `order` topic**（字典 §2.2 + WebFetch confirm）：PostOnly 限價單 reject 時：
- `orderStatus = "Cancelled"`（不是 `"Rejected"` — Bybit V5 設計 PostOnly cross 是 cancel 不是 reject，因 retCode=0 訂單已建立過）
- `rejectReason = "EC_PostOnlyWillTakeLiquidity"`
- 後續 fill 事件不再推送

**對 commands.rs 下游 `tracker.update` 影響**：
- close path 收到 `orderStatus=Cancelled + rejectReason=EC_PostOnly...` → 必走 maker_rejection.rs `MakerRejectionCategory::PostOnlyCross` 分支 → spec §5.3 Race C → 立即 market re-dispatch
- 若 tracker 只看 `orderStatus=Cancelled` 不看 `rejectReason` → 會誤認為「我方 cancel」走 noop 路徑 → 持倉裸露

**Rust source 已驗 OK**：`maker_rejection.rs:121` `"EC_ReachMaxPendingOrders" => TooManyPending`、line 117（grep）有 `EC_PostOnlyWillTakeLiquidity → PostOnlyCross`；entry side 已 production-tested。Close side 復用同 classifier 是 100% 正確設計（PA verdict §5.2 確認）。

**必修 4 (HIGH)**：spec §6.2 必明文「close side `MakerRejectionCategory` 復用 entry side classifier，不新建 `Self::CloseTooManyPending` / `Self::ClosePostOnlyCross` variant」——spec §6.2 當前文字「close side 加入 `Self::CloseTooManyPending`」會造成 enum duplicate 與 entry side 偏離維護。改：用既有 5 variant + close path routing 邏輯分流。

## §7 Symbol-level 限制

**Bybit symbol-level instrument filter**（字典 §1.9 instrument cache）：
- `tick_size` per symbol：close limit price 必 round（compute_close_limit_price strict-skip 設計符合）
- `price_filter` / `lot_size_filter`：minimum / maximum order size
- `postOnlyMaxLimitOrderSize`：spot 2025-01-14 新增 field（**linear perp 不適用**）

**已知 25 symbol universe corner case**：
- 部分 alt symbol（如 1000PEPEUSDT, 1000BONKUSDT）tick_size = 0.000001，maker price offset 0.5 bps 可能 < 1 tick → buffer_ticks=1 設計（spec §6 表）會「向我方偏 1 tick」實際只 0.001%，可能直接過市 reject
- Symbol 上下架（字典 §4.2 retCode 110074 `ContractNotLive`）：close path 若 symbol 突 delisted 走 maker → reject 可能不是 `EC_PostOnlyWillTakeLiquidity` 而是 110074 path

**應修 3 (MEDIUM)**：spec §6 compute_close_limit_price 加 instrument filter check：
- 若 `offset_bps / 10000 * price < tick_size` → 自動 escalate buffer_ticks 至 2-3
- 若 symbol `status != "Trading"` → strict-skip → market
- E1 必驗 1000PEPEUSDT / 1000BONKUSDT / 小 tick alt corner case

**建議補錄 2**：字典手冊 §1.9 instrument cache 加「per-symbol PostOnly minimum effective offset = max(offset_bps, 2 * tick_size / price * 10000)」computation guidance。
