### E3-CRITICAL-1：openclaw_proxy 無認證
**代碼位置：** `app/main.py:186-209`

**實際代碼：**
```python
@app.api_route("/openclaw/{path:path}", methods=["GET", "POST", "PUT", "DELETE"], include_in_schema=False)
async def openclaw_proxy(path: str, request: Request):
    """Reverse proxy to OpenClaw Gateway"""
    # 無任何 Depends(current_actor)
    import os as _os
    _oc_host = _os.getenv("OPENCLAW_GATEWAY_HOST", "127.0.0.1")
    target = f"http://{_oc_host}:18789/{path}"
    ...
```

**驗證結論：** 確認屬實。函數簽名中無任何認證依賴，任何未認證請求者均可訪問。

**嚴重性修正：** 維持 CRITICAL。

**補充技術分析：**
- `_oc_host` 默認為 `127.0.0.1`（loopback），意味著局域網或 Tailscale 部署環境中，此漏洞僅在本機可訪問時有效。但通過 Tailscale Serve（HTTPS 代理），外部訪問者仍可利用此端點。
- 請求頭透傳（`headers={k: v for k, v in request.headers.items()...}`）意味著攻擊者可以偽造 X-Forwarded-For 等頭部。
- E5 報告同時指出此函數使用了同步阻塞 `urllib.request.urlopen`（存在 10 秒超時阻塞事件循環的性能問題），修復認證的同時應一并修復為異步調用。

**修復方案評估：** 添加 `actor=Depends(current_actor)` 參數即可，技術可行，無副作用。但需注意：OpenClaw 目前是 GUI 前端調用此代理，前端必須正確傳遞 Bearer Token，否則 GUI 的 OpenClaw 相關功能會全部 401。修復前需確認 GUI 中所有 `/openclaw/` 調用均已加認證頭。

### E3-CRITICAL-2：`_require_operator_role()` 類型不匹配——特權端點永遠 401

**代碼位置：** `app/governance_routes.py:95-131`

**實際代碼（完整流程）：**
```python
# _get_auth_actor() 函數（第 95-113 行）：
def _get_auth_actor() -> Any:
    from . import main_legacy as base
    actor = base.current_actor   # ← 返回 current_actor 函數對象本身！
    if actor is None:
        raise HTTPException(...)
    return actor                  # 返回的是函數，不是 dataclass 實例

# 路由中的使用（第 502 行）：
actor: Any = Depends(_get_auth_actor())
# Depends 會調用 _get_auth_actor()，得到 current_actor 函數
# FastAPI Depends 發現返回值是可調用的，會再次調用它
# 結果：actor 最終是 AuthenticatedActor(slots=True) dataclass 實例

# _require_operator_role（第 116-130 行）：
def _require_operator_role(actor: Any) -> None:
    if not actor or not isinstance(actor, dict):  # AuthenticatedActor 不是 dict！
        raise HTTPException(status_code=401, ...)
    is_operator = (
        actor.get("operator_role") == "Operator" or  # dataclass 無 .get() 方法
        ...
    )
```

**驗證結論：** 確認屬實，但根因分析需要修正。

**E3 報告根因分析的準確性：** E3 報告描述「`_get_auth_actor()` 返回函數對象本身，FastAPI Depends 再次調用它」的描述在技術上**部分正確**：

1. `_get_auth_actor()` 確實返回 `base.current_actor`（函數對象），而非 dataclass 實例
2. `Depends(_get_auth_actor())` — 注意括號，這裡 `_get_auth_actor()` 已被立即調用，返回 `current_actor` 函數
3. FastAPI 的 `Depends(current_actor)` 會調用 `current_actor(authorization=Header(...))` 並注入
4. 最終 `actor` 是 `build_authenticated_actor()` 返回的 `AuthenticatedActor` dataclass 實例

**實際問題：** `_require_operator_role` 中 `isinstance(actor, dict)` 永遠為 False——`AuthenticatedActor` 是 `@dataclass(slots=True)` 而非 dict。因此：
- 所有 `_require_operator_role(actor)` 調用永遠拋出 `HTTPException(401)`
- 受影響端點完全不可用（功能性 DoS）
- 即便認證成功，operator 操作仍被拒絕

**嚴重性修正：** 維持 CRITICAL。雖不直接導致資金損失，但使所有治理寫操作端點（批准授權、風控覆蓋等）永遠不可用，系統無法正常運作。

**修復方案評估：**
```python
# 正確修復：
from . import main_legacy as base

def _require_operator_role(actor: Any) -> None:
    if not actor or not isinstance(actor, base.AuthenticatedActor):
        raise HTTPException(status_code=401, detail="Authentication required")
    # AuthenticatedActor 有 .roles: set[str]，不是 dict
    if "operator" not in actor.roles and "operator_guarded" not in actor.roles:
        raise HTTPException(status_code=403, detail="Operator role required")
```

需要確認 `settings.auth_roles` 中包含的實際角色名，與 `actor.roles` 匹配。

## 二、其他高優先級問題技術評估

### E3-HIGH-5：CORS allow_credentials=True 配置

**實際代碼（已確認）：**
```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=_cors_origins.split(",") if _cors_origins else [],
    allow_credentials=True,
    allow_methods=["GET", "POST"],
    allow_headers=["Authorization", "Content-Type"],
)
```

**PA 評估：** E3 報告的風險描述**過於誇大**。系統使用 Bearer Token 認證（非 Cookie），CORS + credentials 主要影響 Cookie 認證場景。`Bearer Token` 放在 `Authorization` 頭部，不受 CORS credentials 設置影響（瀏覽器的 CORS 限制是針對 `Cookie`/`Authorization header` 的自動附加，而 Bearer Token 是由 JavaScript 主動設置的）。

**實際風險：** 若 `_cors_origins` 為空（`allow_origins=[]`），所有跨域請求被瀏覽器攔截——比有 CORS 更安全。若設置了特定來源，credentials 允許 Cookie 發送，但 Cookie 在此系統中不使用，風險極低。

**嚴重性建議：** 從 HIGH 降為 **MEDIUM**（最佳實踐違反，非立即可利用漏洞）。

### E3-MEDIUM-1：GovernanceHub 授權緩存 100ms 競態窗口

**PA 評估：** 報告的場景描述技術上存在錯誤。緩存讀取是鎖外，但 `FROZEN` 狀態轉換是在鎖內清除緩存的（報告提及「revoke/freeze 路徑需確認」但未實際讀代碼）。100ms 窗口在 Paper Trading 環境下（無真實資金）影響極低，在 Live 環境下需要重視。建議維持 **MEDIUM** 但降低緊迫性。

### E3-HIGH-4：web-pilot subprocess 接收 AI 生成查詢

**PA 評估：** 此問題評級合理。但報告提到「若 web-pilot 自身存在參數注入漏洞」——這是一個假設性條件。修復建議（添加 `--` 分隔符）是最佳實踐，但不解決更根本的問題：AI 生成內容被直接傳入外部工具，AI prompt injection 攻擊面存在。更完整的修復應包含查詢字符串白名單/長度限制（已建議）。

## 四、架構層面問題補充（報告未涵蓋）

### 4.1 Dependency Injection 模式不統一，存在系統性風險

**問題描述：**
系統中存在三種不同的依賴注入模式：
1. `main_legacy.py` 路由：`Depends(current_actor)` — 直接使用 FastAPI DI，返回 `AuthenticatedActor`
2. `governance_routes.py` 路由：`Depends(_get_auth_actor())` — 先調用 `_get_auth_actor()` 得到函數，再讓 FastAPI 調用，最終也得到 `AuthenticatedActor`
3. 某些函數內部直接調用 `base.current_actor` — 靜態調用

這種不統一導致 `_require_operator_role` 誤用了 `dict` 類型檢查（可能作者以為 actor 是 dict，因為某些舊版本可能返回過 dict）。

**架構影響：** 任何新增治理端點的開發者可能繼續犯同樣錯誤，因為 `_require_operator_role` 看起來是封裝好的工具函數，但實際上是損壞的。

**建議方向：** 統一使用 `Depends(current_actor)` 模式，在 `governance_routes.py` 中刪除 `_get_auth_actor()` 這個不必要的中間層。`_require_operator_role` 的類型檢查改為 `isinstance(actor, AuthenticatedActor)`。

### 4.2 三層防線均為可選注入，缺乏啟動時完整性驗證

**問題描述：**
`paper_trading_engine.py` 的 `submit_order()` 依賴：
- `self._governance_hub`（可選）
- `self.risk_manager`（可選）
- `self._learning_tier_gate`（隱含在 line 1032）

`pipeline_bridge.py` 的 `_process_pending_intents()` 依賴：
- `self._governance_hub`（可選）
- `self._guardian_agent`（可選）

5 個核心安全組件全部使用 `if self._xxx:` 模式——在 Python 初始化為 `None` 時默默跳過。系統在 `phase2_strategy_routes.py` 模塊初始化時注入這些依賴，但如果導入順序或啟動過程出現問題，可能出現靜默的「不安全模式」運行。

**架構影響：** 缺乏啟動時的完整性驗證（startup check）。系統應在 FastAPI 的 `startup` 事件中驗證所有關鍵依賴已被注入，若未注入應拒絕服務啟動而非靜默降級。

**建議方向：** 在 `main.py` 的 `@app.on_event("startup")` 中添加斷言：`assert PIPELINE_BRIDGE._guardian_agent is not None, "GuardianAgent not injected"`。同時，所有「可選」的關鍵安全組件應提供 `require_all_dependencies_injected()` 方法。

### 4.3 `_get_auth_actor()` + `Depends` 的語義陷阱構成技術債

**問題描述：**
```python
# 當前代碼（governance_routes.py）：
actor: Any = Depends(_get_auth_actor())
```
`_get_auth_actor()` 被**立即調用**（有括號），返回的是 `current_actor` 函數，然後被 `Depends` 包裝。這等同於 `Depends(current_actor)`，但通過一個返回函數的函數間接實現。

此模式的問題：
1. 開發者閱讀代碼時難以追蹤 actor 的實際類型
2. `_get_auth_actor()` 的注釋說「返回 Actor dict with role and identity」——這是錯誤的（實際返回 AuthenticatedActor dataclass）
3. 造成了 `_require_operator_role` 使用 `isinstance(actor, dict)` 的根本原因

**架構影響：** 所有基於此認知錯誤寫的代碼都潛在有同樣的類型錯誤。

**建議方向：** 刪除 `_get_auth_actor()` 中間函數，直接使用 `Depends(current_actor)` 模式，與 `main_legacy.py` 保持一致。

### 4.4 PipelineBridge 依賴注入的時間窗口風險（CC G8 的延伸）

**問題描述：**
`phase2_strategy_routes.py` 在模塊級別注入依賴（模塊導入時執行），如：
```python
PIPELINE_BRIDGE.set_guardian_agent(GUARDIAN_AGENT)
```
但 GuardianAgent、GovernanceHub 等對象本身也在模塊級別初始化。如果 Python 的模塊導入順序導致 PIPELINE_BRIDGE 先被創建、先開始接收 tick，而 GuardianAgent 尚未注入，這個窗口期內的所有 intent 都會以 pass-through 方式提交。

**時間窗口大小：** 由 uvicorn 啟動序列的模塊導入時間決定，通常 < 1 秒，但在系統壓力下可能更長。

**架構影響：** 在服務重啟恢復過程中，市場流（已設置為常駐，不依賴 Paper 會話）可能在治理組件完全就緒前就開始分發 tick，觸發 intent 提交。

**建議方向：** PipelineBridge 應在所有必要依賴注入完成後才啟動（當前 `activate()` 方法是分離的，應添加前置條件檢查）。

### 4.5 `governance_routes.py` 中 `hub._enabled` 直接訪問私有屬性

**問題描述（E3 和 CC 均未提及）：**
```python
# governance_routes.py:514
if not hub._enabled:
    raise HTTPException(status_code=403, detail="Governance hub disabled")
```
直接訪問 `GovernanceHub` 的私有屬性 `_enabled`，繞過了任何封裝。若 `GovernanceHub` 未來重構此屬性（改名、改為 property、改為方法），`governance_routes.py` 會靜默出錯。

**建議方向：** 在 GovernanceHub 添加 `is_globally_enabled() -> bool` 公開方法。

### 4.6 E4 報告未識別的最高優先測試缺口：治理端點功能性 DoS

**問題描述（E4 報告未涵蓋）：**
E4 報告將 `governance_routes.py` 標記為 CRITICAL 測試缺口（10% 覆蓋），但未具體說明以下已知 Bug 的測試狀態：

由於 `_require_operator_role` 永遠返回 401，以下端點的測試用例在**真實行為下**應該全部返回 401：
- `POST /governance/auth/approve`

## 六、測試策略補充

對 E4 報告的測試設計，提供架構師視角的補充：

### 6.1 pipeline_bridge.py 測試必須使用集成測試而非單元測試

E4 建議的測試用例（`test_intent_submitted_via_governance_gate` 等）需要 mock 的組件包括：GovernanceHub、GuardianAgent、PaperTradingEngine。關鍵的是這些 mock **必須模擬 fail-closed 行為**（mock.side_effect = Exception），而不僅測試 happy path，否則測試無法驗證 Guardian=None 的 pass-through 行為（即修復前的 Bug）。

**建議的測試設計：**
```python
# 應該存在（但目前沒有）：
def test_guardian_none_blocks_intent_after_fix():
    """Guardian 為 None 時，intent 應被 fail-closed 拒絕"""
    bridge = PipelineBridge(engine=mock_engine)
    bridge._guardian_agent = None  # 不注入 Guardian
    bridge._intents.append(mock_intent)
    bridge._process_pending_intents()
    # 修復後：mock_engine.submit_order 不應被調用
    mock_engine.submit_order.assert_not_called()
```

### 6.2 governance_routes.py 測試必須區分認證層和授權層

E4 建議的測試必須覆蓋兩個維度：
1. **認證失敗（401）**：無 Token 或 Token 無效
2. **授權失敗（403）**：Token 有效但角色不足
3. **成功路徑（200/201）**：Token 有效且 operator 角色

當前 `_require_operator_role` 損壞的情況下，情況 3 永遠返回 401 而非 200，測試用例如果只驗證「無 Token 時 401」，不能發現此 Bug。

**關鍵測試（E4 未包含）：**
```python
def test_approve_authorization_with_valid_operator_token_returns_200():
    """具有 operator 角色的有效 Token 應能成功調用 approve 端點"""
    # 這個測試目前應該失敗（因為 _require_operator_role 損壞）
    response = client.post("/api/v1/governance/auth/approve",
                          headers={"Authorization": f"Bearer {OPERATOR_TOKEN}"},
                          json={"approval_note": "approved"})
    assert response.status_code in (200, 201)  # 不應是 401
```

### 6.3 並發測試應覆蓋 PipelineBridge 的 _lock

E4 指出並發測試局限於 GovernanceHub，`PipelineBridge._lock` 未覆蓋。具體設計建議：

```python
def test_pipeline_bridge_concurrent_intents_no_deadlock():
    """10 個並發線程同時向 bridge 提交 intent，不應死鎖，stats 計數準確"""

    bridge = make_test_bridge()
    threads = [threading.Thread(target=lambda: bridge.on_tick(MOCK_TICK)) for _ in range(10)]
    for t in threads: t.start()
    for t in threads: t.join(timeout=5.0)
    # 驗證無死鎖，stats["ticks_received"] == 10
```

## 八、PA 最終建議

### 8.1 系統整體安全評估

在現有測試框架（2,227 tests passed）下，系統的**治理核心**（GovernanceHub、SM-01/SM-02/SM-04、RiskManager）設計扎實，fail-closed 語義在關鍵路徑得到落實，這是一個工程質量良好的基礎。

但存在兩個**結構性問題**需要立即關注：

1. **治理控制面板 API 功能性癱瘓**：由於 `_require_operator_role` 永遠返回 401，所有需要 operator 角色的治理操作端點（批准授權、風控覆蓋等）完全無法使用，這意味著當前系統的 Operator 無法通過 API 執行任何治理審批操作。這是 E3-CRITICAL-2 的直接後果，在任何 Live 準備工作之前必須修復。

2. **文檔聲稱的 fail-closed 在多處未落實為代碼**：`pipeline_bridge.py` 的 Guardian=None pass-through 是典型案例。注釋說 fail-closed，代碼說 pass-through。在審計中，代碼是真相，注釋是意圖——意圖與真相不符是系統性問題。

### 8.2 A3 GUI 報告的架構評估

A3 報告的 42 項 GUI 問題整體真實可信，沒有發現重大誤報。PA 補充以下架構層面觀點：

1. **A01（模式切換無路徑感）和 O01（Demo 激活流程分散）** 實際上是同一個根本問題：系統的操作流程本質上是一個狀態機（design → observe → shadow → demo → live），但 GUI 將其設計為獨立的、無關聯的按鈕和 Tab，沒有引導用戶按照正確順序操作。建議在架構層面引入「引導模式」（Wizard Mode）來統一此流程。

2. **內部術語暴露（SM-01、Decision Lease、fail-closed）** 在短期內可以通過 tooltip 和文案修改緩解，但長期解決方案是建立一個「術語映射層」（類似 i18n 的概念），在代碼層使用技術術語，在 UI 層統一渲染為業務術語。

3. **侧边栏信息與主內容重複（I03）** 在技術層面的根因是：GUI 沒有「全局狀態」和「局部狀態」的明確分離。全局狀態（系統模式、執行權限、Paper 會話狀態）應由一個 Store 管理並由 common.js 統一渲染，而非各 Tab 各自維護。

### 8.3 Phase 1 架構師最高優先建議

基於上述分析，PA 建議 Phase 1 在規劃的「安全閘補全」中，額外包含以下兩項架構清理工作：

1. **依賴注入模式統一**（半天）：刪除 `_get_auth_actor()`，統一使用 `Depends(current_actor)`，修復 `_require_operator_role` 的類型檢查，使用 `isinstance(actor, AuthenticatedActor)` 和 `actor.roles`。

2. **啟動時完整性驗證**（半天）：在 FastAPI startup hook 中驗證所有關鍵安全組件已被注入，防止靜默降級運行。

這兩項工作合計 1 天，可在 Phase 1 Batch 1A（H0 Gate，5 天）的同時進行，且不影響其他開發工作。
