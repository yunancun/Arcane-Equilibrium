## 二、CC 審計發現：執行前必須修復的兩個問題

### 問題 1：執行層繞過了 Decision Lease（違反根原則 3）

**什麼問題**：ExecutorAgent 在執行訂單前，沒有先向 GovernanceHub 申請 Decision Lease。這意味著 AI 評估後的意圖可以不經過時效控制和可撤銷機制直接執行，違反了「AI 輸出不等於即時命令」的核心設計原則。

**影響**：目前系統在 `demo_only` 模式下因為有其他保護還算安全，但如果直接進 Live 前不修復，這是一個架構層面的安全漏洞。

**修復方案**：在 `executor_agent.py` 的執行路徑插入 `acquire_lease()` 調用，失敗時 fail-closed 拒絕執行。工時約 2h。

### 問題 2：AI 每日花費硬上限設錯了（違反 DOC-08 §12 安全不變量）

**什麼問題**：代碼中 AI 每日花費上限是 `$15.00`，DOC-08 設計文件規定的是 `$2.00`，差了 7.5 倍。

**影響**：demo_only 模式下沒有真實花費，但這個設定值會在接通 AI 後直接影響預算控制。

**修復方案**：5 個位置改值（代碼 + 前端顯示）。工時約 1h。

## 三、Wave 5 工作內容（三個 Sprint）

### Sprint 0 — 先修 BLOCKER（約 6h，並行，立即可啟動）

修上述兩個問題，E2 審查 + E4 全量回歸後進入 Sprint 5a。

### Sprint 5a — H1-H5 AI 治理管線接通（約 15h）

讓 AI 評估真正插入交易決策流程：

- **Scout 情報鏈路確認**：確認市場掃描情報確實傳遞給策略師 Agent
- **H0 Gate 正式啟用 blocking 模式**：目前是 warn-only，改為實際攔截不合格的交易意圖
- **H1 ThoughtGate**：每次 AI 調用前先做確定性判斷（預算夠不夠？信號複雜度值得 AI 介入嗎？剛觸發止損在冷卻期嗎？）
- **策略師 Agent 從「影子模式」轉為「真實模式」**：目前策略師評估後只記錄日誌，不產生真實交易意圖。Sprint 5a 完成後開始產生真實意圖送 Guardian 審核。
- **H2/H3 接入**：AI 預算門控 + 模型路由（簡單任務用 9B，複雜任務用 27B，深度分析用 Sonnet）

### Sprint 5b — Agent 落地完善（約 13h）

- H4 AI 輸出驗證（防止幻覺輸出直接成為交易意圖）
- H5 成本記錄（每次 Ollama 調用追蹤，AI ROI 計算加免責標記）
- Scout 後台定時掃描（每 30 分鐘自動觸發，不依賴 API 手動調用）
- L0 降級集成測試（確認 Ollama 崩潰後系統純本地模式仍可運行）
