## § 1. 背景與當前架構（必先讀）

### § 1.1 SL/TP 三層架構（per memory `project_agent_p2_dynamic_sl_tp.md`）

當前 OpenClaw SL/TP 三層：

| 層 | 字段 | 角色 | 修改路徑 |
|---|---|---|---|
| **P1（操作員硬上限）** | `RiskConfig.limits.stop_loss_max_pct` / `take_profit_max_pct` | **硬頂**，所有策略**不可**突破 | TOML edit + IPC reload, operator only |
| **P2（Agent 軟值，全局）** | `RiskConfig.agent.stop_loss_pct` / `take_profit_pct` | Agent 可調全局軟值（None = 動態 ATR）| IPC `patch_risk_config` |
| **物理層（Track P）** | `ExitConfig.giveback_*` / `min_*` | 微利保護 + giveback gates | IPC `patch_risk_config` |

**重要**：當前**沒有** per-strategy SL/TP 字段。`MaCrossoverParams`（per `strategy_params.rs:23-77`）有 cooldown / adx / confluence / maker entry 等，**完全無** SL/TP 字段。

對比：bb_breakout **有** `trailing_stop_atr_mult`（per `bb_breakout/params.rs:101`），是策略內 ATR 倍數，但這只控 trailing stop 距離，不是完整 SL/TP。

### § 1.2 ma_crossover 當前 SL/TP 路徑

ma_crossover 訂單觸發 → `risk_checks::tick_risk_action`（per `risk_checks.rs:170+`）:
1. Hard stop: `pnl_pct <= -limits.stop_loss_max_pct`
2. Dynamic stop: ATR-based `compute_dynamic_stop_pct(...)`
3. Take profit: `limits.take_profit_max_pct * regime_multipliers(regime).tp`
4. Trailing stop: `agent.trailing_activation_pct + trailing_distance_pct`
5. Time stop: `limits.holding_hours_max * regime_mult.time`
6. PHYS-LOCK: ExitConfig 4-Gate

**所有路徑均共用 `RiskConfig.limits` + `RiskConfig.agent`**，無 per-strategy 切片。ma_crossover 與 grid_trading 共用同一 SL/TP 上限。

### § 1.3 G2-02 counterfactual 與 G2-03 關係

| 議程 | 範圍 | 輸出 |
|---|---|---|
| **G2-02** | ma_crossover R:R 對稱性 counterfactual replay（用 decision_outcomes + exit_features 重算「若 fee=2bps 會如何」）| 結論：fee 改變後 R:R 可能 ~對稱（0.9-1.1）但 win rate 仍 < 50%（QC §Q2 預測） |
| **G2-03** | 基於 G2-02 結論決策 SL/TP 定制（若 fee 修不能救 R:R）| 本 RFC 規範 |

G2-02 是診斷，G2-03 是處方。**G2-03 不可在 G2-02 完成前啟動**（無診斷依據）。

## § 2. Option B 定義（決策最關鍵）

FA report §gap-3 指出 Option B 散落多處，需要 PA 級 RFC 統一。以下三個候選層次：

### § 2.1 候選 B1：策略 params 加 SL/TP ATR 倍數字段

**改動**：在 `MaCrossoverParams` 加 4 個字段：
```rust
pub struct MaCrossoverParams {
    // ... existing fields ...

    /// Optional per-strategy SL ATR multiplier override.
    /// `None` (default) = use global RiskConfig.limits.stop_loss_max_pct path.
    /// `Some(x)` = override stop loss = ATR × x (still bounded by P1 max).
    pub sl_atr_mult: Option<f64>,

    /// Optional per-strategy TP ATR multiplier override.
    pub tp_atr_mult: Option<f64>,

    /// Optional per-strategy trailing activation pct override.
    pub trailing_activation_pct_override: Option<f64>,

    /// Optional per-strategy trailing distance pct override.
    pub trailing_distance_pct_override: Option<f64>,
}
```

接線：`risk_checks::tick_risk_action(...)` 接收新 param `strategy_overrides: Option<&StrategySLTPOverride>` → 若 Some 則用 strategy 值算 SL/TP 並 cap 於 P1 max。

**優點**：與既有 `MaCrossoverParams` 機制一致（已有 `update_params_json` / TOML hot-reload 路徑）；per-strategy stratification 自然落在 strategy struct
**缺點**：改 risk_checks 接口（增加參數）；ma_crossover / grid / bb 各加類似字段 → 重複 boilerplate

### § 2.2 候選 B2：RiskConfig.per_strategy 擴 SL/TP 字段

**改動**：擴展 `StrategyOverride`（per `risk_config.rs:558`，目前已有 `position_size_max_pct` / `max_concurrent_positions` 等）：
```rust
pub struct StrategyOverride {
    // ... existing fields ...

    pub stop_loss_max_pct_override: Option<f64>,    // 必 ≤ limits.stop_loss_max_pct
    pub take_profit_max_pct_override: Option<f64>,  // 必 ≤ limits.take_profit_max_pct
    pub trailing_activation_pct_override: Option<f64>,
    pub trailing_distance_pct_override: Option<f64>,
}
```

接線：`risk_checks::tick_risk_action` 從 `RiskConfig.per_strategy[strategy_name]` 查 override，若 Some 則使用，否則 fallback `RiskConfig.limits`。

**優點**：與既有 `per_strategy` 機制一致（已 IPC hot-reload 通）；不污染 strategy params；validate 容易（cross-field invariant 直接寫在 RiskConfig.validate）
**缺點**：原 `StrategyOverride` 設計意圖偏 risk gate（enabled / position cap），加 SL/TP 屬語意擴張

### § 2.3 候選 B3：Strategy trait 加 sl_tp_advice() hook

**改動**：Strategy trait 增加 advisory method：
```rust
trait Strategy {
    // ... existing ...

    /// Optional SL/TP advisory; returns (sl_pct, tp_pct) as override.
    /// Default impl returns None (use global path).
    fn sl_tp_advice(&self, ctx: &TickContext) -> Option<(Option<f64>, Option<f64>)> {
        None
    }
}
```

ma_crossover 實作 `sl_tp_advice`（基於 ATR + regime），risk_checks 從 strategy advisory 拿值。

**優點**：strategy 完全自主決定（per memory `feedback_agent_autonomy`）；可 per-tick 動態（不只是 config 值）
**缺點**：trait 擴張影響 5 strategy；on_tick path 多一次 dispatch；測試難度上升

### § 2.4 PA 推薦：候選 B2

**推薦理由**：
1. **架構債最低**：`per_strategy` 機制已 live + IPC hot-reload + TOML persist；只擴 4 個 Optional 字段，無新接線點
2. **與 §1.1 三層分離一致**：B2 是「每策略的 P1 軟頂」（在 P1 硬頂之下），三層架構保持
3. **validate 集中**：cross-field invariant `override ≤ limits` 寫在 `RiskConfig.validate()` 一次（B1 需在 5 strategy params validate 各寫一次）
4. **與 G3-02 ExecutorConfig 模式對齊**：G3-02 將 hardcoded shadow_mode 改為 RiskConfig.executor.shadow_mode；B2 將「per-strategy SL/TP」改為 RiskConfig.per_strategy.sl_tp_override 是同模式

**B1 否決理由**：MaCrossoverParams 純策略行為參數（cooldown / adx / confluence），加 SL/TP 違反 separation of concerns（risk gate ≠ strategy logic）

**B3 否決理由**：trait 擴張 5 strategy 工時 ≥3x；當前 G2-03 只針對 ma_crossover，B3 屬 over-engineering；未來若需 per-tick 動態 SL/TP 再評估 B3 RFC

## § 3. P1 max 硬頂 vs 策略軟值的 boundary

### § 3.1 enforce 點

採 **三道 enforce**：

| 點 | 檢查 | 失敗動作 |
|---|---|---|
| **A. RiskConfig.validate()** | `per_strategy[*].stop_loss_max_pct_override ≤ limits.stop_loss_max_pct`（同樣 take_profit / trailing）| validate Err → IPC patch reject + TOML load fail |
| **B. risk_checks::tick_risk_action()** | runtime fallback：override `> limits` 時用 limits（防 race condition + edge case）| Log WARN + 用 limits 值 |
| **C. counterfactual_calibrator dry-run** | calibrator 寫入前最後一道驗證 | 拒絕寫 + 報告 operator |

**A 是主要防線**，B 是兜底，C 是離線安全網。

### § 3.2 escalate path：counterfactual 出 SL ATR mult > P1 max

若 G2-02 counterfactual 結論為「ma_crossover SL/TP optimal = P1 max × 1.2」（例）：
1. **G2-03 不可自動套用**（違反 P1 硬頂）
2. **PA 升 PM**：需要重審 P1 max 是否合理
3. **PM + Operator 決策**：是否擴大 P1 max（屬 §四 硬邊界相關，operator approval 必須）
4. P1 max 確認後，G2-03 才能在新 P1 max 之下落 strategy override

**永遠不可**讓策略軟值繞過 P1 硬頂。memory `project_agent_p2_dynamic_sl_tp.md` 確認此原則。

## § 4. G2-02 counterfactual → G2-03 binding 邏輯

### § 4.1 binding 模式選擇

**選項 A**：自動 binding（G2-02 出 R:R 1.6 → G2-03 自動寫 sl_atr_mult / tp_atr_mult）
**選項 B**：manual confirm binding（QC + PM review counterfactual → operator approve → manual IPC patch）

**PA 推薦 B（manual confirm）**，理由：
1. **QC §Q2 預期 alpha 結構問題**：fee 改變後 R:R 可能~對稱但 win rate < 50%。**自動 binding 會掩蓋根本問題**（看似對稱但 net edge 仍負）
2. **counterfactual 是離線研究** vs binding 是 live 風控變更；中間需人類 sanity check
3. **與 EDGE-P1b RFC § 2.2 一致**：所有 RiskConfig bind 動作採 manual approve 模式
4. **第一次 binding 之後**穩定下來再評估自動化（屬 W4-W5 follow-up RFC）

### § 4.2 binding 流程 SOP

```
G2-02 counterfactual 完成
    ↓
QC + FA 共同 review counterfactual report
（指標：post-fee R:R / win rate / Sharpe / DSR / cohort sample）
    ↓
判斷：alpha 結構是否健康？
    ↓ YES                              ↓ NO
G2-03 binding（manual IPC patch）       G2-04 disable 議程
```

binding **必須**有 QC + FA 雙簽，operator approve 後才能 IPC 寫入。

### § 4.3 per-regime 切片（per FA gap-5 + DOC-03 對齊）

當前 `risk_checks::tick_risk_action` 已用 `regime_multipliers(regime).tp / time` 對 SL/TP 做 regime-aware 調整。G2-03 binding 時：

**推薦**：override 提供 base 值；regime multiplier 仍由 `regime_multipliers()` 套用
- `stop_loss_max_pct_override = 4.0` → regime trending 時 effective SL = 4.0 × regime_mult.sl
- 不開 per-regime override（會炸 schema：4 字段 × 4 regime = 16 字段）

**例外**：若 counterfactual 揭示 ma_crossover 在 trending vs ranging regime 的 R:R 差異 ≥ 50%，再評估 per-regime override（new RFC G2-03-FUP）

## § 8. E2 重點審查 3 點

1. **cross-field invariant 不可被 patch_risk_config 繞過**：T1 validate 必須在 ConfigStore::apply_patch 路徑被呼叫；E2 必驗 IPC patch 嘗試寫入 `override > limits` 時必 reject + log
2. **risk_checks runtime cap fallback 不可 silently 降級**：T2 若 runtime 發現 override > limits（race condition）必須 log WARN + 用 limits，**不可** silently 用 override（任何超 P1 硬頂的執行 = 違反 §四 硬邊界）
3. **TOML 範本不可預設啟用 override**：T3 三 engine TOML 必須註解 `# stop_loss_max_pct_override = 4.0  # commented out by default`，**不可** uncomment 預設值（避免 operator 不察覺 overrides 已 active）

## § 10. 不確定 / 未決問題

1. **G2-02 counterfactual 若揭示 ma_crossover 整體 alpha = 負**：G2-03 binding 變得無意義（任何 SL/TP 設定救不了負 alpha 策略），應走 G2-04 disable 路；本 RFC §5.1 順序圖已涵蓋此分支
2. **per-regime override 的觸發門檻**：QC §Q2 預期 ma_crossover post-G7-09 後 R:R ~對稱但 win rate < 50%；若 per-regime 切片揭示 trending regime 顯著好 → G2-03-FUP per-regime override（new RFC）
