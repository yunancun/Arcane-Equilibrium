## §1 數學模型風險

### 1.1 「Close path = alpha-bearing pathway」claim 的數學正當性

**AMD §1 第 1 條**白紙黑字：「Close path is now an alpha-bearing pathway.」

從 **transaction cost economics**（Almgren-Chriss 1999 / Kissell-Glantz 2003）嚴格分解：
```
NetPnL = α_entry + α_holding + α_exit − (fee + slippage + impact + funding)
       = α (informational) − C (transaction cost)
```

- **Fee saving from PostOnly maker** = ΔC 減少（cost reduction），**屬 C 項**
- **Alpha-bearing** = 從 information / behavioral inefficiency / structural mispricing 提取 +α（屬 α 項）

兩者在數學上是不同 component。把「maker fee saving 4.5 bps」稱為 alpha-bearing **與你自己（CLAUDE.md §三 P0-EDGE-1）認定 5 textbook 策略 structural alpha-deficient 的事實邏輯不一致** — 真 alpha 是 +α 改善，fee saving 只是 C 減少，**它不改變 α 本身**。

**諷刺點**：spec §1.2 自己承認「真正價值：消除執行成本掩蓋 alpha 信號，讓 P0-EDGE-1 edge measurement 更乾淨」— 這恰恰證明 close path **不是 alpha-bearing pathway，而是 measurement-cleaning refactor**。AMD §1 措辭製造 alpha-bearing 印象但 spec §1.2 已自證不是。

**對 governance 影響**：AMD §1 觸發更嚴的 5-stage canary 框架（合理 fail-safe），但 framing 錯誤會讓未來 P0-EDGE-1 closure 評估時混淆「fee 改善是否算 alpha 改善」。**必修**：AMD §1 改為「close-path execution-quality refactor with measurable cost reduction effect; treated as alpha-bearing for canary governance discipline only」。

### 1.2 phys_lock_gate4_giveback maker pending 的條件期望陷阱

gate4_giveback **fire 條件本身就帶方向偏好資訊**：peak ATR 後利潤回吐到 threshold = 趨勢偏弱 + ROC 由正轉零或負。

```
條件期望分析（reference model，假設 mid-price martingale + drift d_t 在 gate4 fire 後 ≤ 0）：

market 鎖（t=0）：E[X] = current_mid + spread/2 (taker)
maker pending 30s：E[Y|fire condition] = E[mid_{t=30s}|gate4_giveback fired] + maker_offset

由於 gate4 fire ⇒ E[mid_{t+30s} − mid_t | fire] ≤ 0 (對 long 倉位 unfavourable)
        ⇒ E[Y] − E[X] = E[Δmid|fire] + (maker_offset − taker_spread/2)
                      ≤ 0 + ~ε bps   (fee saving ε ≈ 4.5 bps)

但 30s 期間 1σ 移動 (12-24 bps p95 per PA verdict slippage estimate)
        ⇒ tail-loss case Y << X 機率非零
```

spec §4.3 給 phys_lock_gate4_giveback `buffer_ticks=2 / offset_bps=1.0` 反而 **擴大** 等待 slippage（限價偏好更深 → 成交機率更低 → timeout fallback 更可能 → 吃更大 slippage）。

**必修**：phys_lock_gate4_giveback timeout 從 30000ms 降到 15000ms（不是 spec 給的 30000ms）；buffer_ticks 改回 1（不要 2，反向收益）；明文紀錄此參數選擇基於「gate4 fire condition 帶 unfavourable drift bias」。

## §2 量化評估 saving / cost

### 2.1 Fee saving 估計可信度

spec §1.2 給 `~$200-$500 fee saving / year`。**未說明 confidence interval、未含 slippage cost on maker timeout fallback**。

**QC 重估**：
- 假設 7d demo 155 active taker close × 52w = ~8060 closes/year
- per close 平均倉位估 $300（per spec §1.2）
- fee saving ≈ 4.5 bps × $300 × 8060 = **~$1088/year gross fee saving**

**但必須扣除**：
- Timeout fallback rate（spec AC-2 上限 30%）× slippage cost on fallback
- 假設 timeout 後 30s 內 1σ ≈ 12 bps（PA verdict 引 7d slippage std）
- fallback 期 expected adverse slippage 取 +6 bps（half-σ conservative）
- Net = $1088 − (8060 × 0.30 × 6 bps × $300 / 10000) = **$1088 − $435 = $653/year net**

spec 給 $200-$500 在 net 區間內，但**未說明這是 net 還是 gross**；應明文。

### 2.2 Breakeven fill rate 推導

```
Net benefit = fill_rate × fee_saving − (1−fill_rate) × adverse_slippage_on_fallback
            = fill_rate × 4.5 − (1−fill_rate) × 6
breakeven: fill_rate × 4.5 = (1−fill_rate) × 6
           fill_rate = 6 / (4.5+6) ≈ 57%
```

**結論**：breakeven fill rate ≈ 57%。spec AC-1「demo close maker 比例 ≥ 60%」剛好過 breakeven，**margin 極窄（3% buffer）**。一旦 maker fill rate 從 70% 跌到 55% 即 net negative。

**應修**：AC-1 應加「7d rolling fill rate 持續 ≥ 60% 不只觸發 1 次 PASS」+ 健康 check 加 WARN threshold = 65%（不只 60% pass / 40% fail）。

## §3 DSR / PBO / Evidence Discipline

### 3.1 AMD §5 「不啟動 Stage 0R replay」對嗎？

**部分對**：replay 用 historical book + fee model 確實無法精準模擬 maker queue position / partial fill / cancel timing。

**但仍有 evidence discipline 可走**（spec 漏）：

| Discipline | 是否該執行 | 理由 |
|---|---|---|
| Historical counterfactual cost simulation | **應做** | 取 demo 7d 真實 close fill price + 估 maker fill probability (from order book depth) → 推算 expected fee + slippage delta；給 95% bootstrap CI |
| Walk-forward 拆 demo 21d → 3 × 7d cohort | **應做** | 避免單期 selection bias；Phase 2a 應分週驗證 |
| Block-bootstrap CI on AC-5 net_bps Δ | **應做** | 1000 次 block-bootstrap，給 ΔBPS 95% CI 不只 point estimate |

**應修**：AMD §5 補「counterfactual cost simulation pre-IMPL」evidence packet 條件（不要求 Stage 0R full replay，但要 cost CI estimate）。

### 3.2 PBO / CSCV 適用性

無參數 sweep（exit_reason 是 categorical），PBO 不適用。**這條 spec 正確跳過 PBO**。

但 **per-exit_reason buffer_ticks / offset_bps / timeout_ms 表（spec §4.3 / AMD §6）= 8 × 3 = 24 個 tunable param**，這是 implicit parameter space，**未來 Phase 2a/2b 若 retune 必跑 DSR + PBO**。

**建議修**：AMD §6 補「Phase 2a / 2b 期間 per-exit_reason 參數不允許 in-flight retune，rollout-end 後若 retune 需走 DSR sweep」。

## §4 AC 設計嚴謹性

### 4.1 AC-5 「close 平均 net_bps 改善 ≥ +3 bps」推導

**QC 復算**：
- 預期 fee saving = 4.5 bps（gross）
- AC-5 = +3 bps target
- Implicit 假設：30% fallback × 6 bps adverse = 1.8 bps drag
- Net = 4.5 − 1.8 = 2.7 bps ≈ +3 bps ✓

**數學上自洽**，但 **未明文推導**。應修：AMD / spec 補推導 footnote。

### 4.2 Per-exit_reason 差異化 AC

spec AC-4 給每 reason ≥ 10 close fill 樣本；bw_squeeze / pctb_revert 各 ≥ 1（套 carve-out）。**樣本量極不對稱**：

| exit_reason | demo 7d 預期樣本 | 顯著性 power（t-test α=0.05, β=0.20）|
|---|---|---|
| grid_close_short | ~100 | power > 0.9 detect ΔBPS = 3 bps ✓ |
| bb_mean_revert | ~20 | power ≈ 0.5 detect ΔBPS = 3 bps ⚠️ |
| ma_reverse_cross | ~1 | power ≈ 0.05 ❌ |
| bw_squeeze / pctb_revert | ~1 | 同上 ❌ |

**ma_reverse_cross / bw_squeeze / pctb_revert 樣本量 < 30 不足以判斷 AC-5 對該 reason 是否真 pass**。

**應修**：AC-5 改為 per-exit_reason 分層 — 樣本 ≥ 50 reason 要求 +3 bps 顯著；樣本 < 30 只要求 directional improvement (not regression)，正式 PASS gate 延至累積 ≥ 50 sample 後 90d 內。

## §5 Multiple Testing

### 5.1 16 同時 AC test point 問題

**8 exit_reason × 2 environments (demo + live_demo) = 16 simultaneous AC tests**

若每 test α=0.05 獨立，全族 (family-wise) α' = 1 − (1−0.05)^16 = **0.56**（**56% false positive 機率**）。

**Bonferroni 修正**：每 test α/16 = 0.003，要求 t-stat ≈ 2.94 才算顯著（vs 原 1.96）。
**FDR (Benjamini-Hochberg)**：較寬鬆，但需顯式聲明 q-value target。

**現 AMD/spec 完全沒提 multiple testing**。

**必修**：AMD §5 補 multiple testing protocol — 推薦 **FDR 0.10 with BH procedure**（不要 Bonferroni 太嚴會把 Phase 1b 卡死，FDR 0.10 是 crypto research 通用 default）。

### 5.2 跨 phase 累積錯誤

Phase 2a → Phase 2b → Phase 3 共 3 phase × 16 AC = 48 test points 累積。若不獨立修正，total false discovery rate 爆炸。

**應修**：Phase 2b 是 Phase 2a 的 replication，t-stat 須在 Phase 2a 已修正 α 條件下 holdout 顯著（避免 in-sample over-fit AC）。

## §6 phys_lock DEFER 數學論證

### 6.1 FA verdict §6 DEFER 數學上 sound 嗎？

**FA 論證**：
1. one-flag-per-phase（治理）
2. demo-loose-live-strict policy（治理）
3. live 0 fires by-design（fail-safe）

**QC 從數學/economics 視角驗證**：

`missing_edge_fallback_bps` 控制 Gate 1 行為：
- live 當前 = −10 bps → effective_edge ≤ floor (5 bps) → 100% Hold
- demo override = +10 bps → Gate 1 pass → phys_lock 7d 86 fires

**啟用 live phys_lock 的真實 trade-off**：

| 維度 | 啟用前（current）| 啟用後 |
|---|---|---|
| 持倉期望時間 | 較長（無 phys_lock 主動鎖利）| 較短（gate4 觸發 profit lock）|
| Opportunity cost | 較高（capital tied up）| 較低 |
| Max profit per trade | 高（騎完整波段）| 受 peak ATR giveback 截斷 |
| Max DD per trade | 高（無中途鎖利）| 受限 |
| **核心問題** | Sharpe-style risk-adjusted return 是否改善？ | 取決於 ROC distribution shape |

**結論**：FA 「live 0 phys_lock fires → 持倉時間不必要長 → opportunity cost」這個 argument **數學上對方向 sound，但 magnitude 未驗證**。在 P0-EDGE-1 active（5 策略 structural alpha-deficient）狀態下，phys_lock 啟用 only makes sense **if** it improves Sharpe in alpha-deficient regime — 而 alpha-deficient 下 phys_lock 鎖的可能是 noise 不是真 alpha，**啟用反向可能更糟**。

**QC verdict on DEFER**：**SUPPORT FA DEFER**，加額外條件：

**應修**：Phase 2b PASS 後另開 phys_lock live PR 時，必先跑 counterfactual analysis on demo 86 fires — 「若 demo 不啟用 phys_lock，這 86 fire 對應的倉位 expected PnL 是？」如果 counterfactual PnL > with-phys-lock PnL，**phys_lock 啟用方向錯，不應 live**。
