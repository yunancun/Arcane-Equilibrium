## 二、信息架构问题（9 项）

**I01 — Tab 顺序不符合使用频率**
当前顺序：系统总览 → 实盘锁 → 测试交易 → K线 → 策略 → 风控 → AI → 学习 → 治理 → 监控 → 设置

问题：「实盘锁」是一个「状态展示页」，不是操作员常用的功能。把它放在第二位（最黄金的位置）会让用户误以为这里有重要操作。操作员检查顺序通常是：总览 → 当前交易状态 → 策略/风控 → 偶尔看 AI/学习 → 很少看治理/设置。

建议顺序：
```
系统总览 → 测试交易 → K线图表 → 策略中心 → 风控止损 → AI引擎 → 学习系统 → 监控 → 治理控制 → 设置 → 实盘（锁定，放末尾）
```

**I02 — 「设置」Tab 混入了不属于设置的操作**
tab-settings.html 里有：Demo 控制面板（Validate/Arm/Enable/Relock 按钮）、产品族配置、成本/PnL 手动录入、计划重启。这些操作性质差异极大，放在「设置」下导致重要的 Demo 控制操作被埋没。

建议：Demo 控制面板拆出，放在测试交易 Tab 内（或作为测试交易的第三个子 Tab）。计划重启放入「监控」Tab。

**I03 — 侧边栏信息与主内容重复**
侧边栏有 Paper PnL、Session State、AI Cost、Control API 状态、OpenClaw Gateway 状态。这些信息在各个 Tab 内部也会显示，而侧边栏用的是更小的字体和更少的上下文，价值有限，但占用了 220px 宽度（约 18% 的屏幕宽）。

建议：侧边栏保留最关键的 3 个「全局状态」（系统模式、执行权限、Paper 会话状态），其余移除，释放宽度给主内容区。

**I04 — 治理 Tab 信息密度过高，专业操作员日常用不到**
治理 Tab 包含：授权状态机、风险治理状态机、决策租约追踪、对账、升级门禁（11 项准入）、学习层级、治理事件流、事件时间线、待批准项目。

一次展示太多，而大多数内容都是「只读状态」。真正需要操作的（批准授权、触发对账）淹没在信息海中。

建议：治理 Tab 上方显示「需要你处理的项目」（ActionRequired 区），把待审核、待批准突出展示，其余状态折叠。

**I05 — Paper Trading Tab 下单功能放置位置不直观**
下单区域（Submit Order 表单）放在右侧栏，但用户视线通常从左到右。左侧是 Active Orders 列表，右侧是下单区——这与大多数交易所 UI 约定（下单在左/中，订单列表在下方）相反。

建议：下单区放在持仓/订单列表的上方，或单独一行显示在页面顶部。

**I06 — 「来源上下文」和「健康摘要」对操作员意义不明确**
系统总览中的「来源上下文 / Source Context」（数据连接状态与完整性）和「健康摘要 / Health Summary」（系统健康评分与关键门禁）标题很专业，但操作员不清楚这两个区块告诉自己什么、发现问题后该怎么做。

建议：改为「数据连接状态」和「系统健康检查」，每个问题项旁边加「如何修复」链接。

**I07 — 「产品族事实」用 details/summary 折叠，但重要程度被低估**
产品族（现货/永续合约/期权等哪些品类处于激活状态）对交易员非常重要，但被折叠在 `<details>` 元素里。用户不会主动展开它。

建议：在系统总览上方加一行简短的「当前激活品类」状态指示（如：USDT永续 ✓ / 现货 ✓ / 期权 ✗），不需要完整表格。

**I08 — 监控 Tab Grafana 嵌入框在未连接时占用大量空间**
未连接时，Grafana 区域（55vh 高度）显示一个灰色的「未连接」占位符，占据屏幕一半空间，下面的实际有用信息（Pipeline Bridge 状态、Telegram 状态）被推到屏幕下方，需要滚动才能看到。

建议：未连接时折叠 Grafana 区域（仅显示标题行和连接说明），有用组件提升到顶部。

**I09 — 学习系统 Tab 的「自动扫描」折叠在 details 里，用户不容易发现**
Scan Observations / Scan Lessons / Scan Hypotheses 是手动触发 AI 分析的核心操作，却被折叠在 `<details>` 元素里。

建议：将三个扫描按钮放在页面顶部控制栏（与 Refresh 按钮并排）。

## 三、操作流程问题（7 项）

**O01 — Demo 激活流程分散在两个 Tab，路径不清晰**
激活 Bybit Demo 需要：系统总览按「Bybit Demo」快捷按钮 → 然后去设置 Tab 执行 Validate → Arm → Enable 三步。这个流程没有任何引导。用户按了快捷按钮后只收到 Toast 提示「请前往 Bybit Demo 页面查看连接状态」，但 Demo 页面显示的不是激活流程，而是「未连接」状态。

建议：在「测试交易 → Bybit Demo」子 Tab 里加入完整的激活向导（当前是哪一步，下一步是什么，按钮在哪）。

**O02 — 模式切换后的行为不一致，用户难以预测**
从影子模式降回仅观察模式后，系统不自动停止已运行的 Paper 服务，需要手动去「测试交易」Tab 关闭。这个副作用写在确认对话框里，但如果用户在赶时间时快速点击，容易错过。

建议：在确认对话框添加一个可选择的 checkbox：「同时停止所有测试交易服务」，默认勾选。

**O03 — 风控设置修改后无即时反馈**
「风控止损」Tab 有输入框修改止损参数，点击「保存设置」后没有明确的视觉反馈（如果后端调用失败，toast 消失了就不知道是否成功），也没有显示「上次修改时间：XX分钟前」这类确认信息。

建议：保存成功后，在「当前生效值」区域显示绿色闪烁动画（1秒），并在标签下方显示「已更新: 14:32」时间戳。

**O04 — 学习审核操作（批准/拒绝）在表格里，位置不显眼**
Review Queue 表格的 Actions 列有批准/拒绝按钮，但表格列太多（ID、Type、Summary、Priority、Created、Actions），在小屏或表格内容多时，Actions 列可能被推到右侧不可见区域，操作员可能完全不知道可以在这里操作。

建议：将审核操作从表格内移出，改为点击行展开详情面板的方式，在面板内提供批准/拒绝/咨询 AI 三个大按钮。

**O05 — AI 引擎「Trigger Session」按钮无法判断何时应该使用**
tab-ai.html 顶部有「⚡ Trigger Session」按钮，但没有说明什么情况下需要手动触发、触发后会发生什么（会调用 AI 分析过去数据？会产生多少成本？多久完成？）。

建议：按钮旁边加说明文字：「手动触发 AI 分析最近交易数据（预计 $0.03，约 30 秒）」，点击后显示进度状态。

**O06 — 计划重启的三步确认流程跨语言混用**
重启 Modal 的第一步是繁体中文「風險說明」「選擇重啟延遲時間」，第二步是「強制清倉？」（繁中），第三步没读到，但整体在简体/繁体之间来回切换，给用户带来语言一致性的困惑。

建议：统一使用简体中文（与界面整体一致），或在设置中增加语言选项。

**O07 — 「Safe Recheck Bundle」按钮没有说明也没有确认对话框**
此按钮直接触发操作，没有确认弹窗，也没有说明会做什么。如果触发失败，toast 只显示「Recheck failed」，没有进一步指引。

建议：添加 tooltip 说明，并在按钮点击后显示一个简单弹窗：「将重新验证 Demo 连接的所有前置条件（约 5 秒）」，点击确认后执行。

## 四、状态显示问题（5 项）

**S01 — 系统总览「快照绑定」和「运行保护」显示同一个数据值**
两个指标都来自 `runtime_still_protected`，但标签不同，数值相同。操作员看到两个不同标签显示完全相同的绿色状态，会产生疑惑：这是重复的吗？还是有区别？

建议：合并为一个指标「运行时保护」，或明确区分两者含义（如果确实不同）。

**S02 — 模拟净盈亏没有基准参照**
侧边栏和主内容区都显示 Net PnL，但没有初始资金参照（100,000 USDT）的对比，单看绝对数字 `-$23.45` 不知道是 0.02% 的小亏损还是 5% 的大亏损。

建议：显示为 `-$23.45（-0.02%）`，同时在 tooltip 里说明「初始资金：USDT 100,000」。

**S03 — 治理 Tab 的「授权到期」时间格式不直观**
`Auth Expiry / 授权到期` 显示的可能是时间戳格式，对于快到期的授权，操作员看到「2026-03-31 16:00:00」不如看到「还有 2 小时 15 分钟」直观。

建议：显示相对时间「剩余 2小时15分」，hover 后显示绝对时间戳。

**S04 — 实盘锁 Tab 前置条件的视觉状态混乱**
前置条件用 dot-done（绿）/ dot-todo（灰）/ dot-warn（黄）三种状态，但没有图例说明各颜色含义。用户看到黄色「H0 Gate」不清楚这是「进行中」还是「警告」还是「优先级高」。

建议：在状态点旁加文字标签：「✓ 已完成」「⏳ 进行中」「○ 未开始」，或在前置条件列表顶部加简单图例。

**S05 — 风控 Tab「Session Halted」布尔值不够直观**
`Session Halted` 显示 `true` / `false` 或类似的布尔值，比较冷漠。操作员需要的是「交易已暂停：原因（回撤超限）」或「交易正常运行」。

建议：改为自然语言状态：「正常运行」（绿）或「已暂停：回撤超过 15%」（红），不要只显示 true/false。

## 五、反人类设计（8 项）

严重程度：Critical / High / Medium

**A01 — [Critical] 系统总览的「模式切换」按钮 `设计 → 观察 → 影子 → Demo` 用户不知道自己当前的状态需要往哪个方向移动**

问题：按钮网格显示四种模式，用「✓ 当前」标记当前模式，但没有直观的进度路径感。用户不清楚这是「升级路径」（有顺序的），还是「随时可切换的选项」。
后果：用户可能直接从「仅设计」跳到「Demo 预留」，跳过「观察」和「影子」阶段，导致数据积累不足就开始 Demo 测试。

建议：用步骤指示器（Step 1 → 2 → 3 → 4）替代按钮网格，清晰显示当前处于哪一步，只允许升到下一步（已满足条件时绿色可点，未满足时显示缺什么条件）。

**A02 — [Critical] 学习系统的「Scan Observations」「Scan Lessons」「Scan Hypotheses」三个按钮是英文，且无确认弹窗**

问题：这三个按钮会触发 AI 分析（产生成本），但没有告知预计成本、执行时间，也没有确认弹窗。对于懂交易不懂 AI 的操作员，「Scan Observations」到底是扫描什么？
后果：用户可能在测试阶段频繁点击，积累不必要的 AI 成本。

建议：按钮改为中文「分析市场观察」「分析交易教训」「分析策略假设」，每个按钮点击后弹窗：「将调用 AI 分析最近 200 条观察数据，预计消耗约 $0.05，约 20 秒。确认？」

**A03 — [High] 治理 Tab「Paper → Live Gate / 升级门禁」的「评估」按钮没有结果存储提示**

点击「评估」按钮后，系统会重新评估 11 项准入条件。但用户不清楚：评估结果是否被保存？是否会影响什么？评估过于频繁是否有副作用？
建议：评估按钮点击后显示进度旋转，完成后 toast 说明「评估完成，结果已记录，评分从 X 变为 Y」。

**A04 — [High] 测试交易 Tab 子 Tab 切换时，Paper 和 Demo 的 iframe 是独立实例，切换会重新加载数据**

用户在 Paper 子 Tab 看到一组持仓数据，切回来后 iframe 状态还在（已实现懒加载），但如果用户在两者之间频繁切换，行情数据的 WebSocket 连接不清楚是否共享，可能造成重复连接或数据不一致的困惑。界面上没有任何状态同步指示。
建议：在子 Tab 切换时明确显示「已连接/断开」状态，让用户了解数据连接状况。

**A05 — [High] 策略 Tab 的 Hard Stop 按钮在策略卡片中的位置（从代码中推断）可能过于隐蔽**

从 explainer 描述「你可以通过策略中心的'Hard Stop'按钮手动关闭任何策略」推断，这是一个高频紧急操作，但从代码看策略列表使用 `.oc-strat-grid` 渲染，Hard Stop 按钮在卡片内，如果策略很多（上限100个），用户要找到并停止某个特定策略需要在卡片堆里找按钮。
建议：策略列表顶部添加「全部停止」的全局紧急停止按钮（醒目的红色），并支持按名称/Symbol 过滤策略。

**A06 — [High] 风控「保存设置」没有验证输入范围的用户反馈**

输入框有 `min`/`max` 属性，但如果用户输入超出范围的值（如 Max Leverage = 200），HTML5 的原生验证可能不会阻止提交（取决于实现），且错误提示样式可能跟界面整体风格不符。
建议：保存前在前端做显式验证，如果范围不对，在对应输入框下方用红色小字提示（而不是靠 HTML5 原生气泡）。

**A07 — [Medium] 系统总览快速操作按钮中，「Bybit Demo」和「自动扫描 Scanner」点击后只弹出 toast 导航提示，不执行操作**

按钮外观与「模拟交易 Paper」「行情流 Feed」完全一样，但行为完全不同：Paper 和 Feed 会弹出确认对话框然后执行操作，Demo 和 Scanner 只弹出「请前往 XX Tab」的导航提示。
这创造了不一致的用户期望：用户以为点了 Demo 按钮就在开关 Demo，实际什么都没发生。

建议：将「Bybit Demo」和「Scanner」快捷按钮改为「跳转按钮」样式（不同的视觉设计，如箭头图标），标签改为「前往 Demo 设置」「前往扫描器」，明确表示是导航而非操作。

**A08 — [Medium] 实盘锁 Tab 中的运行状态 chips 显示英文技术值**

例如：`system_mode: read_only`、`execution_state: disabled`，用户看到英文小字 chip 里的技术字段，在这个本来是给 Operator 展示「你还不能开实盘」的页面上，这些内容反而增加了阅读负担。

建议：将 chips 改为：「模式：只读（安全）」「执行：已禁用（安全）」「授权：未授予（安全）」，用自然语言表达，用绿色表示安全状态。

## 六、视觉设计问题（4 项）

**V01 — 「快捷按钮」的点状态指示器（绿点/灰点）太小，在暗色背景下几乎看不见**
每个快速操作按钮左侧有一个 8px 的状态圆点，灰色（off）在 `#0d1117` 背景上几乎不可辨认。
建议：增大到 10px，或在按钮文字后面加文字状态「运行中」/「已停止」。

**V02 — 「策略」和「设置」Tab 使用相同的齿轮图标 ⚙**
Tab 栏中策略（⚙ 策略中心）和设置（⚙ 设置）都用齿轮图标。用户在快速扫视 tab 栏时容易混淆。
建议：策略中心改用 📊 或 📈 图标，只有「设置」保留齿轮。

**V03 — 模式按钮网格的激活状态（蓝色边框+蓝色背景）与普通 hover 状态（蓝色边框）区分不够明显**
`active` 状态的背景色 `rgba(88,166,255,0.1)` 与 hover 状态 `rgba(88,166,255,0.05)` 差距极小，快速扫视难以区分「当前模式」和「鼠标悬停」。
建议：active 状态加一个更明显的视觉差异，如 `border-width: 2px` 或在按钮内加蓝色实心左侧边条。

**V04 — Paper Trading Tab 的「Start / Pause / Resume / Stop」按钮排列时，在 Paper 未启动状态下 Pause/Resume/Stop 是灰色 disabled，但外观与 Start 几乎相同**
操作员不熟悉时可能不清楚哪个按钮是当前可用的。
建议：未启动状态下，Pause/Resume/Stop 用更暗的背景和更淡的文字颜色，或者直接隐藏（只显示 Start），启动后再显示 Pause/Stop。

## 九、做得好的地方（值得保留的设计亮点）

**G01 — 双层解释系统（ocExplain）非常优秀**
每个区块有「简单说明」和「详细了解」两层，非常适合不同经验水平的操作员。普通用户看简单说明即可，需要深入了解时点击展开。这个模式值得扩展到更多区域。

**G02 — 确认弹窗内容完整**
系统模式切换的确认对话框写出了每种操作的具体影响（哪些服务会变化、哪些不变、注意事项）。这是业界最佳实践，比大多数 trading GUI 的「确认执行」一句话强得多。

**G03 — 货币三档切换（USDT/USD/EUR）**
通过点击 header 上的货币标签切换显示货币，且实时获取汇率。对国际化操作员非常友好。

**G04 — 实盘锁定 Tab 设计**
用前置条件清单替代「你不能用这个」的简单禁止，让 Operator 清楚地了解还差什么、哪些已满足。锁定页面的信息密度恰到好处。

**G05 — 计划重启的多步确认流程**
三步确认（风险说明 → 是否清仓 → 最终确认）防止误操作，且每步说明清晰。特别是「只平盈利仓位，不强平亏损仓位」这条规则的呈现方式值得肯定。

**G06 — AI 供应商管理页面**
六个供应商卡片布局清晰，每个都有价格参考，帮助操作员做成本权衡。Ollama 本地 LLM 的检测按钮和状态提示尤其实用。

**G07 — 颜色系统一致性**
全局使用 green=盈利/正常、red=亏损/危险、yellow=警告/待定、blue=中性信息，这个约定在整个系统中保持一致，操作员可以快速通过颜色判断状态。

**G08 — 托管状态（Paper vs Demo 对比）**
tab-demo.html 中的 Paper vs Demo 对比表，能帮助操作员验证本地模拟引擎是否准确，是一个有实际价值的分析工具。
