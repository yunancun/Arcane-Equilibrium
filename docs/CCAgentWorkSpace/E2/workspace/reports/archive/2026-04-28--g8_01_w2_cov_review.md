## §4 對抗反問結果

### Q1：「100% line cov」是 false confidence 還是真覆蓋？

**事實鏈**:
- 已 grep 確認 production 唯一 caller `tick_cognitive_modulator` (`strategist_cognitive.py:265-270`) **總是傳** `regret_data={}` / `dream_data={}`。
- 因此產製 cov 涵蓋的 regret/dream 分支（lines 119-123, 147-155 部分）在 production hot-path **結構性不可達**。
- W2 測試 Cases 7/8/9/13/14/15 直呼 `update(...)` 餵 schema dict 才達 100% cov。

**評估**:
- 屬「**API 契約測試**」（test against the public method signature），非「production behavior assertion」。
- 與 `feedback_no_dead_params` **不衝突** — `update(...)` 簽名本身仍是 active production API（`tick_cognitive_modulator` 即時調用），kwargs 為合法輸入空間。dead 的是 **producers**（`OpportunityTracker` / `DreamEngine`），不是 modulator API。
- E1 MODULE_NOTE 已明確標註「regret/dream branches 反模式說明」，REGRET-DREAM escalation 已在 PA 層開 backlog 票（Option C defer，Option B 重啟為未來 wave）。
- **結論**：100% cov 數字本身**無誤導**，但 cov 數字 ≠ behavior coverage 是讀者必須自己理解的事；E1 已用 docstring 主動說明，符合 CLAUDE.md §二 原則 #10「認知誠實」。

**Push back 方向（記錄，不退回）**:
- 若未來 Option B 落地（重實作 OpportunityTracker/DreamEngine schema 改變），W2 這些 contract test 會「綠通過但未驗整合」。屬已知 trade-off，PA escalation 已記錄。
- 不退回 E1：增 `# pragma: no cover` 或 `omit` 反而更糟（會在重實作落地時失去 regression baseline）。

### Q2：22-case → 26 sub-test drift？

PA RFC §3.2 表列 22 entry，E1 落 26 collected items：
- Case 20「Clamp bounds verified」拆 2 method（直接 _clamp + runtime stay-in-bounds）— 合理：兩個獨立 invariant
- Case 22「Getter rounding contract」拆 4 method（confidence_floor / qty_ceiling / stoploss_multiplier / scan_interval）— 合理：4 個 getter 各自合約

**結論**：非 spec drift，屬合理細化。RFC table cell #22 本身就描述「4 個 getter rounding」，拆 4 method 是更乾淨的 unit。

### Q3：Mirror test antipattern？

部分測試（Cases 4-11, 13, 15-18）用 `expected = _EMA_ALPHA * target + (1-_EMA_ALPHA) * _BASE_*` 與 production 公式同形 — 屬輕度 mirror。

**評估**:
- 若 production EMA 公式 bug-flip（α/(1-α) 寫反），test 會同步 flip → bug 被掩蓋。
- 但**非 pure mirror**：Cases 5（cap=5）、9（[R1-5] ignore）、11（worst-case min 而非 sum）、18（min of two condition）測的是**概念 invariant**，不是公式 echo。
- 用 hardcoded `0.606` 等魔數會在 `_EMA_ALPHA` tuning 時 brittle break。
- **結論**：可接受 trade-off。**建議（不阻 merge）**：未來 W3 integration test 加一個「EMA 反方向 sanity」（餵 1 cycle pos → confidence 必上升 not 下降）作為公式方向的 oracle test。

### Q4：邊界條件覆蓋？

- None：✅ Case 2（None regret/dream）
- 空 dict：✅ Cases 3, 4 等
- 0：✅ Case 3
- 負：✅ Cases 6, 11, 16, 18
- 極大：✅ Cases 12（consec=100）, 20（pathological pressure 50 cycle）
- unicode：✅ N/A（無 string input 路徑）
- direction enum 未知值：❌ **未測**（如 `direction='unknown_string'` 應該走 default `balanced` 邏輯，code line 119 default 處理，但無 test 直接覆蓋）— **LOW priority gap**，line 已 cov（fall-through path），純 robustness 加分項

### Q5：Race / 並發？

`CognitiveModulator` 為 stateless pure-Python（non-singleton；每次 `tick` 從 strategist_cognitive 拿同一 instance）。內部 `_confidence_floor` 等為 instance attr，無 thread sync。E1 test 全 sync 不測 race — **屬正確不測**（單 worker 設計）。但 production 若 multi-worker uvicorn 場景，每個 worker 有獨立 modulator → 無共享 state → **無 race 風險**。✓

### Q6：100% cov 在未來 Option B 重實作下會變 trap 嗎？

**情境**：
- 未來重實作 `OpportunityTracker` 餵真 `regret_data` schema，假如 schema key 從 `net_regret_direction` 改成 `direction`，W2 test 會繼續綠（測 modulator side），integration 會死。
- 屬「contract drift」標準風險，不限於本 review 範圍。

**緩解**：
- `OpportunityTracker` 重實作時 PA RFC 必同步修 modulator API contract（schema 變更 = breaking change，需新 migration）
- 屆時 W2 test 會 fail（assert direction 不再生效）→ 強制更新
- **結論**：trap 風險**極低**且有自然檢測機制；不需先發制人加 pragma。
